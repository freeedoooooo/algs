# 开发流程指令

> 控制需求→设计→任务→实现→验收各阶段的流转。
> 前提：必须已通过 `START` 或 `MODULE` 激活工作流。

---

## `NEW {编号} {功能描述}`

启动新功能，进入需求澄清阶段（阶段 1）。

**示例**：`NEW 0005 开发同比变动率算子` / `NEW 9101 修复银行账号长度校验`

**前提**：编号必须由用户提供，AI 不得自行生成编号。若用户未提供编号，AI 必须先询问：
> 请提供本次需求的编号（内部序号如 `0005`，或 issue 编号如 `9101`）。

**AI 执行动作**：
1. 使用用户提供的编号和功能描述生成目录名：`{编号}-{模块}-{功能描述-kebab-case}`
   - 示例：`0005-data-yoy-change-rate`
   - 示例：`9101-report-bank-account-length-fix`
2. 创建目录 `docs-c1/00 AI-DOCS/specs/{功能目录名}/`
3. 主动追问需求细节（参考专属宪法中的业务域）
4. 基于 `templates/requirements-template.md` 生成 `requirements.md`
5. 生成后向用户发起确认：
   > 需求文档已生成，请确认以下内容是否正确：
   > 1. 业务规则和计算逻辑是否完整准确？
   > 2. 输入参数是否齐全？
   > 3. 输出字段是否符合预期？
   > 4. 边界条件是否覆盖？
   >
   > 确认无误后请回复"需求确认"，或告知需要修改的地方。

---

## `需求确认`

确认需求文档，进入技术设计阶段（阶段 2）。

**触发时机**：`requirements.md` 已生成，用户审阅完毕。

**AI 执行动作**：
1. 确认 `requirements.md` 存在且内容完整
2. 阅读公共宪法 + 专属宪法中的技术规范
3. 参考同类现有代码
4. 基于 `templates/design-template.md` 生成 `design.md`
5. 生成后向用户发起确认：
   > 技术设计文档已生成，请确认以下内容是否正确：
   > 1. 架构设计和数据流是否合理？
   > 2. 核心逻辑/SQL 是否符合预期？
   > 3. 影响范围是否完整？
   >
   > 确认无误后请回复"设计确认"，或告知需要修改的地方。

---

## `设计确认`

确认技术设计，进入任务拆解阶段（阶段 3）。

**触发时机**：`design.md` 已生成，用户审阅完毕。

**AI 执行动作**：
1. 阅读 `requirements.md` 和 `design.md`
2. 将设计拆解为可执行的小任务（每个任务 30 分钟内可完成）
3. 基于 `templates/tasks-template.md` 生成 `tasks.md`
4. 生成后向用户发起确认：
   > 任务清单已生成，请确认以下内容是否正确：
   > 1. 任务拆解是否合理，粒度是否适当？
   > 2. 任务顺序是否正确？
   > 3. 是否有遗漏的任务？
   >
   > 确认无误后请回复"任务确认"，或告知需要修改的地方。

---

## `任务确认`

确认任务清单，进入实现阶段（阶段 4）。

**触发时机**：`tasks.md` 已生成，用户审阅完毕。

**AI 执行动作**：
1. 确认 `tasks.md` 存在
2. 提示用户可开始执行：
   ```
   ✅ 任务清单已确认，可以开始实现。
   使用 DO NEXT 执行下一个任务，或 DO ALL 自动执行所有任务。
   ```

---

## `验收`

所有任务完成后触发验收检查（阶段 5）。

**触发时机**：`tasks.md` 中所有任务已标记 `[x]`。

**AI 执行动作**：
1. 对照 `requirements.md` 逐条检查验收标准
2. 对照公共宪法 + 专属宪法检查代码规范
3. 执行 `templates/checklist-template.md` 中的自检清单
4. 输出验收报告：
   ```
   📋 验收报告：{功能名}

   需求验收：
   ✅ 标准1：...
   ✅ 标准2：...

   宪法自检：
   ✅ 分层架构：符合
   ⚠️  注解规范：Controller 缺少 @Api(tags)

   结论：需修复 1 项后可提交。
   ```
