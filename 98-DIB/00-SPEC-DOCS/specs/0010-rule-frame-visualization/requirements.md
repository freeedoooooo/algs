# 框架准备-可视化 需求文档

> 编号：`0010` | 模块：`rule` | 服务：`dib-agent-service-rule` | 创建时间：`2026-04-14`

## 背景

当前 `rule` 服务已经具备评分框架的基础管理、节点管理、规则挂载、计算、结果查询与导出能力，但这些能力主要面向“框架定义与计算”，并不直接提供面向业务前端页面的可视化展示配置。

在现有业务场景下，前端需要基于“已发布框架”的计算结果，快速构建风险总览、等级分布、风险清单等展示页面。产品原型中的“框架准备-可视化”正是为此增加的一层配置能力：由业务人员为某个已发布框架维护一套独立的展示配置，并生成稳定的前端访问地址，用于菜单挂载和页面访问。

该能力的核心目标不是改变框架计算逻辑，而是为已存在的框架结果增加“展示规则”和“访问入口”。

## 目标用户

- 运营/产品配置人员：为已发布框架配置展示页的可视化参数
- 前端页面接入方：消费可视化配置和框架结果，渲染风险总览、等级分布、风险清单等页面
- 系统管理员：通过生成的前端 URL 配置菜单或发布新的展示入口

## 功能描述

新增“框架准备-可视化”能力，用于围绕“已发布框架”维护一套独立的展示配置。配置完成后，系统需要支持保存草稿配置，并在用户显式执行“生成前端 URL / 发布为新地址”时，生成稳定的前端访问地址，供业务菜单配置和前端页面访问使用。

本能力重点覆盖以下内容：

1. 选择一个可配置的正式框架
2. 配置统计层级
3. 配置风险等级及其颜色、分值区间
4. 保存当前可视化配置
5. 生成前端访问 URL，形成稳定展示入口

本期需求明确不包含：

- 不改动框架计算逻辑
- 不改动框架结果表结构和计算产出逻辑
- 不包含“待完成-批量计算”页面对应能力实现
- 不要求保存后立即更新已发布的前端页面地址

## 核心业务规则

### 1. 配置对象

- 可视化配置的绑定对象是“已发布框架”
- 一个可视化配置从业务上隶属于一个正式框架，而不是草稿框架
- 可视化配置独立于框架计算逻辑，修改配置不会触发框架结构、节点、权重、规则定义变化

### 2. 框架选择规则

- 只能从“已发布/正式”的框架中选择
- 非正式、草稿、禁用或不满足发布条件的框架，不应出现在可配置列表中
- 选择框架后，后续配置均围绕该框架的计算结果展开

### 3. 统计层级规则

- 可视化配置中需要维护“统计层级”
- 统计层级用于定义展示页聚合和切换口径
- 统计层级会影响展示页中的“前十大风险”和“风险清单”页签切换逻辑
- 统计层级需要作为可持久化配置保存，供前端页面读取

### 4. 风险等级规则

- 一个框架可配置多个风险等级
- 每个风险等级至少包含：
  - 等级名称
  - 分值范围
  - 展示颜色
- 等级名称使用中文
- 分值范围用于将框架计算结果映射到风险等级
- 若未设置颜色，则前端默认使用系统黑色展示
- 风险等级数量会直接影响综合分析页下方风险等级公司卡片的展示数量

### 5. 分值区间规则

- 风险等级必须形成明确、可解释的分值区间
- 同一套配置下，分值区间不允许存在明显冲突：
  - 不能出现起始值大于结束值
  - 不能出现两个等级区间完全重叠
  - 不能出现等级名称重复
- 是否必须覆盖全部分值空间，由产品后续策略决定；但系统至少要校验已配置区间的格式合法性和互斥性

### 6. 配置保存规则

- 保存当前页面配置，仅保存配置本身
- 保存后不自动刷新已生成的前端 URL 指向内容
- 保存后不自动替换线上菜单入口
- 保存动作应支持后续继续编辑

### 7. URL 生成/发布规则

- 系统需支持生成前端 URL
- 生成出的 URL 用于前端菜单配置或直接访问
- 原型语义明确：已完成页面 URL 不随时更新
- 因此，URL 生成应视为显式发布动作，而不是保存动作的附带行为
- 当配置发生后续修改时，系统应允许再次生成“新地址”，而不是静默覆盖原地址

### 8. 展示与计算解耦规则

- 可视化配置消费的是框架计算结果，不改变计算结果本身
- 可视化配置仅定义“如何展示”“如何映射等级”“如何组织页面入口”
- 框架计算、手工试算、结果分页、结果导出等现有能力继续按原逻辑运行

## 输入参数

| 参数名 | 类型 | 来源 | 说明 |
|--------|------|------|------|
| frameId | Long | 页面选择 / 请求参数 | 已发布正式框架 ID |

| statLevel | String / Integer | 页面配置 | 统计层级 |
| levelConfigs | List | 页面配置 | 风险等级配置集合 |
| levelName | String | 页面配置 | 风险等级中文名称 |
| scoreMin | Decimal / Integer | 页面配置 | 分值范围下界 |
| scoreMax | Decimal / Integer | 页面配置 | 分值范围上界 |
| levelColor | String | 页面配置 | 风险等级展示颜色 |
| publishFlag | Boolean | 页面操作 | 是否执行 URL 生成/发布动作 |
| visualizationId | Long | 系统生成 | 可视化配置主键 |

## 用户故事

- 作为产品配置人员，我希望从已发布框架中选择一个正式框架，以便基于真实计算结果配置展示页。
- 作为产品配置人员，我希望设置统计层级，以便前十大风险和风险清单按业务需要切换展示维度。
- 作为产品配置人员，我希望配置多个风险等级及颜色和分值范围，以便前端可以渲染等级分布和风险公司卡片。
- 作为产品配置人员，我希望保存当前配置而不立刻影响线上展示地址，以便我可以反复调整配置内容。
- 作为产品配置人员，我希望在确认配置后生成一个新的前端 URL，以便将其挂到菜单或作为新的可访问入口。
- 作为前端页面接入方，我希望读取一份稳定、结构化的可视化配置，以便直接渲染风险总览、等级分布、风险清单页面。

## 验收标准

- [ ] 只能选择已发布的正式框架作为可视化配置对象，草稿框架不可被选择
- [ ] 系统支持维护统计层级配置，并可在查询配置时完整返回
- [ ] 系统支持维护多个风险等级，每个等级至少包含等级名称、分值范围、颜色
- [ ] 风险等级名称支持中文，颜色未设置时页面按默认黑色处理
- [ ] 风险等级配置数量可驱动综合分析页下方风险等级公司卡片展示数量
- [ ] 保存配置时，仅保存配置内容，不改动框架计算逻辑，不刷新已发布 URL
- [ ] 系统支持显式执行“生成前端 URL / 发布为新地址”，并返回稳定可用的访问地址
- [ ] 后续修改配置后，不会自动覆盖既有已完成页面 URL，只有再次执行发布动作才生成新地址
- [ ] 前端可通过接口拿到完整可视化配置，用于风险总览、等级分布、风险清单等页面渲染
- [ ] 本期需求不包含“待完成-批量计算”的实现内容

## 边界条件

| 场景 | 处理方式 |
|------|---------|
| 未选择框架直接保存配置 | 拒绝保存，提示必须先选择正式框架 |
| 选择的框架不是已发布正式框架 | 拒绝保存或发布，提示框架状态不合法 |
| 风险等级名称为空 | 拒绝保存，提示等级名称必填 |
| 风险等级名称重复 | 拒绝保存，提示等级名称不能重复 |
| 分值区间格式非法 | 拒绝保存，提示分值范围不合法 |
| 两个风险等级分值区间冲突 | 拒绝保存，提示等级区间冲突 |
| 未设置颜色 | 允许保存，前端按默认黑色展示 |
| 已存在配置但尚未发布 | 允许继续编辑并保存，不生成新 URL |
| 已发布配置再次编辑 | 允许保存草稿，但不自动影响历史 URL |
| 用户显式再次发布 | 生成新的前端地址，供菜单重新配置 |
| 框架暂无结果数据 | 允许先保存配置，但前端展示效果依赖后续结果数据 |

## 非功能需求

- **性能**：查询可视化配置时应能直接返回前端渲染所需核心字段，避免前端发起过多二次请求
- **可维护性**：配置能力应与框架计算能力解耦，避免将展示规则写入现有计算链路
- **一致性**：所有接口响应统一使用 `GeneralResult<T>`，遵循 `rule` 服务现有 Controller 命名与分层规范
- **扩展性**：当前配置模型需为后续扩展页面卡片、图表口径、额外展示组件预留扩展空间
- **安全性**：仅允许有权限的业务用户维护和发布可视化配置；URL 生成规则不得暴露服务内部敏感路径

## 待确认问题

- [ ] 前端 URL 是由 `rule` 服务直接生成完整地址，还是仅生成路由参数/编码，交由网关或前端应用拼接
- [ ] 风险等级的分值区间是否要求必须完整覆盖全部分数空间，例如 `0-100`
- [ ] 统计层级的数据来源是固定枚举、框架维度字段，还是需要从框架结果字段中动态选择
- [ ] 一个正式框架是否允许维护多套并行可视化配置，还是仅允许一套草稿配置 + 多次发布历史
- [ ] 新地址发布后，是否需要保留历史发布记录并支持回查
