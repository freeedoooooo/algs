# 框架可视化-综合分析 需求文档

> 编号：`0013` | 模块：`rule` | 服务：`dib-agent-service-rule` | 创建时间：`2026-05-09`

---

## 背景

当前 `rule` 服务已经具备框架结果计算、结果分页查询、结果字段映射等能力，也已经补充了“框架可视化”配置能力，可以为已发布框架维护统计层级、风险等级区间和展示模块。

在产品原型中，“综合分析”页面需要直接消费框架结果表，并基于框架可视化配置展示：

- 风险等级分布
- 前 10 大风险
- 各风险等级下的企业名单分页

本次需求目标是在不改动框架计算逻辑、不改动结果表产出逻辑的前提下，为“框架可视化 > 综合分析”补充独立的查询接口，供前端页面拆分调用。

## 目标用户

- 前端页面开发人员：根据框架 ID 和用户筛选条件调用综合分析接口
- 业务使用人员：在页面上查看风险等级分布、重点风险和企业名单
- 产品/运营配置人员：通过既有“框架可视化”配置控制统计层级和风险等级映射口径

## 功能描述

围绕“框架可视化”的“综合分析”模块，新增 3 个独立数据接口：

1. 风险等级统计接口
2. 前 10 大风险接口
3. 各风险等级企业名单分页接口

前端已知当前框架 ID，也已知框架有哪些维度，因此本期不新增“查询综合分析条件元数据”接口。

综合分析接口统一遵循以下原则：

- 查询条件来源于框架配置的维度
- 统计结果来源于框架结果表 `com_frame_value`
- 统计层级来源于框架可视化配置的 `statLevel`
- 风险等级来源于框架可视化配置的等级区间，而不是直接使用结果表中的 `node_score_level`
- 页面数据分 3 个接口拆开提供，不将所有数据混在一个接口返回

## 核心业务规则

### 1. 查询前置条件

- 前端必须传入 `frameId`
- `frameId` 对应的框架必须存在
- `frameId` 必须已存在可视化配置
- 只有可视化配置中已启用且未删除的“综合分析”模块才允许查询综合分析接口

### 2. 维度筛选规则

- 查询条件由前端传入 `dimFilterList`
- 传入的维度 ID 必须属于当前框架的维度配置
- 后端根据框架维度映射关系，将维度 ID 转为 `com_frame_value` 中真实字段名后参与查询
- 不允许对当前框架未配置的维度发起筛选

### 3. 统计结果来源规则

- 所有综合分析统计均直接基于 `com_frame_value` 表
- 不读取 DataEase 看板统计结果
- 不修改框架计算逻辑，不回写结果表

### 4. 统计层级规则

- 统计层级取自框架可视化配置中的 `statLevel`
- 综合分析页面的统计口径围绕该层级展开
- “该层级及以下的节点层级”解释为：统计范围包含 `node_level >= statLevel` 的节点结果
- 其中企业名单分页以“统计层级对应节点”为主口径，前 10 大风险按“统计层级及以下节点”聚合

### 5. 风险等级映射规则

- 风险等级区间来源于 `com_frame_visualization_level`
- 结果表中的 `node_score` 作为风险等级映射输入值
- 后端根据配置区间计算每条记录所属风险等级
- 若某条结果分值无法命中任何已配置区间，则该条结果不进入风险等级统计和企业名单

### 6. 风险等级统计规则

- 统计对象为“统计层级对应节点”的结果记录
- 统计结果按风险等级分组
- 每个等级至少返回：
  - 风险等级名称
  - 风险等级颜色
  - 企业数量
  - 占比
- 占比基于本次查询命中的全部有效企业记录计算

### 7. 前 10 大风险规则

- 统计范围为 `node_level >= statLevel`
- 统计对象为节点风险项，而不是风险等级
- 每个风险项的统计值定义为：命中该节点的企业数量
- 按风险企业数倒序排序，返回前 10 条
- 若不足 10 条，则按实际条数返回

### 8. 各等级企业名单分页规则

- 按指定风险等级查询企业名单
- 分页接口一次只查询一个风险等级
- 企业编码默认取维度字段常量 `dim_sec_code`
- 所属行业默认取维度字段常量 `dim_industry_code`
- 企业名单至少返回：
  - 企业编码
  - 企业名称
  - 所属行业
  - 报告期
  - 分值
  - 风险等级名称
- 企业名称、所属行业等展示字段来自框架结果表中的维度字段及其名称映射
- 企业名单默认排序固定为 `score desc, companyCode asc`

### 9. 页面拆分规则

- 风险等级统计、前 10 大风险、企业名单分页必须拆成 3 个接口
- 后端不提供“综合分析整页数据汇总接口”

## 输入参数

| 参数名 | 类型 | 来源 | 说明 |
|--------|------|------|------|
| frameId | Long | 请求体 | 当前框架 ID |
| dimFilterList | List<DimFilterBO> | 请求体 | 维度筛选条件列表 |
| levelId | Long | 请求体 | 企业名单分页接口中的目标风险等级 ID |
| pageNum | Integer | 请求体 | 企业名单分页页码 |
| pageSize | Integer | 请求体 | 企业名单分页每页条数 |
| topN | Integer | 请求体 | 前 10 大风险查询条数，默认 10 |

## 用户故事

- 作为前端开发人员，我希望按页面模块分别查询风险等级分布、前 10 大风险和企业名单分页，以便页面按区域独立渲染。
- 作为业务用户，我希望风险等级统计口径严格跟随框架可视化配置的等级区间，而不是框架内部固定等级，以便页面展示和产品配置保持一致。
- 作为业务用户，我希望前 10 大风险展示统计层级及以下的重点风险节点，以便我快速定位风险最集中的指标项。
- 作为业务用户，我希望点击某个风险等级后分页查看该等级下的企业名单，以便逐页查看风险企业详情。

## 验收标准

- [ ] 后端提供 3 个独立接口：风险等级统计、前 10 大风险、各等级企业名单分页
- [ ] 后端不提供“综合分析整页汇总接口”
- [ ] 所有统计数据均来自 `com_frame_value` 表
- [ ] 所有风险等级映射均基于框架可视化等级区间配置计算
- [ ] 查询维度仅允许使用当前框架已配置的维度
- [ ] 统计层级口径受可视化配置 `statLevel` 控制
- [ ] 前 10 大风险返回结果按风险企业数倒序排列
- [ ] 企业名单分页接口一次仅返回一个风险等级的数据

## 边界条件

| 场景 | 处理方式 |
|------|---------|
| 框架不存在 | 拒绝查询，提示框架不存在 |
| 框架未配置可视化 | 拒绝查询，提示未配置框架可视化 |
| 综合分析模块未启用 | 拒绝查询，提示综合分析模块不可用 |
| 传入维度不属于当前框架 | 拒绝查询，提示维度条件非法 |
| 结果表无数据 | 返回空统计结果或空分页结果，不报错 |
| 某分值未命中任何等级区间 | 该记录不纳入等级统计和企业名单 |
| 企业名单查询的 levelId 不属于当前可视化配置 | 拒绝查询，提示风险等级非法 |

## 非功能需求

- **性能**：风险等级统计和前 10 大风险优先通过数据库聚合完成，避免全量结果拉回 Java 内存计算。
- **可维护性**：综合分析能力与现有可视化配置 CRUD 解耦，建议独立聚合类与独立请求响应模型。
- **一致性**：所有接口统一使用 `GeneralResult<T>`，分页接口统一使用 `PageResp<T>`。
- **扩展性**：本期设计需为后续“预警报告”“风险清单”保留相同风格的扩展空间。
