## 题目
给定一个数组 `arr`，其中 `arr[i]` 表示第 `i` 个人喝完咖啡、把杯子放到待处理区的时间。

现在每只杯子变干净有两种方式：

- 用洗杯机洗：一次只能洗一只，洗一只需要 `a` 时间
- 自然挥发：每只杯子都可以自己挥发干净，耗时 `b`，彼此互不影响

问：怎样安排每只杯子是“机器洗”还是“自然挥发”，才能让所有杯子都变干净的时间尽量早。

输入：
- `int[] arr`
- `int a`
- `int b`

输出：
- 一个整数，表示所有杯子都干净的最早时间

## 一句话结论
这题最关键的不是“现在处理第几个杯子”，而是“洗杯机下一次最早什么时候能用”。

所以递归状态不能只写 `index`，还必须把真正卡住后续决策的业务限制 `washLine` 一起带上。

## 理论 / 应用价值
- 这是“寻找业务限制”模型里非常典型的一题。
- 难点不在分支本身，而在于先识别出真正影响后续决策的那个量。
- 很多业务题、调度题、资源占用题，表面上看是在枚举对象，实际上真正决定后续的是“资源何时释放”。

## 核心知识点
- 每只杯子都有两种选择：机器洗，或者自然挥发。
- `arr` 默认要按杯子进入待处理区的时间从小到大处理，也就是默认按时间顺序看杯子。
- 当前状态不能只看 `index`，因为同样处理第 `index` 只杯子，洗杯机空闲时间不同，答案就可能完全不同。
- `washLine` 不是某只杯子的时间，而是“洗杯机这条时间线目前推到了哪里”。
- 一条方案内部，要看“这一条方案什么时候整体收工”，所以取 `max`。
- 两条方案之间，要挑“哪一条整体收工更早”，所以取 `min`。

## 图解
### 这题真正影响后续的是什么
```mermaid

    A["当前处理到第 index 只杯子"] --> C["当前状态"]
    B["洗杯机最早可用时间 washLine"] --> C
    C --> D["决定这只杯子怎么处理"]
```

同样都是处理第 `index` 只杯子：

- 如果洗杯机现在空着
- 和洗杯机还要等很久才空出来

这两个状态的最优答案通常不一样。

### 当前杯子的两种选择
```mermaid
graph LR
    A["杯子在 arr[index] 时刻出现"] --> B["机器洗"]
    A --> C["自然挥发"]
    B --> D["完成时间 = max(arr[index], washLine) + a"]
    C --> E["完成时间 = arr[index] + b"]
```

### 为什么一条方案里要取 `max`
```mermaid
graph LR
    A["当前杯子的完成时间"] --> C["整条方案的最终结束时间"]
    B["后续杯子的最终结束时间"] --> C
    C --> D["取 max"]
```

因为我们求的不是“每只杯子各花了多少时间的总和”，而是：

```text
这一整条方案，要到什么时候，所有杯子才算全部干净
```

谁更晚，整条方案就得等谁。

## 解题思路
### 1. 先找到真正的业务限制
这题最容易想偏的地方，是把它看成“每只杯子各自独立做选择”。

但其实不行，因为：
- 自然挥发不占用公共资源
- 机器洗会占用唯一的一台洗杯机

所以当前杯子如果选了机器洗，就会直接改写后面杯子的排队起点。

这就是为什么这题的关键状态不是“杯子编号”，而是：

```text
洗杯机最早什么时候能再次使用
```

### 2. 先补一个容易忽略的前提：杯子要按出现时间顺序处理
这题里默认：

```text
arr[0] <= arr[1] <= arr[2] <= ...
```

也就是 `arr` 里的时间要按从小到大排列。

原因不是为了“写起来方便”，而是因为递归里的 `index` 本来就代表：

```text
按时间顺序，当前处理到第几只已经出现的杯子
```

如果 `arr` 是乱序的，比如先看到时间 `8` 的杯子，再看到时间 `2` 的杯子，那就不符合真实的出现顺序了，后面的递归含义会直接变乱。

所以你可以把这题理解成：

- 如果 `arr` 本身已经是每个人喝完咖啡的先后顺序，就直接用
- 如果只是无序的一组完成时间，那应该先排序，再进入这道题

一句话记住：

```text
这题不是按杯子编号处理，而是按杯子出现的时间顺序处理
```

### 3. 定义递归状态
定义：
```text
process(index, washLine)
```

表示：
- 从第 `index` 只杯子开始处理
- 当前洗杯机最早可用时间是 `washLine`
- 返回后续所有杯子都变干净的最早完成时间

### 4. 单独理解一下 `washLine` 到底是什么
`washLine` 很容易被看成“当前杯子的时间”，但它其实不是。

它真正表示的是：

```text
洗杯机这条时间线，最早什么时候会重新空出来
```

你可以把它想成洗杯机自己的日程表指针。

比如：

- `washLine = 0`：说明洗杯机现在就是空的
- `washLine = 4`：说明洗杯机至少要到时间 `4` 才能接下一只杯子
- `washLine = 10`：说明前面某些机器洗决策已经把它占用到 `10` 了

所以在代码里写：

```text
max(arr[index], washLine)
```

不是在比较两个“同类时间”，而是在回答：

```text
当前这只杯子，最早什么时候真的能开始洗？
```

它要同时满足两件事：

1. 杯子自己已经出现
2. 洗杯机也已经空出来

所以必须取更晚的那个时刻。

### 5. 先别背 `max` / `min`，先看它们各自回答什么问题
这一题最容易卡住的地方，就是下面这两句：

- 一条方案内部，要取 `max`
- 两条方案之间，要取 `min`

它们其实分别在回答两个不同的问题。

#### 问题 1：如果当前已经选定了一条方案，这条方案什么时候结束
假设当前杯子决定“机器洗”。

那么会发生两件事：

1. 当前杯子会在 `wash` 时刻变干净
2. 后面的杯子会递归算出一个“全部处理完的时刻”

这时真正的问题不是“哪个值更好”，而是：

```text
这一条方案，要等到什么时候才算全部结束？
```

答案只能是：

```text
当前杯子和后续杯子，谁更晚结束，就按谁算
```

所以方案内部要写成：

```text
max(当前杯子的完成时间, 后续杯子的完成时间)
```

这里的 `max` 不要理解成“偏爱大值”，而要理解成：

```text
整条方案的收工时间，本来就等于最后结束的那个时刻
```

#### 问题 2：机器洗和自然挥发，最终选哪一条
当前杯子有两种处理方式：

1. 机器洗
2. 自然挥发

这两种方式都会各自产生一条完整方案，也都会各自算出“整条方案什么时候结束”。

这时我们真正要比较的是：

```text
哪一条完整方案更早收工
```

所以最后才会写成：

```text
min(机器洗方案的结束时间, 自然挥发方案的结束时间)
```

也就是说：

- `max` 出现在一条方案内部，是在算“这条方案什么时候结束”
- `min` 出现在两条方案之间，是在选“哪条方案更好”

#### 用一句话压缩
```text
方案内部取 max：因为整条方案要等最后结束的那一步
方案之间取 min：因为我们只保留更早结束的那条方案
```

## 典型例子
下面换一个更有代表性的例子：

```text
arr = [1, 2, 4, 8]
a = 3
b = 5
```

这里有 4 只杯子，比只看 2 只杯子更能体现：

- 杯子是按时间顺序出现的
- 当前选择会不会占用洗杯机
- 洗杯机排队会不会影响后续
- 为什么方案内部一定要取 `max`
- 为什么最后还要在两条方案之间取 `min`

初始状态是：

```text
process(0, 0)
```

表示：
- 从第 0 只杯子开始处理
- 洗杯机一开始空闲

### 第 0 只杯子选机器洗
先算当前杯子什么时候洗完：

```text
wash = max(1, 0) + 3 = 4
```

说明第 0 只杯子会在时间 `4` 洗完。  
后续状态变成：

```text
process(1, 4)
```

假设递归继续算出：

```text
process(1, 4) = 11
```

那这条“第 0 只杯子选机器洗”的完整方案，最终结束时间就是：

```text
max(4, 11) = 11
```

这里的含义是：

- 当前杯子 4 时刻已经干净
- 但后面 3 只杯子要到 11 时刻才全部结束
- 所以整条方案不能算 4，也不能相加成 15，只能算 11

因为：

```text
这一条方案什么时候收工，要等最后那只最慢结束的杯子
```

### 第 0 只杯子选自然挥发
先算当前杯子什么时候自然挥发完：

```text
dry = 1 + 5 = 6
```

因为没占用洗杯机，后续状态还是：

```text
process(1, 0)
```

假设递归继续算出：

```text
process(1, 0) = 12
```

那这条“第 0 只杯子选自然挥发”的完整方案，最终结束时间就是：

```text
max(6, 12) = 12
```

这里的含义是：

- 当前杯子要到 6 时刻才自然挥发完
- 后面 3 只杯子要到 12 时刻才全部结束
- 所以这条方案整体在 12 时刻才收工

### 最后为什么再取 `min`
现在两条完整方案分别得到：

```text
机器洗方案：11
自然挥发方案：12
```

这时候我们才开始做最终选择：

```text
min(11, 12) = 11
```

所以应该选择“第 0 只杯子机器洗”这条路。

### 这个例子里最该记住的点
这个例子真正想说明的是：

```text
max 不是在比较“哪个值更优秀”
而是在求“这一条方案到底什么时候整体结束”
```

以及：

```text
min 不是在某一步里随便选小的
而是在两条完整方案都算完以后，选更早收工的那条
```

## 代码 / 伪代码
```java
int process(int[] arr, int a, int b, int index, int washLine) {
    if (index == arr.length - 1) {
        int wash = Math.max(arr[index], washLine) + a;
        int dry = arr[index] + b;
        return Math.min(wash, dry);
    }

    int wash = Math.max(arr[index], washLine) + a;
    int next1 = process(arr, a, b, index + 1, wash);
    int p1 = Math.max(wash, next1);

    int dry = arr[index] + b;
    int next2 = process(arr, a, b, index + 1, washLine);
    int p2 = Math.max(dry, next2);

    return Math.min(p1, p2);
}
```

## 代码思路说明
第一次看这段代码时，可以不要一下子盯所有变量，而是按下面顺序读。

### 第一步：先看函数在求什么
```java
process(arr, a, b, index, washLine)
```

它不是在求“当前杯子怎么处理”，而是在求：

```text
从第 index 只杯子开始，
当洗杯机最早可用时间是 washLine 时，
后面所有杯子最早什么时候能全部变干净
```

所以这个函数返回的是“一个完整子问题的最优结束时间”。

### 第二步：先看 base case
```java
if (index == arr.length - 1) {
    int wash = Math.max(arr[index], washLine) + a;
    int dry = arr[index] + b;
    return Math.min(wash, dry);
}
```

如果只剩最后一只杯子，事情就很简单了：

- 要么机器洗
- 要么自然挥发

没有后续杯子要考虑，所以直接比较这两种结束时间，取更小的即可。

这里的 `wash` 不是“开始洗的时间”，而是：

```text
最后一只杯子机器洗完的时间
```

### 第三步：一般情况永远只做两件事
对于当前杯子，只尝试两条路。

#### 路 1：当前杯子机器洗
```java
int wash = Math.max(arr[index], washLine) + a;
int next1 = process(arr, a, b, index + 1, wash);
int p1 = Math.max(wash, next1);
```

可以拆成三层理解：

1. 先算当前杯子什么时候洗完  
   `wash`

2. 再递归去算“后面的杯子，在洗杯机下一次可用时间变成 `wash` 后，最早什么时候全部结束”  
   `next1`

3. 把这两部分拼成一条完整方案  
   `p1 = max(wash, next1)`

这里的 `p1`，就代表：

```text
当前杯子选机器洗时，
整条方案最终结束的时间
```

#### 路 2：当前杯子自然挥发
```java
int dry = arr[index] + b;
int next2 = process(arr, a, b, index + 1, washLine);
int p2 = Math.max(dry, next2);
```

同样拆成三层：

1. 先算当前杯子什么时候自然挥发完  
   `dry`

2. 再递归去算后面的杯子最早什么时候全部结束  
   注意这里后续状态还是 `washLine`，因为自然挥发不会占用洗杯机

3. 拼成这一条完整方案  
   `p2 = max(dry, next2)`

### 第四步：最后才在两条方案之间选更优
```java
return Math.min(p1, p2);
```

到这一步，`p1` 和 `p2` 都已经是“完整方案”的结束时间了。  
所以这里只是在问：

```text
机器洗这条路更早结束？
还是自然挥发这条路更早结束？
```

于是取 `min`。

### 用一句流程话串起来
你可以把整段代码理解成：

```text
先决定当前杯子怎么处理
-> 再递归求后面的最优结束时间
-> 每条方案内部都要等“当前杯子 + 后续杯子”都结束，所以取 max
-> 两条完整方案里只保留更早结束的那条，所以取 min
```

## 易错点
- 求的是“所有杯子都干净的最早时刻”，不是所有时间简单相加。
- `washLine` 不是当前杯子的出现时间，而是洗杯机最早可用时间。
- 机器洗的开始时刻要写成 `max(arr[index], washLine)`。
- `max` 表示“这一条方案要等最后结束的那一步”，不是“随便取大值”。
- `min` 表示“两条完整方案之间挑更优的那条”，不是“局部随便挑小值”。

## 记忆点
- 这题的核心状态不是杯子编号，而是 `index + washLine`。
- 方案内部取 `max`，因为整条方案要等最后结束的那一步。
- 方案之间取 `min`，因为我们只保留更早结束的那条方案。
- 机器洗会改写后续状态，自然挥发不会。
