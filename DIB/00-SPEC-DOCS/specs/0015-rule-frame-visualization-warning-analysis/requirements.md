# 框架可视化-预警报告页分析数据 需求文档
> 编号：`0015` | 模块：`rule` | 服务：`dib-agent-service-rule` | 创建时间：`2026-05-14`

---

## 背景

当前 `rule` 服务已经具备：

- 框架结果计算与结果表查询能力
- 框架可视化配置能力
- 预警报告模块下的节点业务规则查询能力

结合页面原型，预警报告页还需要补充一组专门的数据查询接口，供前端展示：

1. 根节点风险等级
2. 统计层级上一级风险分布
3. 统计层级下一级风险节点列表

这些接口统一归口在 `ComFrameVisualizationWarningController` 中，面向“框架可视化 > 预警报告”页面消费。

---

## 目标用户

- 前端开发人员：按页面区域拆分调用接口，分别渲染页头、分布图、风险节点列表
- 业务使用人员：查看某企业某报告期下的预警报告分析结果
- 产品/运营人员：确保页面展示口径与框架可视化配置保持一致

---

## 功能描述

围绕预警报告页，新增 3 个独立查询接口：

1. 查询预警报告头部信息
2. 查询统计层级上一级风险分布
3. 查询统计层级下一级风险节点列表

页面口径已确认如下：

- 红字 `1`：展示框架根节点的风险等级
- 红字 `2`：统计层级的上一级节点风险分布，风险数为 0 的节点不参与统计
- 红字 `3`：统计层级的下一级节点风险列表，风险数为 0 的节点不参与统计

## 所属模块

- 服务：`dib-agent-service-rule`
- 控制器归属：`ComFrameVisualizationWarningController`
- 业务域：`frame visualization warning`

## 核心业务规则

### 1. 查询前置条件

- 前端至少传入 `frameId`
- 查询企业和报告期场景下，还需要传入页面筛选条件对应的维度过滤
- `frameId` 对应框架必须存在
- 当前框架必须已配置可视化
- 当前可视化中的“预警报告”模块必须已启用

### 2. 统计层级来源

- 统计层级固定取自框架可视化配置中的 `statLevel`
- 本期不按左侧树当前点击节点动态改写统计层级

### 3. 根节点风险等级口径

- 页头展示的风险等级固定取框架根节点
- 风险等级的计算结果来源于当前企业、当前报告期、当前筛选条件命中的根节点结果
- 展示等级名称、颜色等信息时，应基于框架可视化等级区间配置映射得到

### 4. 上一级风险分布口径

- 统计对象为“统计层级的上一级节点”
- 每个节点统计自身对应的风险数
- 风险数为 0 的节点不参与返回
- 返回结果用于页面圆环图和图例

### 5. 下一级风险节点列表口径

- 统计对象为“统计层级的下一级节点”
- 每个节点统计自身风险数
- 风险数为 0 的节点不参与返回
- 返回结果用于页面右侧风险节点标签/卡片列表

### 6. 风险数定义

- 风险数的定义应基于当前查询条件下命中的风险记录数
- “0 风险”节点视为无效展示节点，不进入分布和列表结果

### 7. 风险等级映射规则

- 风险等级区间来源于 `com_frame_visualization_level`
- 节点结果分值通过等级区间映射到等级名称、等级颜色
- 若未命中任何等级区间，则该节点等级信息为空，但不影响节点风险数统计能力

### 8. 页面拆分规则

- 页头、上一级分布、下一级节点列表必须拆为 3 个接口
- 不提供整页聚合接口

## 输入参数

### 通用查询参数

| 参数名 | 类型 | 来源 | 说明 |
|------|------|------|------|
| frameId | Long | 请求体 | 当前框架 ID |
| dimFilterList | List<DimFilterBO> | 请求体 | 页面筛选条件，对应企业、报告期等维度过滤 |

说明：

- 企业、报告期等查询条件仍由前端按框架维度配置传入
- 后端不额外提供“查询条件元数据接口”

---

## 输出结果

### 1. 预警报告头部信息

至少包含：

- 根节点 ID
- 根节点编码
- 根节点名称
- 根节点风险等级 ID
- 根节点风险等级名称
- 根节点风险等级颜色

### 2. 上一级风险分布

每条记录至少包含：

- 节点 ID
- 节点编码
- 节点名称
- 风险数
- 占比

### 3. 下一级风险节点列表

每条记录至少包含：

- 节点 ID
- 节点编码
- 节点名称
- 节点层级
- 风险数
- 风险等级名称（如可映射）
- 风险等级颜色（如可映射）

## 用户故事

- 作为前端开发人员，我希望把预警报告页拆成 3 个区域分别请求，这样页面加载和渲染更清晰。
- 作为业务用户，我希望页面顶部看到的是整个框架根节点的风险等级，而不是局部节点的等级。
- 作为业务用户，我希望圆环图只统计真正有风险的上一级节点，不要混入风险数为 0 的节点。
- 作为业务用户，我希望右侧风险节点列表只展示真正有风险的下一级节点，方便快速定位重点问题。

## 验收标准

- [ ] 提供 3 个独立接口：头部信息、上一级分布、下一级节点列表
- [ ] 3 个接口统一归口在 `ComFrameVisualizationWarningController`
- [ ] 根节点风险等级固定取框架根节点
- [ ] 上一级分布只统计 `statLevel` 上一级节点
- [ ] 下一级节点列表只统计 `statLevel` 下一级节点
- [ ] 风险数为 0 的节点，不出现在上一级分布结果中
- [ ] 风险数为 0 的节点，不出现在下一级节点列表中
- [ ] 查询口径统一受当前可视化配置和预警报告模块启用状态约束

---

## 边界条件

| 场景 | 处理方式 |
|------|---------|
| 框架不存在 | 拒绝查询，提示框架不存在 |
| 框架未配置可视化 | 拒绝查询 |
| 预警报告模块未启用 | 拒绝查询 |
| 统计层级不存在上一级 | 头部正常返回，上一级分布返回空列表 |
| 统计层级不存在下一级 | 头部正常返回，下一级节点列表返回空列表 |
| 全部节点风险数均为 0 | 分布和下级节点列表返回空列表 |

---

## 非功能需求

- 性能：优先通过数据库聚合完成统计，不将大结果集全量拉回内存统计
- 一致性：3 个接口的风险等级映射逻辑必须统一
- 可维护性：预警报告页查询能力与现有“节点业务规则查询”解耦，建议使用独立请求/响应模型
