## 3. 升级结论

本次升级并不是简单的版本号提升，而是一次完整的运行时体系切换，核心变化包括：

- JDK 从 8 体系过渡到 17
- Spring Boot 升级到 `3.4.5`
- Spring Framework 升级到 `6.2.x`
- Spring Cloud 升级到 `2024.0.1`
- Spring Cloud Alibaba 升级到 `2023.0.3.2`
- Swagger 体系从旧版 Springfox/早期 springdoc 兼容模式，切换到 `Knife4j 4.5.0 + OpenAPI 3`

升级过程中暴露的问题可以归纳为 8 类：

1. Swagger / Knife4j / springdoc 兼容性问题
2. Spring 6 严格要求方法参数名可反射读取
3. Spring Boot 3 默认禁止循环依赖
4. Devtools + CGLIB 增强下的重复类定义问题
5. Docker 基础镜像差异导致构建脚本失效
6. OAuth2 / Spring Security 认证链行为变化
7. 部分老代码在 JDK 17 下的兼容性调整
8. BOM / 文件编码问题在新编译链下更容易暴露

其中，最值得沉淀为“统一方案”的，是 parent 层面的 Swagger 兼容、`-parameters` 编译参数、JDK17 基线和共享 starter 兼容处理。

## 4. 共性问题与统一解决方案

### 4.1 Swagger / Knife4j / springdoc 兼容问题

#### 现象 1

服务启动时报错：

```text
Invalid mapping pattern detected:
/swagger-ui/**/*swagger-initializer.js
No more pattern data allowed after {*...} or ** pattern element
```

日志在以下服务中均出现过：

- `auth-resource`
- `mdm`

#### 根因

`springdoc` 在不同版本中，对 Spring Boot 3.4.x / Spring Framework 6.2.x 的路径解析适配并不一致。

本轮排查结论是：

- `springdoc 2.8.17` 会触发 `/swagger-ui/**/*swagger-initializer.js` 路径模式报错
- 单纯通过 `spring.mvc.pathmatch.matching-strategy=ant_path_matcher` 规避，不符合“统一在 parent 解决”的目标

#### 统一处理

在 `dib-agent-parent/data-cloud-dependencies/pom.xml` 中，将：

- `springdoc.version` 统一收敛到 `2.6.0`

相关提交：

- `f40e4ac update springdoc`
- `b3cc38c update 解决springdoc路径报错`

#### 现象 2

Swagger 页面打开时再次报错：

```text
java.lang.NoSuchMethodError:
'java.util.List org.springdoc.core.properties.SpringDocConfigProperties.getGroupConfigs()'
```

#### 根因

`Knife4j 4.5.0` 对 `springdoc` 的调用逻辑，和 `springdoc 2.6.0` 的 `getGroupConfigs()` 返回签名不一致：

- 老逻辑按 `List`
- 新版 `springdoc 2.6.0` 为 `Set`

这会导致：

- 启动问题解决了
- 但 `doc.html` 打开时再报 `NoSuchMethodError`

#### 统一处理

不在各服务里各自打补丁，而是在 parent 的 `data-cloud-web-starter` 中提供兼容类：

- [Knife4jOpenApiCustomizer.java](/D:/PROJECT/Project_DIB/dib-agent-parent/data-cloud-web-starter/src/main/java/com/github/xiaoymin/knife4j/spring/extension/Knife4jOpenApiCustomizer.java)

该类的作用：

- 复写 Knife4j 在 OpenAPI 扩展阶段的处理逻辑
- 兼容 `springdoc 2.6.0` 的 `Set<GroupConfig>` 签名
- 统一解决所有 WebMVC / WebFlux 服务的 `doc.html` 访问异常

相关提交：

- `9ee4c70 update，新增 Knife4jOpenApiCustomizer 解决doc.html 报错`

#### 现象 3

部分服务升级后还出现过：

```text
java.lang.NoSuchMethodError:
'void org.springframework.web.method.ControllerAdviceBean.<init>(java.lang.Object)'
```

#### 根因

这是另一类 Swagger 相关依赖未完全对齐导致的 Spring 6 API 兼容问题，说明旧的文档依赖链或旧版配置类仍在影响运行时。

#### 统一处理原则

- 统一使用 `knife4j-openapi3-jakarta-spring-boot-starter`
- 清理旧版 Swagger/Springfox 风格配置
- 尽量删除 Boot 2.x 时期自定义的旧资源配置类
- 优先在 parent 依赖层统一收敛，而不是在单服务用路径匹配参数兜底

#### 补充配置

部分服务保留了：

```yaml
springdoc:
  override-with-generic-response: false
```

该配置的作用是关闭 springdoc 自动用通用返回结构覆盖接口响应描述，避免升级后文档展示异常或泛型响应干扰接口展示。

### 4.2 Spring 6 方法参数名反射问题

#### 现象

接口运行时报错：

```text
Name for argument of type [java.lang.String] not specified,
and parameter name information not available via reflection.
Ensure that the compiler uses the '-parameters' flag.
```

已在以下链路明确出现：

- gateway OAuth callback
- report 的接口参数绑定

#### 根因

Spring 6 / Boot 3 对如下写法依赖更强：

```java
@RequestParam String code
@PathVariable String id
```

如果编译产物里没有保留方法参数名，运行时就无法从反射中拿到 `code`、`id` 等真实参数名。

#### 统一处理

在 parent 的 Maven 编译插件中全局开启 `-parameters`：

- [dib-agent-parent/pom.xml](/D:/PROJECT/Project_DIB/dib-agent-parent/pom.xml)

统一增加：

- `maven.compiler.parameters=true`
- `<parameters>${maven.compiler.parameters}</parameters>`
- `compilerArgs` 中显式加入 `-parameters`

相关提交：

- `aece11d update 开始编译读取参数名`

#### 结论

升级后推荐规范：

1. 全局编译必须开启 `-parameters`
2. 关键控制器接口建议显式写参数名，例如：

```java
@RequestParam("code") String code
```

这样即使某次构建链路漏掉编译参数，接口仍更稳。

### 4.3 Spring Boot 3 默认禁止循环依赖

#### 现象

启动时报错：

```text
Relying upon circular references is discouraged and they are prohibited by default.
```

已明确出现在：

- extract
- data
- rule

#### 根因

Boot 2.x 时代部分服务依赖了“字段互相注入也能启动”的默认宽松行为，升级到 Boot 3 后，循环依赖不再被默认容忍。

#### 处理方式

这类问题没有一刀切配置，最终采取“按代码关系拆环”的方式：

##### extract

在：

- [ComExtractDocTypeBusServiceImpl.java](/D:/PROJECT/Project_DIB/dib-agent-service-extract/dib-agent-service-extract-web/src/main/java/com/dib/agent/extract/web/service/extract/impl/ComExtractDocTypeBusServiceImpl.java)

对 `ComExtractDocTypeService` 使用 `@Lazy` 延迟注入，打断 `docTypeBusService <-> docTypeService` 创建环。

相关提交：

- `d16fd825 update 解决循环依赖报错`

##### data

在：

- [ComDataIndexRefItemAggregate.java](/D:/PROJECT/Project_DIB/dib-agent-service-data/dib-agent-service-data-web/src/main/java/com/dib/agent/data/web/aggregate/index/ComDataIndexRefItemAggregate.java)
- [ComDataIndexValueFieldAggregate.java](/D:/PROJECT/Project_DIB/dib-agent-service-data/dib-agent-service-data-web/src/main/java/com/dib/agent/data/web/aggregate/index/ComDataIndexValueFieldAggregate.java)

对相互依赖的聚合对象引入 `@Lazy`，打断：

- `indexInfoAggregate`
- `indexRefItemAggregate`
- `indexValueFieldAggregate`

之间的创建环。

相关提交：

- `830f492d update 解决循环依赖问题`

##### rule

在：

- [ComRuleBizAggregate.java](/D:/PROJECT/Project_DIB/dib-agent-service-rule/dib-agent-service-rule-web/src/main/java/com/dib/agent/rule/web/aggregate/ComRuleBizAggregate.java)

不再字段注入 `BizRuleCalculator`，而是运行时通过 `SpringContextUtil.getBean(BizRuleCalculator.class)` 按需获取，彻底打断：

- `ComRuleBizAggregate`
- `BizRuleCalculator`

的双向依赖链。

相关提交：

- `b553e58f update 解决循环依赖报错`

#### 原则总结

升级到 Boot 3 后，不建议再用：

```yaml
spring.main.allow-circular-references=true
```

做兜底。优先顺序应是：

1. 拆职责
2. `@Lazy`
3. 按需从上下文获取 Bean

### 4.4 Devtools + CGLIB 重复类定义问题

#### 现象

extract 启动时报错：

```text
attempted duplicate class definition for
FeignConfiguration$$SpringCGLIB$$0
```

#### 根因

共享 `FeignConfiguration` 在 Devtools `RestartClassLoader` 场景下被增强时，`@Configuration` 默认的代理增强与重复加载叠加，导致 CGLIB 生成同名增强类。

#### 统一处理

在共享的 mdm feign 配置中关闭配置类代理：

- [FeignConfiguration.java](/D:/PROJECT/Project_DIB/data-cloud-mdm/data-cloud-mdm-feign/src/main/java/com/dib/data/cloud/feign/s2/mdm/config/FeignConfiguration.java)

改为：

```java
@Configuration(proxyBeanMethods = false)
```

这不是 extract 私有问题，而是共享 feign 配置在 Boot 3 + Devtools 环境下的兼容改法。

### 4.5 Docker 基础镜像差异问题

#### 现象

Jenkins 构建镜像时报错：

```text
sed: can't read /etc/apt/sources.list: No such file or directory
```

#### 根因

服务使用的基础镜像为：

```dockerfile
FROM 10.0.6.183:8088/c1/amazoncorretto:17
```

该内部镜像并不保证一定是带标准 Debian `sources.list` 的布局，因此原来依赖：

- `/etc/apt/sources.list`
- `apt-get update`
- 手工替换 Debian 源

的脚本并不稳。

#### 处理策略

本轮没有切换到新的公开基础镜像，而是保守处理为：

1. 保留字体、LibreOffice、`unoconv` 相关注释
2. 将依赖 `/etc/apt/sources.list` 的脚本注释掉或回退
3. 保留字体目录挂载与 `fc-cache` / `fc-list` 验证
4. 按服务实际需要保留 JVM 参数与日志目录

相关提交示例：

- mdm：`8460a4f08 update dockerFile`，后续又有回退
- extract：`01a9c270 update dockerFile & swagger`、`64bb8c86 update，解决jvm参数问题`
- data：`728488b2 update dockerFile`
- report：`08afa29c update dockerFile`
- rule：`a5fd1961 update dockerFile`
- dg：`d66d1b8 update dockerFile`

#### 经验结论

如果后续还要继续装系统包，建议先确认基础镜像真实发行版，再决定：

- 用 `apt`
- 还是 `yum` / `microdnf`
- 还是直接切换成明确可控的 Debian/Ubuntu 变体

在未确认前，不建议直接写死 `/etc/apt/sources.list`。

### 4.6 OAuth2 / Spring Security 行为变化

#### auth-server 登录失败现象

日志出现：

```text
javax.crypto.BadPaddingException: Padding error in decryption
```

并且调用栈进入：

- `CustomerDaoAuthenticationProvider`
- `BasicAuthenticationFilter`

#### 分析结论

升级到 Spring Security 6 后，默认 `httpBasic()` 会引入标准 Basic 认证过滤链，和系统原有“前端公钥加密密码 + 自定义认证 provider”的登录流程发生干扰，导致密码解密链路异常。

#### 处理方式

在：

- [WebSecurityConfig.java](/D:/PROJECT/Project_DIB/data-cloud-auth-server/src/main/java/com/dib/data/cloud/auth/server/component/oauth/config/WebSecurityConfig.java)

中保留表单登录、自定义 provider、自定义 details source，但移除默认：

```java
httpBasic(Customizer.withDefaults())
```

这样不会影响原系统的表单登录认证，反而是恢复到和历史系统一致的认证入口。

相关提交：

- `f4472b3 update swagger httpbasic`
- `b85818c update 解决启动报错`

#### 兼容旧 OAuth2 客户端

为兼容历史 OAuth 客户端数据结构，新增了：

- `LegacyOauthClientRegisteredClientRepository`

相关提交：

- `e31ab0f update，兼容 oauth2`

### 4.7 BOM / 编码问题

#### 现象

多个服务在升级后都有：

- `encode err`
- `bom encode err`

相关提交，如：

- extract：`5071dd11`
- data：`1d948777`
- report：`5909305a`
- rule：`b2b6444c`

#### 根因

JDK17 + 新编译链对源码编码、BOM 头、旧文件中的隐式字符更敏感，部分：

- Feign 模块
- 枚举类
- 控制器
- 请求模型

在编译、序列化或 IDE 保存后更容易暴露问题。

#### 处理原则

- 统一使用 UTF-8
- 源码尽量去 BOM
- 对历史中文注释乱码，只在必要时修复，不把“注释乱码”和“运行问题”混为一谈

### 4.8 Gateway 文档聚合与启动日志差异

#### 现象

gateway 出现过两个独立问题：

1. `doc.html` 打不开或聚合配置不完整
2. 启动时没有像其他服务一样打印 `doc.html` 地址

#### 根因

gateway 使用的是：

- `knife4j-gateway-spring-boot-starter`
- WebFlux / Gateway 模式

它和普通 Servlet 服务：

- `knife4j-openapi3-jakarta-spring-boot-starter`
- `WebAutoConfiguration`

走的不是同一条自动打印链路。

#### 处理方式

gateway 最终保留了两类兼容能力：

1. `SwaggerProvider` / `SwaggerController`，兼容旧的 `swagger-resources` 初始化请求
2. `Knife4jStartupLogger`，补齐启动日志中的 `doc.html` 地址输出

日志验证结果：

```text
Swagger Pretty UI地址: http://localhost:20000/doc.html
```

相关提交：

- `c879406 update controller param err`
- `e027437 todo，临时解决编译问题 CustomBlockingLoadBalancerClient`
- `738fa3a revert yml 删除多余的doc配置`

说明：

- gateway 的文档配置在这轮中有过“先补、再精简”的过程
- 最终思路是尽量不堆过多重复的 YML doc 配置，而是保留兼容代码与基础输出能力

## 6. 统一改造清单

后续如果还有服务需要升级到 `dib-agent-parent 3.0.0`，建议按下面顺序执行：

1. 升级 parent 版本到 `3.0.0-SNAPSHOT`
2. 确认 JDK 基线为 17
3. 对齐 Swagger 依赖到 `knife4j-openapi3-jakarta-spring-boot-starter`
4. 不再额外引入旧版 Swagger / Springfox 配置
5. 清理 Boot 2.x 风格的 Security / MVC / ResourceServer 旧配置
6. 重新检查所有 `@RequestParam String x`、`@PathVariable String x` 绑定接口
7. 全量排查循环依赖
8. 检查共享 `@Configuration` 是否需要 `proxyBeanMethods = false`
9. 检查 Dockerfile 是否假设基础镜像一定支持 `apt`
10. 统一清理源码 BOM 与编码问题

## 8. 最终结论

本次升级的真正价值不只是把版本号升到了 Spring Boot 3.4.5，而是把以下能力沉淀到了 parent 和共享模块中：

- 统一的 Swagger 兼容方案
- 统一的 `-parameters` 编译策略
- 统一的 JDK17 依赖基线
- 统一的 Docker 构建收口思路
- 统一的循环依赖拆解经验

这意味着后续同类服务再升级到 `dib-agent-parent 3.0.0` 时，不需要再从零踩一遍坑，重点只需要处理：

- 本服务自己的业务循环依赖
- 本服务自己的 Security 特殊逻辑
- 本服务自己的 Docker 特殊依赖

其余大部分基础兼容问题，已经可以直接复用本轮的处理成果。
