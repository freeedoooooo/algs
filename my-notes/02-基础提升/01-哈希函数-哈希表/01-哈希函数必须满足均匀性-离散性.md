## 一句话结论
哈希函数的核心任务，是把任意输入稳定映射到固定范围的输出空间里，并尽量把结果打散。
课程里强调的“均匀性、离散性”，本质上是在说：输出要分布均匀、输入微小变化时输出要明显变化，这样哈希表、布隆过滤器、一致性哈希等结构才好用。

## 核心知识点
- 哈希函数输入空间通常很大，输出空间通常是有限范围。
- 相同输入一定得到相同输出，这是确定性要求。
- 均匀性：不同输入映射到输出空间时，要尽量平均分布。
- 离散性：输入稍有变化，输出就应明显变化，避免结果过于接近。
- 哈希函数本身不是为了“加密”，而是为了“高效分桶、快速定位、稳定分流”。

## 图解
可以把哈希函数理解成“把海量输入扔进有限编号桶里”：

```text
输入：
apple
apply
banana
cat
dog

经过 hash()

输出桶编号：
3
17
8
3
12
```

### 1. 什么叫均匀性
如果输出足够均匀，那么很多数据会比较平均地落到不同桶里：

```text
桶0: 12个
桶1: 10个
桶2: 11个
桶3: 9个
...
```

如果均匀性很差，就会变成：

```text
桶0: 1个
桶1: 0个
桶2: 0个
桶3: 97个
```

这会导致某些桶特别拥挤，哈希表查询效率明显下降。

### 2. 什么叫离散性
如果两个输入只差一点点：

```text
hello
hellp
```

好的哈希函数不会让输出也只差一点点，而是会把结果充分打散：

```text
hello -> 18472613
hellp -> 90514402
```

可以把它理解为：

```text
输入相近
输出不应“跟着相近”
```

这样才能减少模式聚集和碰撞风险。

## 解题思路
### 1. 哈希函数到底在做什么
哈希函数本质上是一个映射：

```text
超大输入空间 -> 有限输出空间
```

例如：

- 任意字符串
- 任意文件内容
- 任意对象序列化结果

最终都可以被映射成一个整数、一个桶下标、一个固定长度摘要。

### 2. 为什么必须强调均匀性
因为输出空间通常是有限的，所以碰撞不可避免。

既然碰撞一定会存在，那么工程上最重要的就不是“绝不碰撞”，而是：

```text
让碰撞尽量平均地发生
不要集中挤在少数位置
```

这就是均匀性的意义。

对哈希表来说，均匀性好意味着：

- 链表 / 红黑树长度更可控
- 查找、插入、删除更稳定

对布隆过滤器来说，均匀性好意味着：

- 位图被设置的位置更分散
- 误判率更可控

### 3. 为什么必须强调离散性
离散性强调的是“打散能力”。

如果输入有轻微变化，而输出变化很小，那么很容易出现：

- 某类相似 key 总落在相近位置
- 数据分布出现模式性偏斜
- 被某些特殊构造输入集中攻击

所以好的哈希函数要做到：

```text
输入哪怕只改一位
输出也尽量整体变化
```

这和密码学里的“雪崩效应”有点像，但这里重点是工程上的分布效果，而不是安全加密强度。

### 4. 还要注意哪些基本性质
除了均匀性、离散性，实际使用中还默认要求：

- 确定性：相同输入每次都得到相同输出
- 高效性：计算不能太慢
- 固定输出形式：便于继续取模、分桶或存储

### 5. 为什么这一页是后续章节的基础
因为后面这章很多内容都建立在哈希函数之上：

- 哈希表：靠哈希函数把 key 分桶
- 布隆过滤器：靠多个哈希函数把样本映射到位图
- 一致性哈希：靠哈希函数把节点和数据映射到环上
- 大数据分流：靠哈希函数把海量样本切到不同机器或文件

所以先理解“均匀性、离散性”，后面很多工具才不会只剩结论。

## 复杂度
- 单次计算时间复杂度：通常视实现而定，常写作 `O(len)`，`len` 为输入长度
- 额外空间复杂度：通常为 `O(1)` 或与输出长度相关

如果把哈希函数当作黑盒使用，在大多数算法题里通常默认它是“常数级可调用组件”。

## 典型例子
### 例 1：哈希表分桶
假设数组长度是 `10`，我们把 `hash(key) % 10` 作为桶下标。

如果哈希函数足够均匀，那么大量 key 会尽量平均地落到：

```text
0~9
```

每个桶中元素数量接近，查询性能稳定。

### 例 2：相似输入被打散
两个字符串：

```text
user_000001
user_000002
```

虽然肉眼看只差最后一位，但经过好的哈希函数后，结果应当差异明显，而不是总落在相邻桶。

## 易错点
- 均匀性不是说“绝不碰撞”，而是说碰撞尽量平均分散。
- 离散性不是说输出要随机，而是说对输入变化要有足够强的打散能力。
- 哈希函数和加密函数不是一个概念，别混为一谈。
- 好哈希函数也不保证哈希表最坏情况一定是 `O(1)`，只是让平均表现更稳定。
- 取模分桶后的均匀性，还会受到桶数量设计是否合理的影响。

## 代码 / 伪代码
工程里通常不会手写底层哈希算法，而是调用成熟实现，再做分桶：

```java
int index = Math.abs(hash(key)) % bucketSize;
```

如果把哈希函数抽象成接口，可以这样理解：

```java
int hash(String key) {
    // 把任意输入稳定映射到整数空间
    return digest(key);
}
```

在后续数据结构里，更关心的是“如何使用哈希值”，而不是从零实现 MD5、SHA 或 MurmurHash。

## 记忆点
- 哈希函数 = 把大输入空间映射到有限输出空间。
- 均匀性：结果尽量平均分布。
- 离散性：相近输入也要被充分打散。
- 哈希表、布隆过滤器、一致性哈希都建立在这两个性质之上。
