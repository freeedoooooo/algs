## 一句话结论
哈希表实现的核心不只是“数组加链表/红黑树”，更重要的是理解为什么它在平均情况下能做到近似 `O(1)`，以及扩容为什么不会把整体复杂度拖垮。
这题真正要掌握的是：哈希分桶、冲突处理、负载因子、扩容与再哈希、以及“均摊复杂度”这个分析口径。

## 核心知识点
- 哈希表底层通常是“桶数组 + 冲突处理结构”。
- 冲突处理常见有拉链法，也有开放寻址法；课程语境里通常先讲拉链法。
- 当元素过多、桶太拥挤时，需要扩容并重新分布元素。
- 单次扩容代价很高，但扩容次数少，所以摊到单次操作上仍近似 `O(1)`。
- 工程实现里还会引入负载因子、渐进式 rehash、树化桶等优化。

## 图片转写 / 题意还原
原始笔记只留下了两句关键信息：

- 扩容次数的复杂度接近 `O(logN)`
- 每次扩容代价是 `O(N)`，但单次均摊后接近常数

把这题补成完整说明，就是：

- 如果让你自己设计一个哈希表，需要支持：
  - `put(key, value)`
  - `get(key)`
  - `remove(key)`
- 你要考虑：
  - 哈希值如何映射到桶
  - 发生冲突时怎么办
  - 桶太满时如何扩容
  - 为什么平均复杂度仍然可以认为接近 `O(1)`

所以这题并不只是问“哈希表怎么写”，而是在问：

```text
哈希表为什么快
冲突怎么处理
扩容为什么没有想象中那么贵
```

## 图解
可以把哈希表理解成一排桶：

```text
index: 0   1   2   3   4   5   6   7
       |   |   |   |   |   |   |   |
```

每个 key 先经过哈希函数，再映射到某个桶：

```text
index = hash(key) % bucketSize
```

例如：

```text
apple  -> 3
banana -> 5
cat    -> 3
dog    -> 1
```

这时桶 3 会出现冲突：

```text
bucket[3] -> apple -> cat
```

### 1. 为什么需要扩容
如果越来越多元素都挤在少数桶里：

```text
bucket[3] -> a -> b -> c -> d -> e -> ...
```

那查询效率就会从接近 `O(1)` 退化。

所以当负载因子过大时，需要把桶数组变大，例如：

```text
8 个桶 -> 16 个桶
```

然后把旧元素重新计算位置并搬过去。

### 2. 为什么扩容后性能会恢复
因为桶数量变多后，同样一批元素会被分散到更多位置：

```text
扩容前：很多元素挤在少数桶
扩容后：每个桶平均元素更少
```

这就是哈希表能持续保持高效的关键。

## 解题思路
### 1. 哈希表的基本结构
最基础的实现通常包含：

- 一个桶数组 `table`
- 每个桶里挂一条链表，或别的冲突处理结构

对于插入：

1. 先算 `hash(key)`
2. 再算桶下标
3. 到对应桶里找有没有同 key
4. 有就更新，没有就插入

对于查询：

1. 先定位桶
2. 再在桶内查找目标 key

### 2. 冲突为什么不可避免
因为：

```text
输入空间很大
桶数组长度有限
```

只要元素足够多，不同 key 映射到同一桶就是必然的。

所以哈希表设计的重点从来不是“消灭冲突”，而是：

- 让冲突尽量少
- 冲突发生后也能高效处理

### 3. 为什么要看负载因子
负载因子通常写作：

```text
loadFactor = size / bucketSize
```

它表示平均每个桶装了多少元素。

如果这个值过大，就意味着：

- 桶内链越来越长
- 查询、插入、删除都会变慢

所以当负载因子超过阈值时，就触发扩容。

### 4. 扩容为什么单次很贵
扩容时通常要做两件事：

1. 申请更大的桶数组
2. 把原有所有元素重新分发到新桶里

这一步显然是 `O(N)` 级别，因为几乎所有元素都要动一次。

### 5. 为什么均摊下来仍接近 O(1)
这正是原始笔记最想强调的地方。

假设每次容量翻倍：

```text
1 -> 2 -> 4 -> 8 -> 16 -> ... -> N
```

那么扩容次数大约是：

```text
O(logN)
```

但更重要的是，总搬运成本不是 `N * logN` 那样逐元素反复全量搬很多次地理解，而应该从“每个元素被搬迁的总次数有限”去看。

经典均摊分析里：

- 插入操作大多数时候只是普通放入
- 只有少数时刻触发扩容
- 所有扩容搬迁总代价加起来是线性的量级

所以把所有操作加总后再平均，单次插入仍可认为是均摊 `O(1)`。

### 6. 为什么原始笔记会提 Java 的“离线扩容”
工程语言里的容器实现，有时不会在一次操作里把所有数据立刻重排完，而会通过更平滑的方式减少单次卡顿。

从算法题角度，我们通常记：

- 理论模型：均摊 `O(1)`
- 单次触发扩容：最坏 `O(N)`

从工程实现角度，则还可能进一步做：

- 渐进式 rehash
- 后台搬迁
- 桶树化

所以“理论分析”和“工程体验”要分开看。

## 复杂度
### 理想平均情况
- `put`：均摊 `O(1)`
- `get`：平均 `O(1)`
- `remove`：平均 `O(1)`

### 极端最坏情况
- 如果大量 key 冲突到同一桶：可能退化到 `O(N)`
- 单次触发扩容时：可能出现 `O(N)` 的搬迁成本

### 空间复杂度
- `O(N)`，外加桶数组本身的容量开销

## 典型例子
假设当前桶数组长度是 `8`，阈值是负载因子 `0.75`。

也就是说当元素数达到：

```text
8 * 0.75 = 6
```

左右时，就准备扩容。

流程可以理解为：

1. 前几个元素插入都很快
2. 当元素数继续增长，桶太挤了
3. 扩容到 `16`
4. 所有旧元素重新分桶
5. 后续一段时间查询和插入又恢复很快

这说明：

```text
哈希表不是永远不贵
而是“偶尔很贵，长期平均很便宜”
```

## 易错点
- 不要把“单次最坏复杂度”和“均摊复杂度”混为一谈。
- 扩容不是简单复制数组，而是要重新计算桶位置并搬迁元素。
- 哈希表平均快，前提是哈希函数分布合理、负载因子控制得当。
- 冲突不可避免，设计重点是冲突处理而不是幻想零冲突。
- 工程实现里的体验接近 `O(1)`，不代表理论上每次操作都严格是 `O(1)`。


## 记忆点
- 哈希表快在“先定位桶，再桶内处理”。
- 冲突不可避免，关键是处理冲突和控制负载因子。
- 扩容单次很贵，但长期均摊仍接近 `O(1)`。
- 算法分析看均摊复杂度，工程实现还会进一步优化扩容过程。
