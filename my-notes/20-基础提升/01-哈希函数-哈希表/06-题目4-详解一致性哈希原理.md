## 一句话结论
一致性哈希解决的核心问题是：在分布式系统里，节点数量发生变化时，如何尽量少地迁移数据，同时又保持整体分布尽量均衡。
它通过“哈希环 + 顺时针找节点”的方式，把大规模重映射压缩成局部迁移；再通过虚拟节点技术，把原本不均匀的物理节点分布拉平。

## 核心知识点
- 普通取模分片在节点数变化时，会导致几乎所有数据重新定位。
- 一致性哈希把数据和机器都映射到同一个哈希环上。
- 一个数据归属“顺时针方向遇到的第一个节点”。
- 增删节点时，只会影响环上一小段区间的数据迁移。
- 虚拟节点的作用是让负载更均匀，并支持不同机器按能力分配不同权重。

## 图片转写 / 题意还原
原始笔记里保留了几句关键提示：

- 数量少的时候怎么均分，怎么负载均衡？
- 解决方案：虚拟节点技术
- 每个物理节点对应很多虚拟节点
- 性能强的机器可以分配更多虚拟节点

把这题补成完整说明，就是：

- 在分布式缓存、分布式数据库、分布式存储中，需要把大量数据分配到多台机器上
- 如果直接用：

```text
hash(key) % 机器数量
```

那么一旦机器增减，绝大多数数据的位置都会改变
- 这会导致缓存大面积失效、数据搬迁成本很高、系统抖动很严重
- 一致性哈希的目标就是在节点变化时，只迁移必要的一小部分数据

因此这篇笔记真正要回答的是：

```text
分布式系统里
怎么把数据映射到机器
才能在扩容缩容时少搬数据
并且尽量负载均衡
```

## 图解
### 1. 普通取模为什么有问题
假设有 3 台机器：

```text
server0
server1
server2
```

分配规则是：

```text
hash(key) % 3
```

这时某个 key 可能去 `server1`。

如果后来扩容成 4 台机器：

```text
hash(key) % 4
```

那么同一个 key 的结果大概率会变。

这意味着：

```text
机器数一变
大量 key 全部重分配
```

### 2. 一致性哈希环怎么做
把哈希空间想成一个首尾相连的环：

```text
0 --------------------------------> MAX
^                                   |
|___________________________________|
```

把机器节点也哈希到环上，例如：

```text
serverA -> 20
serverB -> 80
serverC -> 150
```

再把数据 key 也映射到环上：

```text
key1 -> 30
key2 -> 100
key3 -> 170
```

归属规则是：

```text
从 key 所在位置开始顺时针找
遇到的第一个机器节点
就是它的归属机器
```

例如：

- `key1(30)` 归 `serverB(80)`
- `key2(100)` 归 `serverC(150)`
- `key3(170)` 顺时针绕回去，归 `serverA(20)`

### 3. 为什么节点变化时只影响局部
如果新加一台机器：

```text
serverD -> 120
```

那么只会影响：

```text
120 往前那一段区间
```

原本属于 `serverC` 的一部分数据，现在改归 `serverD`。

其他区域的数据完全不用动。

这就是一致性哈希最重要的价值：

```text
局部迁移
而不是全量洗牌
```

### 4. 为什么还要虚拟节点
如果物理机器数量少，直接落到环上的点可能很稀疏：

```text
serverA -------- serverB -- serverC
```

这样不同机器负责的区间长度差异会很大，负载不均衡。

解决办法是：

- 每台物理机器映射成很多个虚拟节点
- 虚拟节点均匀撒在环上

例如：

```text
serverA#1, serverA#2, ..., serverA#1000
serverB#1, serverB#2, ..., serverB#1000
```

这样每台机器在环上就不再只占一个点，而是占很多个分散位置，整体负载会平滑很多。

## 解题思路
### 1. 普通取模分片的根本缺陷
普通分片方式：

```text
bucket = hash(key) % N
```

的问题是它把：

```text
机器数量 N
```

直接写进了分配公式里。

一旦 `N` 变化，从模运算的角度看，原来的分配规则整体变了，所以几乎所有 key 都可能换机器。

### 2. 一致性哈希的核心思想
它不再直接按“机器下标”做取模，而是：

1. 把机器映射到哈希空间
2. 把数据也映射到同一个哈希空间
3. 通过环上的相对位置决定归属

这样机器增减时，不是整个公式重写，而只是环上多了或少了几个点。

### 3. 为什么一致性哈希适合缓存和分布式数据库
因为这些系统最怕：

- 扩容一次，缓存命中率瞬间跌光
- 加机器后，大量数据要全量重分布
- 系统在搬迁期抖动严重

一致性哈希把迁移范围压缩到了局部，因此特别适合：

- 分布式缓存
- 分布式 KV
- 分布式数据库分片
- 对象存储或文件路由

### 4. 虚拟节点为什么必不可少
理论上的哈希环很好，但如果机器数量本来就少，物理节点在环上的位置可能非常不均匀。

这会造成：

- 某些机器负责的区间很大，压力很重
- 某些机器负责的区间很小，资源浪费

虚拟节点的作用就是：

```text
把一台机器拆成很多个逻辑点
分散到整张环上
从而让总体分配更均匀
```

### 5. 为什么高性能机器可以给更多虚拟节点
如果机器配置不同：

- 强机器可以多承担一点流量
- 弱机器少承担一点

那就可以通过“虚拟节点个数”来加权：

- 强机器给更多虚拟节点
- 弱机器给更少虚拟节点

这样不需要改核心算法，就能做加权负载均衡。

## 复杂度
如果使用有序结构维护环上节点，例如 `TreeMap`：

- 查找某个 key 对应机器：`O(log M)`
- 新增 / 删除一个节点：`O(log M)`

其中 `M` 是哈希环上节点总数；如果用了虚拟节点，`M` 可能远大于物理机器数。

## 典型例子
假设当前有 3 台机器：

```text
A, B, C
```

数据按一致性哈希分布。

现在新增机器 `D`：

- 不是所有 key 全部重算后重分布
- 只会把“原本顺时针归属于某个旧节点的一部分区间”切给 `D`

所以系统代价是：

```text
局部数据迁移
```

而不是：

```text
全量缓存失效 / 全量数据重洗牌
```

## 易错点
- 一致性哈希解决的不是“绝对平均”，而是“节点变化时尽量少迁移”。
- 没有虚拟节点时，负载均衡效果可能很差。
- 虚拟节点不是新机器，它只是物理机器在环上的多个逻辑映射点。
- 一致性哈希和普通 `hash % N` 的本质区别，在于是否把机器数直接写进分配公式。
- 这套方法更偏工程分布式设计，不是单纯的哈希表技巧。

## 代码 / 伪代码
可以把哈希环抽象成一个有序映射：

```java
TreeMap<Long, Server> ring;
```

查找某个 key 归属机器的伪代码：

```java
Server getServer(String key) {
    long hash = hash(key);
    Map.Entry<Long, Server> entry = ring.ceilingEntry(hash);
    if (entry == null) {
        entry = ring.firstEntry();
    }
    return entry.getValue();
}
```

如果使用虚拟节点，则不是往环里放一条机器记录，而是放很多条：

```java
for (int i = 0; i < virtualCount; i++) {
    long hash = hash(serverName + "#" + i);
    ring.put(hash, server);
}
```

## 记忆点
- 普通取模分片最大问题：机器数变化时几乎全量迁移。
- 一致性哈希：数据和机器都映射到哈希环上。
- 数据归顺时针第一个节点。
- 虚拟节点负责把负载打散、做加权均衡。
