## 一句话结论
这题是典型的大数据哈希分流题：如果数据规模太大，不能直接一次性用哈希表统计，就先把所有数字按哈希分到多个小文件里，再分别统计每个文件中的最高频数字，最后做一次全局比较。
核心思想不是“直接求众数”，而是“利用相同数字一定会被分到同一个桶里”这个性质，把大问题拆成多个内存可承受的小问题。

## 核心知识点
- 输入范围是 32 位无符号整数，值域很大，但真正难点通常是数据量极大。
- 如果数据总量能一次装进内存，直接哈希表计数即可。
- 如果数据量装不下内存，就要用哈希函数做分流。
- 同一个数字经过同一个哈希函数后，结果一定相同，所以它的所有出现都会落到同一个桶。
- 分桶后，每个桶可以单独统计局部最高频，再在所有桶的候选里取全局最大。

## 图片转写 / 题意还原
原始笔记只有标题，但这页其实是在讲一个非常标准的大数据题模型。

把题目补成完整描述，就是：

- 给定一批 32 位无符号整数
- 数据量可能非常大，大到一台机器无法直接把“数字 -> 次数”的完整哈希表一次性放进内存
- 目标是找出出现次数最多的那个数字，或者至少找出最高频数字及其出现次数

这题重点不在数值范围本身，而在于：

```text
值域很大
数据量也很大
直接统计可能顶爆内存
```

因此真正考的是：

- 如何借助哈希函数把数据切开
- 如何保证同一个数字不会被切散到不同子问题里
- 如何在分治后仍然得到全局正确答案

## 图解
假设有一大堆数字流：

```text
7, 12, 7, 9, 3, 12, 7, 5, 9, 9, ...
```

如果直接统计，理想情况是：

```text
7  -> 3次
12 -> 2次
9  -> 3次
3  -> 1次
5  -> 1次
```

但当数据量大到内存不够时，可以先按哈希值分流：

```text
num -> hash(num) % 100
```

于是所有数据被拆进 100 个小文件：

```text
file0
file1
file2
...
file99
```

### 为什么这样做是安全的
因为同一个数字例如 `7`：

```text
hash(7) % 100
```

结果永远固定，所以所有 `7` 都会进同一个文件。

这意味着：

```text
同一个数字的出现次数
不会被分散到多个桶里
```

所以我们只需要在每个小文件里统计局部最高频，再汇总比较即可。

## 解题思路
### 1. 小数据量时的直接做法
如果数据总量不大，直接用哈希表：

```java
Map<Integer, Integer> countMap
```

边读边统计，最后扫一遍 `countMap` 找最大值即可。

### 2. 大数据量时为什么不能直接做
如果数据量极大，例如：

- 数十亿条记录
- 可用内存只有几百 MB

那么直接维护一个超大哈希表就可能内存不够。

此时问题不是算法不会，而是：

```text
答案思路对
但机器装不下
```

### 3. 用哈希函数先分流
第一阶段把所有数字写入多个桶文件：

```text
bucket = hash(num) % M
```

这里 `M` 的选择目标是：

- 让每个桶的数据量都足够小
- 小到后续可以单独读入内存统计

### 4. 为什么分流后还能找全局最高频
这是最关键的一点。

因为哈希函数有确定性：

```text
同一个 num
一定去同一个 bucket
```

所以某个数字的总次数不会被拆散。

因此如果全局最高频数字是 `x`，那么：

- 它一定会在某个具体桶里成为这个桶中的高频候选
- 只要我们拿到每个桶的局部冠军，再做全局比较，就不会漏掉它

### 5. 两阶段流程
第一阶段：分桶

1. 逐条读取原始数据
2. 对每个数字计算 `hash(num) % M`
3. 写入对应的小文件

第二阶段：桶内统计

1. 每次读取一个小文件
2. 用内存哈希表统计该文件中的词频
3. 找出该文件的最高频数字和次数
4. 和全局答案比较并更新

### 6. 为什么这题属于哈希函数专题
因为这题的关键不是普通频次统计，而是利用了哈希函数的两个工程价值：

- 确定性：相同数字进同一桶
- 均匀性：尽量让数据分布到多个桶，避免某个桶过大

## 复杂度
如果设：

- 总数据量为 `N`
- 分桶数量为 `M`

那么整体上：

- 时间复杂度：`O(N)` 级别，外加文件读写常数
- 空间复杂度：单次只需要容纳一个桶的统计哈希表，约为单桶大小

更准确地说，这题是用：

```text
多一次磁盘分发 / 读回
换取内存可承受
```

## 典型例子
假设现在有海量整数，内存最多只能容纳 1/100 的数据量。

可以这样做：

1. 把所有数按 `hash(num) % 100` 分成 100 个文件
2. 逐个处理每个文件
3. 在每个文件里用普通哈希表统计频次
4. 拿到这个文件里的：
   - 出现最多的数字
   - 该数字的次数
5. 扫完 100 个文件后，取最大值

最终得到全局最高频数字。

## 易错点
- 不要被“0 ~ 2^32-1”这个值域吓住，这题真正难的是数据量，不是数值比较。
- 分桶数量 `M` 不是越大越好，要和磁盘文件数量、单桶大小、内存上限一起权衡。
- 只有使用同一个哈希函数和同一个取模规则，同一个数字才会稳定落入同一桶。
- 每个桶里找的是“局部最高频”，最后还必须做一次全局比较。
- 如果哈希分布过差，可能导致某些桶过大，因此实践里要尽量使用分布好的哈希函数。

## 代码 / 伪代码
大数据场景下的核心框架可以写成：

```java
// 第一阶段：分桶
for (num in stream) {
    int bucket = hash(num) % M;
    writeToFile(bucket, num);
}

// 第二阶段：逐桶统计
int bestNum = 0;
int bestCount = 0;
for (int i = 0; i < M; i++) {
    Map<Integer, Integer> map = new HashMap<>();
    for (num in file[i]) {
        map.put(num, map.getOrDefault(num, 0) + 1);
    }
    for (entry in map) {
        if (entry.count > bestCount) {
            bestCount = entry.count;
            bestNum = entry.num;
        }
    }
}
```

如果数据本身能直接装进内存，当然可以退化成最普通的哈希表计数。

## 记忆点
- 海量频次统计题，先想“能不能一次装下内存”。
- 装不下就用哈希函数分流。
- 相同数字一定进同一桶，这是正确性的核心。
- 每桶求局部冠军，最后再求全局冠军。
