## 一句话结论
递归序指的是：对于二叉树中的每个节点，递归函数会“来到它这里 3 次”。
只要理解这 3 次分别发生在什么时候，先序、中序、后序遍历其实都只是“在不同到达时机处理节点”的结果。

## 核心知识点
- 递归不是黑盒，本质上是函数调用栈在树结构上的来回穿梭。
- 对每个节点而言，递归过程会经历：第一次到达、左子树返回后第二次到达、右子树返回后第三次到达。
- 二叉树的递归遍历，本质都是在这 3 个时机里选择一个时机处理当前节点。
- 后面很多树形 DP、递归套路题，都是建立在“看懂递归序”之上的。

## 图片转写 / 题意还原
原始笔记只有题目名称，没有配图。

这题真正想讲的是一个底层视角：

- 写一个标准递归函数 `f(head)`
- 对于每个非空节点 `head`
- 它不是只“来到一次”，而是会因为递归进入和返回，逻辑上经历 3 个处理时机

这就是“递归序”。

## 图解
假设二叉树如下：

```text
        1
      /   \
     2     3
    / \
   4   5
```

再看最基础的递归框架：

```java
void f(Node head) {
    if (head == null) {
        return;
    }
    // 位置1
    f(head.left);
    // 位置2
    f(head.right);
    // 位置3
}
```

对于节点 `2` 来说，递归过程中的 3 次“到达”可以这样理解：

```text
第一次到达 2：刚进入 f(2)
第二次到达 2：左子树 f(4) 处理完返回
第三次到达 2：右子树 f(5) 处理完返回
```

如果把整棵树的调用感觉画出来，可以粗略理解成：

```text
来到 1
  来到 2
    来到 4
    离开 4
  第二次回到 2
    来到 5
    离开 5
  第三次回到 2
第二次回到 1
  来到 3
  离开 3
第三次回到 1
```

这就是递归序最核心的观察方式。

## 解题思路
### 1. 为什么每个节点会经历 3 次
因为标准二叉树递归函数通常长这样：

1. 先处理当前节点
2. 再递归左子树
3. 再递归右子树

所以一个节点会经历：

- 刚进入函数时
- 左子树结束返回时
- 右子树结束返回时

这 3 个逻辑时机。

### 2. 为什么这很重要
很多人只记“先序、中序、后序”，但不知道它们为什么成立。

实际上：

- 先序：第一次到达时处理
- 中序：第二次到达时处理
- 后序：第三次到达时处理

所以递归序是三序遍历的“母模版”。

### 3. 为什么只有二叉树特别适合这样讲
因为二叉树恰好有：

- 当前节点
- 左子树
- 右子树

因此天然就出现 3 个清晰的阶段。

如果是多叉树，也有类似思想，但不会像二叉树这样对应得这么整齐。

## 典型例子
还是用这棵树：

```text
    1
   / \
  2   3
```

递归框架中，节点 `1` 的 3 个时机分别是：

- 第一次到达：刚进入 `f(1)`
- 第二次到达：`f(2)` 执行完返回
- 第三次到达：`f(3)` 执行完返回

节点 `2` 的 3 个时机分别是：

- 第一次到达：刚进入 `f(2)`
- 第二次到达：左子树返回后
- 第三次到达：右子树返回后

如果这时在不同位置打印值：

- 位置 1 打印，就是先序
- 位置 2 打印，就是中序
- 位置 3 打印，就是后序

## 复杂度
- 时间复杂度：`O(N)`
- 空间复杂度：`O(H)`

其中：

- `N` 是节点数
- `H` 是树高

空间复杂度来自递归调用栈。

## 易错点
- “每个节点来到 3 次”是逻辑上的处理时机，不是说代码里真的写了 3 次函数调用。
- 空节点 `null` 会直接返回，不参与这 3 次讨论。
- 理解递归序时，不要一开始就死记先中后序，要先看函数在一个节点上的完整生命周期。
- 这题是认知题，不是刷模板题，理解不到位，后面很多树题都会越做越乱。

## 记忆点
- 递归序 = 每个节点的 3 个逻辑处理时机。
- 先序在第一次到达时处理。
- 中序在第二次到达时处理。
- 后序在第三次到达时处理。
