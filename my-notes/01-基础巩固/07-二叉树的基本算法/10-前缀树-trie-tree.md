## 一句话结论
前缀树本质上是“按字符一层一层建路”的多叉树。

它最适合解决的，不是普通的字符串查找，而是这一类问题：

- 某个单词是否出现过
- 某个单词出现了几次
- 有多少单词以某个前缀开头

它的核心优势是：

```text
把公共前缀复用掉，
让查询复杂度主要只和字符串长度有关。
```

## 先想清楚：为什么需要 Trie
假设我们维护这样一批单词：

```text
cat
car
care
dog
```

如果只是问：

```text
"car" 在不在？
```

那用 `HashSet` 也能做。

但如果我们经常问的是：

- 有多少单词以 `ca` 开头
- 有多少单词以 `car` 开头
- 某个单词被加入过几次

那普通哈希结构就不够顺手了。

Trie 的思路是：

```text
不把字符串当成一个整体存，
而是把它拆成一个个字符，按顺序往下挂。
```

这样，所有公共前缀就天然会共用同一段路径。

## 图片转写 / 题意还原
原图要表达的核心其实就是下面几句话：

- 前缀树是一种多叉树结构
- 插入字符串时，从左到右依次处理每个字符
- 某条路不存在就新建，存在就复用
- 沿途经过的节点记录“有多少字符串经过这里”
- 单词结束的位置记录“有多少字符串在这里结束”

所以 Trie 不是在比较字符串整体大小，而是在做：

```text
按字符逐层建路
```

## 结构图：公共前缀是怎么复用的
把这几个单词插进去：

```text
cat
car
dog
```

Trie 可以画成：

```text
root
 ├─ c
 │   └─ a
 │       ├─ t
 │       └─ r
 └─ d
     └─ o
         └─ g
```

这里最关键的是：

```text
cat 和 car 的 c -> a 这段路径是共用的
```

也就是说，Trie 的本质不是“把字符串存进树里”，而是：

```text
把相同前缀压缩成同一段公共路径
```

## Trie 节点里到底存什么
课程里最常见的 Trie 节点会维护两个计数：

- `pass`
- `end`

它们非常容易混，所以最好单独记清楚。

### 1. `pass` 是什么
`pass` 表示：

```text
有多少字符串经过当前节点
```

注意是“经过”，不是“停在这里”。

### 2. `end` 是什么
`end` 表示：

```text
有多少字符串恰好在当前节点结束
```

注意是“在这里结尾”，不是“经过这里”。

### 3. 用一个例子理解
依次插入：

```text
apple
app
apple
```

那么在前缀 `app` 对应的节点上：

- `pass = 3`
  因为这 3 次插入都经过了 `app`
- `end = 1`
  因为只有一个字符串恰好在 `app` 结束

在 `apple` 结尾节点上：

- `pass = 2`
- `end = 2`

所以可以把它压成一句话：

```text
pass 管“路过多少次”
end 管“在这结束多少次”
```

## 图解：插入一个单词时发生了什么
假设现在插入：

```text
cat
```

流程就是：

```text
root -> c -> a -> t
```

每往下走一步，都做两件事：

1. 如果这条路不存在，就新建节点
2. 当前来到的节点，`pass++`

最后走到单词末尾时，再做：

```text
end++
```

如果之后再插入：

```text
car
```

那么：

```text
root -> c -> a
```

这段路径就不用新建了，直接复用。

只是在原来的基础上继续：

- 让沿途节点的 `pass` 增加
- 在 `r` 这个结尾节点让 `end` 增加

所以插入的感觉是：

```text
没有路就建路
有路就复用
沿途加 pass
结尾加 end
```

## 四个核心操作
Trie 真正常用的，其实就是下面四个动作。

### 1. 插入 `insert`
从根节点出发，按字符逐个往下走。

- 没有路就创建
- 有路就复用
- 每到一个节点，`pass++`
- 到最后一个字符对应节点时，`end++`

### 2. 查询完整单词出现几次
沿着单词一路往下走：

- 中途断路，说明这个单词从来没插入过，返回 `0`
- 能完整走到结尾，返回最后节点的 `end`

这表示：

```text
这个单词完整出现了几次
```

### 3. 查询某前缀命中了多少单词
沿着前缀一路往下走：

- 中途断路，返回 `0`
- 能走到前缀末尾，返回当前节点的 `pass`

这表示：

```text
有多少单词以这个前缀开头
```

### 4. 删除 `erase`
删除是最容易写错的，因为它不是简单把 `end--` 就完了。

正确流程是：

1. 先确认这个单词确实存在
2. 从根节点开始沿路往下，把经过节点的 `pass--`
3. 如果某一条路减完后 `pass == 0`，说明后面整棵子树都没人用了，可以直接断开
4. 如果没有提前断开，最后把结尾节点的 `end--`

这里最关键的理解是：

```text
一旦某条路 pass 变成 0，
后面那整段结构就再也不会被任何单词用到
```

所以可以直接删枝。

## 把四个操作串成一个完整例子
依次插入：

```text
apple
app
apple
```

这时会得到：

- `countWordsEqualTo("apple") = 2`
- `countWordsEqualTo("app") = 1`
- `countWordsStartingWith("app") = 3`

为什么最后一个是 `3`？

因为 3 次插入的字符串都经过了 `app` 这段路径。

如果这时删除一次 `apple`：

- `apple` 这条路径沿途的 `pass` 会减一
- `apple` 结尾节点的 `end` 会减一

删除后就变成：

- `countWordsEqualTo("apple") = 1`
- `countWordsEqualTo("app") = 1`
- `countWordsStartingWith("app") = 2`

这个例子正好说明：

- `end` 管完整命中
- `pass` 管前缀经过

## 为什么 Trie 查询快
Trie 的查询复杂度通常是：

```text
O(L)
```

其中 `L` 是字符串长度。

它为什么快？

因为它不是拿一个目标字符串去和很多字符串逐个比较，而是：

```text
目标字符串有几位，
我就顺着走几步
```

所以它特别适合：

- 查询很多次
- 前缀查询很多
- 字符串有大量公共前缀

## 数组版实现 vs `HashMap` 版实现
Trie 最容易让人抽象不起来的地方，其实不是 `pass` 和 `end`，而是：

```text
一个节点往下的“路”到底怎么存？
```

课程代码里给了两种实现：

- 数组版：`Trie1`
- `HashMap` 版：`Trie2`

它们的核心区别，不在于“Trie 原理变了”，而只在于：

```text
当前节点怎么记录它的下一层分支
```

### 先看最本质的区别
假设当前节点是某个前缀，比如来到 `ca` 这个节点。

接下来可能往下走的字符有：

```text
t
r
```

也就是说，这个节点需要维护“从字符到下一个节点”的映射关系。

这件事有两种做法。

### 1. 数组版：提前把所有可能的路都预留出来
如果字符集固定，比如题目明确只处理：

```text
a ~ z
```

那可以直接给每个节点准备一个长度为 `26` 的数组：

```java
TrieNode[] nextNodes = new TrieNode[26];
```

它的意思是：

```text
nextNodes[0]  表示走字符 'a' 的那条路
nextNodes[1]  表示走字符 'b' 的那条路
...
nextNodes[25] 表示走字符 'z' 的那条路
```

所以当我们想走字符 `c` 时，只需要做：

```java
path = word.charAt(i) - 'a';
node = node.nextNodes[path];
```

可以把数组版想成下面这样：

```text
当前节点
├─ a -> nextNodes[0]
├─ b -> nextNodes[1]
├─ c -> nextNodes[2]
...
└─ z -> nextNodes[25]
```

哪怕其中大部分路根本不存在，这 26 个位置也都会先留出来。

所以数组版的特点是：

- 查找哪条路很快，直接下标定位
- 写法很直接
- 但如果字符集很大，会浪费很多空位置

### 2. `HashMap` 版：只存真正存在的路
如果字符种类不固定，或者字符集很大，比如可能是：

- 大小写字母
- 数字
- 中文
- 各种特殊字符

那就不适合再开一个巨大的定长数组了。

这时更自然的做法是：

```java
HashMap<Integer, TrieNode> nextNodeMap;
```

它的意思是：

```text
key   = 当前要走的字符
value = 这个字符对应的下一层节点
```

于是只有真正出现过的路，才会被放进 `HashMap`。

比如当前节点下面只出现过 `t` 和 `r` 两条路，那就只会存：

```text
't' -> 对应节点
'r' -> 对应节点
```

而不会把所有别的字符位置都提前占好。

可以把它想成：

```text
当前节点
└─ nextNodeMap
   ├─ 't' -> ...
   └─ 'r' -> ...
```

所以 `HashMap` 版的特点是：

- 只给真实存在的路分配空间
- 字符集更自由
- 但查询和维护的常数通常比数组版更大一点

### 用同一个例子对比一下
假设只插入下面两个单词：

```text
cat
car
```

来到 `ca` 这个节点时：

#### 数组版会怎样
它仍然有一个长度为 `26` 的数组：

```text
nextNodes[0...25]
```

其中只有：

```text
nextNodes['t' - 'a']
nextNodes['r' - 'a']
```

这两个位置非空，其余大部分位置都是 `null`。

#### `HashMap` 版会怎样
它只会维护两条真正存在的边：

```text
't' -> 节点
'r' -> 节点
```

不会多存其他字符。

所以你可以把两者的区别记成一句话：

```text
数组版：可能的路都先占坑
HashMap 版：只有真实存在的路才建记录
```

### 代码里各自长什么样
仓库中的数组版节点：

```java
static class TrieNode {
    public int pass;
    public int end;
    public TrieNode[] nextNodes;

    public TrieNode() {
        pass = 0;
        end = 0;
        nextNodes = new TrieNode[26];
    }
}
```

仓库中的 `HashMap` 版节点：

```java
static class TrieNode {
    public int pass;
    public int end;
    HashMap<Integer, TrieNode> nextNodeMap;

    public TrieNode() {
        pass = 0;
        end = 0;
        nextNodeMap = new HashMap<>();
    }
}
```

对照着看会更清楚：

- `pass` 和 `end` 完全一样
- 真正不同的只有“路”的存法

### 插入时两者分别怎么走
#### 数组版
核心是先把字符转成数组下标：

```java
path = word.charAt(i) - 'a';
if (node.nextNodes[path] == null) {
    node.nextNodes[path] = new TrieNode();
}
node = node.nextNodes[path];
```

这里的重点是：

```text
字符 -> 下标
下标 -> 下一条路
```

#### `HashMap` 版
核心是不再转成 `0~25`，而是直接用字符本身做 key：

```java
path = word.charAt(i);
if (!node.nextNodeMap.containsKey(path)) {
    node.nextNodeMap.put(path, new TrieNode());
}
node = node.nextNodeMap.get(path);
```

这里的重点是：

```text
字符 -> Map 的 key
key -> 下一条路
```

### 查询和删除为什么也只是“路的查法不同”
无论是查询完整单词、查询前缀数量，还是删除，整体流程其实都没变：

- 还是从根开始按字符往下走
- 还是中途断路就说明不存在
- 还是靠 `pass` / `end` 统计

真正变化的，只有“下一跳怎么拿到”。

数组版是：

```java
node.nextNodes[path]
```

`HashMap` 版是：

```java
node.nextNodeMap.get(path)
```

所以不要把这两种实现当成两个完全不同的数据结构。  
它们只是：

```text
同一个 Trie，
换了两种不同的存边方式。
```

### 到底怎么选
可以直接按下面这个判断：

| 场景 | 更适合的实现 |
| --- | --- |
| 字符集固定且很小，比如只含 `a~z` | 数组版 |
| 字符集很大，或者不固定 | `HashMap` 版 |
| 更看重访问速度、代码直接 | 数组版 |
| 更看重节省无效空间、支持更多字符 | `HashMap` 版 |

再压缩成一句话就是：

```text
字符集小且固定，用数组版
字符集大或不固定，用 HashMap 版
```

### 一个很容易忽略的点
很多人会觉得：

```text
HashMap 版更高级，所以应该总是优先用它
```

这其实不对。

如果题目已经明确：

```text
只处理小写字母 a~z
```

那数组版通常更自然，也更快，因为：

- 不需要哈希计算
- 不需要 `containsKey / get`
- 下标访问非常直接

所以这里不是“谁更高级”，而是：

```text
谁更适合当前字符集
```

## 复杂度
- 时间复杂度：插入、删除、查询都为 `O(L)`，`L` 是字符串长度。
- 空间复杂度：和实际创建的节点总数有关，最坏可以到所有字符串总字符数级别。

## 易错点
- `pass` 不是“当前字符出现几次”，而是“有多少字符串经过这个节点”。
- `end` 不是“这条路被走了几次”，而是“有多少字符串在这里结束”。
- 删除前一定要先判断单词是否存在，否则会把计数减错。
- 一旦某条路的 `pass == 0`，后面整棵子树都可以直接断掉。
- 数组版 Trie 默认通常只支持固定字符集，比如 `a~z`。

## 代码 / 伪代码
仓库中的对应实现：

- [TrieTree.java](../../../solutions-java/practice/algs03_tree/TrieTree.java)

这份实现给了两种版本：

- `Trie1`：数组存边，适合固定小字符集
- `Trie2`：`HashMap` 存边，适合更通用字符集

核心结构可以简化成：

```java
class TrieNode {
    int pass;
    int end;
    TrieNode[] nextNodes = new TrieNode[26];
}
```

插入时的关键动作是：

```java
node.pass++;
for (...) {
    if (node.nextNodes[path] == null) {
        node.nextNodes[path] = new TrieNode();
    }
    node = node.nextNodes[path];
    node.pass++;
}
node.end++;
```

删除时最关键的判断是：

```java
if (--node.nextNodes[path].pass == 0) {
    node.nextNodes[path] = null;
    return;
}
```

这句代码的意思不是“随便删一个节点”，而是：

```text
这条路以后再也不会被任何字符串经过了，
所以整个后续分支都可以直接断开
```

## 记忆点
- Trie = 字符串按位建树，不是整串比较。
- 公共前缀共用路径，是它最大的结构优势。
- `pass` 管“经过”，`end` 管“结尾”。
- 查询完整单词看 `end`，查询前缀命中数量看 `pass`。
