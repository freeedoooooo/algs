## 一句话结论
暴力递归不是乱写递归，而是一种“尝试法”：把原问题拆成规模更小的同类子问题，列出停止条件，再把子问题答案整合回当前问题。
它是后面动态规划的源头，因为任何 DP 都对应着某个有重复子问题的暴力递归。

## 核心知识点
- 原问题要能拆成同类子问题。
- 必须有 `base case`。
- 必须能说明如何由子问题答案得到当前答案。
- 默认不缓存子问题解，所以可能存在大量重复计算。

## 图片转写 / 题意还原
原图内容为：

- 暴力递归就是尝试
1. 把问题转化为规模缩小了的同类问题的子问题
2. 有明确的不需要继续递归的条件 `base case`
3. 有当得到了子问题结果之后的决策过程
4. 不记录每一个子问题的解

## 图解
递归套路可以抽象成：

```text
原问题
-> 拆成若干更小子问题
-> 子问题继续拆
-> 到达 base case
-> 子问题答案向上返回
-> 当前层做决策整合
```

## 解题思路
### 1. 先定义函数职责
递归题第一步不是写代码，而是先说清楚：

```text
这个函数返回什么
参数分别代表什么状态
```

### 2. 再写停止条件
所有递归最终都要停在某个最小规模状态。

### 3. 最后列出决策过程
当前层如何利用子问题结果得到自己的答案，这一步才是递归真正的“业务逻辑”。

## 复杂度
- 时间复杂度：依题而定，很多暴力递归是指数级
- 空间复杂度：通常为递归深度 `O(H)` 或 `O(N)`

## 典型例子
例如汉诺塔：

- 把 `N` 层塔从左移到右
- 可以拆成“先搬上面 `N-1` 层”“再搬最大盘”“再搬 `N-1` 层”

这就是标准递归尝试。

## 易错点
- 递归不是先写代码，而是先定义函数含义。
- 没有 `base case` 的递归一定有问题。
- 如果子问题没有重复，递归未必能优化成动态规划。

## 代码 / 伪代码
通用骨架：

```java
int process(状态参数...) {
    if (base case) {
        return baseAnswer;
    }
    int ans = 根据子问题答案整合;
    return ans;
}
```

## 记忆点
- 暴力递归就是尝试。
- 先定义函数职责。
- 再写停止条件。
- 再写子问题整合逻辑。
