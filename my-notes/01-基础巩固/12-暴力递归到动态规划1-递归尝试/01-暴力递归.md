## 一句话结论
暴力递归不是乱写递归，而是一种“尝试法”：

```text
先把问题拆成规模更小的同类子问题
再让子问题自己去求答案
最后把子问题答案组合回当前问题
```

它最重要的价值是：

```text
很多动态规划题
都要先能写出暴力递归
才能真正看懂状态转移是怎么来的
```

## 理论 / 应用价值
这篇虽然名字叫“暴力递归”，但它讲的不是一种低级写法，而是一种非常核心的建模能力。

它在整套知识体系里的位置可以理解成：

```text
暴力递归
-> 发现重复子问题
-> 改记忆化搜索
-> 改动态规划
```

也就是说，很多 `DP` 题如果一上来就硬写状态表，反而容易越写越乱；  
先把暴力递归的“函数含义、尝试过程、停止条件”想清楚，后面的优化才会自然。

这篇最值得学的不是某个固定模板，而是下面这件事：

```text
面对一个复杂问题
如何把它改写成“更小规模的同类问题”
```

如果这一步不会，后面的递归、记忆化搜索、动态规划都会很虚。

## 核心知识点
- 原问题要能拆成同类子问题。
- 必须有 `base case`。
- 必须能说明如何由子问题答案得到当前答案。
- 默认不缓存子问题解，所以可能存在大量重复计算。

## 图片转写 / 题意还原
原图内容为：

- 暴力递归就是尝试
1. 把问题转化为规模缩小了的同类问题的子问题
2. 有明确的不需要继续递归的条件 `base case`
3. 有当得到了子问题结果之后的决策过程
4. 不记录每一个子问题的解

## 图解
暴力递归的整体流程可以抽象成：

```mermaid
graph LR
    A["原问题"] --> B["拆成更小的子问题"]
    B --> C["子问题继续递归求解"]
    C --> D["到达 base case"]
    D --> E["子问题答案开始返回"]
    E --> F["当前层整合答案"]
```

如果把它翻译成更贴近做题的话术，就是：

```text
先定义当前函数到底求什么
再思考它能把任务交给哪些更小的子问题
等子问题算完以后
当前层只负责“做选择”和“合答案”
```

## 解题思路
### 1. 先定义函数职责
递归题第一步不是写代码，而是先说清楚：

```text
这个函数返回什么
参数分别代表什么状态
```

这是最关键的一步。

因为只要函数职责没定义清楚，后面的递归分支、返回值、停止条件都会一起乱掉。

### 2. 再写停止条件
所有递归最终都要停在某个最小规模状态。

这一层通常对应：

- 规模已经小到不能再拆
- 题目已经有直接答案
- 再往下递归已经没有意义

也就是常说的 `base case`。

### 3. 最后列出决策过程
当前层如何利用子问题结果得到自己的答案，这一步才是递归真正的“业务逻辑”。

很多题表面看起来是在写递归，其实真正难的是：

```text
当前层到底有哪些尝试分支
每个分支返回什么
当前层如何从这些结果里选出答案
```

所以“暴力递归”里的“暴力”，本质上是：

```text
把所有可能的尝试路径都枚举出来
```

## 为什么它是动态规划的前置概念
这部分一定要先建立好。

暴力递归和动态规划的关系不是：

```text
两种毫不相干的方法
```

而更像是：

```text
先写出暴力递归
发现有重复子问题
再把重复计算优化掉
```

例如有些递归树会出现：

```text
process(7, 3) 算了一次
后面又在别的分支里重复算 process(7, 3)
```

这时候你才会自然意识到：

```text
这个子问题答案明明可以记下来
```

这就是记忆化搜索和动态规划的起点。

## 复杂度
- 时间复杂度：依题而定，很多暴力递归是指数级
- 空间复杂度：通常为递归深度 `O(H)` 或 `O(N)`

## 典型例子
例如汉诺塔：

- 把 `N` 层塔从左移到右
- 可以拆成“先搬上面 `N-1` 层”“再搬最大盘”“再搬 `N-1` 层”

这就是标准递归尝试。

如果只看这一题，它训练你的不是“会不会背汉诺塔代码”，而是：

```text
一个大任务
能不能拆成几个结构相同、规模更小的子任务
```

再比如后面很多字符串、背包、范围尝试模型，本质上也都在重复这个动作：

- 当前位置要不要选
- 当前范围怎么切
- 当前状态往下有哪些分支

所以学暴力递归，其实是在学“如何设计尝试过程”。

## 易错点
- 递归不是先写代码，而是先定义函数含义。
- 没有 `base case` 的递归一定有问题。
- 如果子问题没有重复，递归未必能优化成动态规划。
- 不要把“递归”理解成固定模板，真正核心是“函数职责 + 尝试分支”。

## 代码 / 伪代码
通用骨架：

```java
int process(状态参数...) {
    if (base case) {
        return baseAnswer;
    }
    int ans = 根据若干子问题答案整合;
    return ans;
}
```

把这段骨架翻成做题语言，就是：

```text
先回答最小规模时答案是什么
再回答当前层能往哪几个更小状态递归
最后回答当前层怎么根据这些返回值得到自己的答案
```

## 记忆点
- 暴力递归就是尝试。
- 先定义函数职责。
- 再写停止条件。
- 再写子问题整合逻辑。
- 很多动态规划，都是从暴力递归优化来的。
