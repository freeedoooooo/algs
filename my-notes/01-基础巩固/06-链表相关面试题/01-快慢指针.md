## 一句话结论
快慢指针的核心不是“背一个固定模板”，而是让两个指针以不同速度前进，再通过“起点在哪、什么时候停”来精确控制 `slow` 的落点。
一旦把“偶数长度时到底要上中点还是下中点”说清楚，这类题基本就稳定了。

## 先理解它在干什么
链表没有下标，不能像数组那样直接写 `arr[n / 2]` 拿中间位置。
所以我们需要一种“只遍历一遍，也能摸到中间区域”的办法，快慢指针就是为这个场景准备的。

最常见的设置是：

```text
slow 一次走 1 步
fast 一次走 2 步
```

当 `fast` 快到链表尾部时，`slow` 通常就走到了中间附近。
接下来要做的，就不是“能不能到中间”，而是“最后到底停在中间的哪个点”。

## 这题真正考的是什么
这题表面上像是在问“找中点”，其实更准确地说，它在考 4 种不同的落点定义：

1. 奇数长度返回中点，偶数长度返回上中点
2. 奇数长度返回中点，偶数长度返回下中点
3. 奇数长度返回中点前一个，偶数长度返回上中点前一个
4. 奇数长度返回中点前一个，偶数长度返回下中点前一个

也就是说，这题的重点不是会不会写循环，而是你有没有先把目标节点定义清楚。

## 用一个例子把 4 种情况看明白
先看偶数长度链表：

```text
1 -> 2 -> 3 -> 4 -> 5 -> 6
```

这条链表里，“中间”其实有 4 种常见说法：

| 名称 | 返回值 |
| --- | --- |
| 上中点 | `3` |
| 下中点 | `4` |
| 上中点前一个 | `2` |
| 下中点前一个 | `3` |

再看奇数长度链表：

```text
1 -> 2 -> 3 -> 4 -> 5
```

这时真正的中点只有一个：

| 名称 | 返回值 |
| --- | --- |
| 中点 | `3` |
| 中点前一个 | `2` |

所以你可以把这题理解成一句话：

```text
奇数时通常没分歧
偶数时要先决定取“靠左”还是“靠右”
```

## 快慢指针为什么能控制落点
本质上有两个可调节开关：

- `slow`、`fast` 从哪里出发
- 循环在什么条件下停止

可以先记一个很实用的直觉：

```text
slow 起点越靠后，最终落点通常越靠后
循环停得越早，slow 最终落点通常越靠前
```

这也是为什么同样叫“快慢指针”，不同题的写法会不一样。
不是谁对谁错，而是目标落点本来就不同。

## 四种情况怎么记
下面这张表最适合复习时直接看。

| 目标 | `slow` 初始位置 | `fast` 初始位置 | 循环条件 | 结果 |
| --- | --- | --- | --- | --- |
| 奇数返回中点，偶数返回上中点 | `head.next` | `head.next.next` | `fast.next != null && fast.next.next != null` | 最常用模板 |
| 奇数返回中点，偶数返回下中点 | `head.next` | `head.next` | `fast.next != null && fast.next.next != null` | 偶数时让 `slow` 多走半拍 |
| 奇数返回中点前一个，偶数返回上中点前一个 | `head` | `head.next.next` | `fast.next != null && fast.next.next != null` | 比第一种更早起步 |
| 奇数返回中点前一个，偶数返回下中点前一个 | `head` | `head.next` | `fast.next != null && fast.next.next != null` | 常用于要找前驱的题 |

如果不想一开始就死记这张表，也可以只先记住第一种，然后自己推另外三种：

- 想让结果更靠后，就把起点往后挪
- 想让结果更靠前，就让 `slow` 更早起步

## 先学会手推，再记模板
建议每种情况都拿下面两条链表自己推一遍：

- 奇数长度：`1 -> 2 -> 3 -> 4 -> 5`
- 偶数长度：`1 -> 2 -> 3 -> 4 -> 5 -> 6`

手推时重点只看两件事：

1. `slow` 初始在哪
2. 循环结束时 `slow` 落在哪

只要这两步能推顺，代码就不容易写错。

## 最常用模板
最常见、也最值得先背下来的版本是：

- 奇数长度返回中点
- 偶数长度返回上中点

代码如下：

```java
if (head == null || head.next == null || head.next.next == null) {
    return head;
}
ListNode slow = head.next;
ListNode fast = head.next.next;
while (fast.next != null && fast.next.next != null) {
    slow = slow.next;
    fast = fast.next.next;
}
return slow;
```

这段代码为什么这样写，可以直接这么理解：

- `slow = head.next`：先把 `slow` 放到第二个节点
- `fast = head.next.next`：让 `fast` 比 `slow` 再领先一步
- `while (fast.next != null && fast.next.next != null)`：只要 `fast` 还能再走两步，就继续

这样一来：

- 奇数长度时，`slow` 会停在中点
- 偶数长度时，`slow` 会停在上中点

## 典型例子
链表：

```text
1 -> 2 -> 3 -> 4 -> 5 -> 6
```

如果题目要求不同，答案也不同：

| 要求 | 返回节点 |
| --- | --- |
| 偶数返回上中点 | `3` |
| 偶数返回下中点 | `4` |
| 偶数返回上中点前一个 | `2` |
| 偶数返回下中点前一个 | `3` |

这就是为什么写代码前一定要先把“题目想要谁”说清楚。

## 边界怎么想
这类题最容易错的不是主循环，而是短链表边界。

尤其要注意这些长度：

- 长度 `0`：空链表
- 长度 `1`：只有一个节点
- 长度 `2`：开始出现“上中 / 下中”分歧
- 长度 `3`：开始能检验奇数场景是否正确

一个很稳妥的习惯是：

- 先单独处理短链表
- 再写统一的快慢指针逻辑

## 易错点
- 没先定义“偶数时取上中还是下中”，代码很容易从一开始就写偏。
- 只记住了代码，没手推过长度 `2`、`3`、`4` 的情况，面试时特别容易懵。
- 循环条件里常写 `fast.next != null && fast.next.next != null`，判断顺序不能反。
- 题目如果要的是“中点前一个”，通常就不能直接照搬找中点模板。

## 记忆点
- 快慢指针不是一个模板，而是一组“通过起点和停止条件控制落点”的写法。
- 真正要先记住的不是代码，而是 4 种目标节点分别是谁。
- 奇数长度通常好处理，偶数长度才是这题的分歧点。
- 一旦后面碰到回文链表、找环、链表切分，这套思路都会继续出现。
