# 不使用头节点，删除链表任意一个节点

## 一句话结论
如果不给头节点，只给你“待删除节点本身”，那对单链表来说并不能真正删除这个节点，只能用“把后一个节点的内容拷贝到当前节点，再跳过后一个节点”的方式伪装成删除。
这种做法只适用于“待删除节点不是尾节点”的场景，所以它是一个经典面试陷阱题，而不是正常工程写法。

## 核心知识点
- 单链表删除一个节点，正常需要知道它的前驱节点。
- 如果没有头节点，也就无法从头找到前驱，就不能按常规方式删除当前节点。
- 所谓“无头删除”本质上不是删当前节点，而是覆盖当前节点内容，再删掉它的后继。
- 如果当前节点正好是尾节点，这招完全失效。

### 图 2：内存视角
图里的意思是：

- 链表中有若干节点，例如 `1 -> 2 -> 3`
- 外部变量可能分别指向这些节点在内存中的地址
- 如果只拿到中间某个节点的引用，比如指向值为 `2` 的节点
- 你其实并不知道谁指向它，也不知道它的前驱是谁

因此：

- 你不能让“前一个节点跳过它”
- 因为你根本摸不到前一个节点

这正是这题考察的本质。

## 解题思路
### 1. 为什么正常删除做不到
单链表删除节点的标准写法是：

```text
pre.next = cur.next
```

可问题是现在只给了 `cur`，没给 `pre`，也没给 `head`。

所以：

- 你没法找到前驱节点
- 也就没法把链表真正从前面断开

### 2. 为什么还能“看起来像删掉了”
如果 `cur` 不是尾节点，那么它后面一定还有一个节点 `next`。

这时可以：

1. 把 `next.val` 复制到 `cur.val`
2. 把 `cur.next` 指向 `next.next`

这样链表从外部看起来，相当于“当前节点被删掉了”。

但实际上发生的是：

- 当前节点还活着
- 被真正跳过的是下一个节点

所以这是一种“内容替换 + 删除后继”的技巧。

### 3. 为什么尾节点做不到
如果待删除节点本身是尾节点：

- 它后面没有 `next`
- 你没法复制别人的值过来
- 也没法让前驱跳过它

因此尾节点一定无解。

这也是面试里最关键的限制条件。

## 典型例子
链表：

```text
1 -> 2 -> 3 -> 4
```

现在只给你值为 `3` 的节点引用，不给头节点。

### 可行做法
把后面节点 `4` 的值拷贝到当前节点：

```text
1 -> 2 -> 4 -> 4
```

然后让当前节点跳过后一个节点：

```text
1 -> 2 -> 4
```

从链表结果上看，像是“3 被删掉了”。

### 实际发生的事
真正被移除出链表的是原来的最后一个 `4` 节点。
原来那个“值为 3 的节点”还在，只是它的值变成了 `4`。

## 复杂度
- 时间复杂度：`O(1)`
- 空间复杂度：`O(1)`

前提是：

- 已经直接拿到了待删除节点
- 并且它不是尾节点

## 易错点
- 这题不是“任意节点都能删”，尾节点删不了。
- 这题不是“真的删除当前节点”，而是“覆盖当前节点，删除后继节点”。
- 如果节点里除了 `val` 还有其他业务字段，也要一并复制，否则语义就错了。
- 工程代码里一般不推荐这么做，因为它会改变节点身份语义，可能让外部引用产生歧义。

## 代码 / 伪代码
仓库里这题没有单独现成实现，但经典写法通常是：

```java
public void deleteNode(ListNode node) {
    if (node == null || node.next == null) {
        throw new IllegalArgumentException();
    }
    node.val = node.next.val;
    node.next = node.next.next;
}
```

如果节点结构不止一个字段，那么本质上要做的是：

```java
把 node.next 的关键内容复制到 node
再让 node 跳过 node.next
```

## 记忆点
- 单链表正常删除，需要前驱节点。
- 没有头节点时，只能“偷后面一个节点”的内容。
- 这招只对非尾节点成立。
- 它考的是链表和内存模型理解，不是推荐写法。
