## 一句话结论
这题看起来像“链表排序”，但它其实不是把链表排成完全有序，而是按某个基准值把链表重新分区。
更准确地说，它是在练一件很典型的链表能力：

```text
一边遍历
一边把节点拆到不同的子链表里
最后再按规则拼回去
```

所以这题的重点，不是排序算法本身，而是“怎么稳定地拆链、接链、不断链不断乱”。

## 这题到底在做什么
先把题意说得更直白一点。

给你一个链表和一个值 `x`，你要把链表调整成：

- 小于 `x` 的节点放前面
- 大于等于 `x` 的节点放后面

有些版本还会更细，要求分成三段：

- 小于 `x`
- 等于 `x`
- 大于 `x`

这类题最容易误解的点是：

```text
它叫分区
不是完整排序
```

比如：

```text
7 -> 9 -> 1 -> 8 -> 5 -> 2 -> 5
```

按 `5` 分区后，结果可以是：

```text
1 -> 2 -> 7 -> 9 -> 8 -> 5 -> 5
```

或者三段版：

```text
1 -> 2 -> 5 -> 5 -> 7 -> 9 -> 8
```

这里并不要求：

```text
1 -> 2 -> 5 -> 5 -> 7 -> 8 -> 9
```

因为题目关心的是“分区”，不是“区内完全有序”。

## 为什么这题很适合练链表
如果这是数组题，很多人第一反应会是：

- 交换元素
- 做 `partition`
- 像快排或荷兰国旗问题那样处理

但链表和数组最大的区别在于：

- 数组擅长交换位置
- 链表擅长摘节点、接节点

所以同样是分区，到了链表里，思路就要变成：

```text
不要想着交换
要想着把节点送去对应的区
```

这也是这题真正的训练价值：

- 让你从“数组思维”切到“链表思维”
- 学会不用下标和交换，也能完成重组

## 如果按数组思路做，会怎样
最直观的写法是把链表先放进数组。

流程大概是：

1. 遍历链表，把所有节点收集到数组
2. 在数组上做 `partition`
3. 再按数组中的顺序把节点重新串回链表

这种写法的优点是：

- 好想
- 好写
- 很适合笔试

但缺点也很明显：

- 需要 `O(N)` 额外空间
- 有点像“先把链表变回数组，再解决”
- 没有真正利用链表结构本身的特点

所以面试里更期待你说出另一种思路：

```text
既然链表最擅长接来接去
那就直接拆成几段链表，再拼接
```

## 真正像链表的做法：分段重连
这题最自然的面试写法，是边遍历边分类。

例如只分两段时：

- 小于 `x` 的节点接到 `left` 链表
- 大于等于 `x` 的节点接到 `right` 链表

如果是课程原题那种三段版，就改成：

- 小于区 `small`
- 等于区 `equal`
- 大于区 `big`

遍历结束后，再按顺序把这些子链表连起来。

你可以把这题想成：

```text
原链表像一串混杂的珠子
我们不是交换珠子
而是按颜色把珠子分别串到几根线上
最后再把几根线接起来
```

## 这题为什么特别强调“稳定”
很多版本都会要求：

```text
每个分区内部，原始相对顺序不能变
```

这句话非常关键。

例如：

```text
7 -> 9 -> 1 -> 8 -> 5 -> 2 -> 5
```

按 `5` 做两段分区时：

- 小于 `5` 的节点是 `1 -> 2`
- 大于等于 `5` 的节点原顺序是 `7 -> 9 -> 8 -> 5 -> 5`

注意右边这段不是随便排的，而是必须保留原链表里出现的先后顺序。

这也是为什么链表版最自然的写法是“维护尾指针追加”。
因为：

- 每来一个节点，都接到对应分区的尾巴
- 那么这个分区内部顺序天然就是稳定的

## 两段版怎么理解
LeetCode 86 这一版只需要两段：

- `< x`
- `>= x`

所以只需要维护两组头尾指针：

```java
leftHead, leftTail
rightHead, rightTail
```

遍历到一个节点时，只做三件事：

1. 判断它属于左区还是右区
2. 接到对应链表尾部
3. 继续处理下一个节点

最后：

- 如果左区为空，直接返回右区
- 如果左区不为空，就把左区尾巴接到右区头部

这就是整题最核心的骨架。

## 三段版怎么理解
如果题目要的是链表版荷兰国旗问题，那就在两段版基础上多一段：

- `small`
- `equal`
- `big`

整体思路完全一样，只是从 2 条子链表变成 3 条子链表。

最后的连接顺序通常是：

1. `small` 接 `equal`
2. `equal` 接 `big`

当然，真正写代码时要考虑某一段可能为空，所以更准确地说是：

- 谁不空，就把谁接到下一段的头上

这里最容易错的不是分类，而是拼接时忘记空链表分支。

## 为什么 `head.next = null` 这么重要
这题最关键、也最容易被忽略的细节，就是每处理完一个节点后，要及时断开它和原链表的旧连接。

仓库代码里是这样做的：

```java
ListNode next = head.next;
head.next = null;
head = next;
```

很多人第一次写时会觉得：

- 反正后面还要接
- 不断开是不是也行

实际上很危险。

因为你把某个节点接到新区尾部时，如果它还带着原来的 `next`，就相当于把原链表后面的残余结构也一起拖过去了。
后面继续拼接时，就很容易出现：

- 脏链
- 串错链
- 甚至形成环

所以这一步的本质是：

```text
节点一旦决定归属
就先从原链表里彻底摘干净
```

这是很多链表重组题都通用的好习惯。

## 用一个例子走完整个过程
链表：

```text
7 -> 9 -> 1 -> 8 -> 5 -> 2 -> 5
```

基准值：

```text
5
```

如果做两段版，遍历时会逐步形成：

- `left`：收集 `< 5` 的节点
- `right`：收集 `>= 5` 的节点

最终得到：

```text
left : 1 -> 2
right: 7 -> 9 -> 8 -> 5 -> 5
```

最后一拼接：

```text
1 -> 2 -> 7 -> 9 -> 8 -> 5 -> 5
```

如果做三段版，则会得到：

```text
small: 1 -> 2
equal: 5 -> 5
big  : 7 -> 9 -> 8
```

最后连接为：

```text
1 -> 2 -> 5 -> 5 -> 7 -> 9 -> 8
```

这两个结果都不是“完全有序”，但都满足分区要求。

## 这题到底在练什么能力
这题看起来像一道分类题，其实核心是在练 4 个链表基本功：

- 遍历时把节点从旧结构里摘出来
- 用头尾指针维护多个子链表
- 在保持稳定性的前提下重组链表
- 处理好多段拼接和空链表边界

所以它和前面的快慢指针、回文链表其实是一条主线：

- 都不是在玩复杂算法
- 都是在练你能不能安全地改链表结构

## 复杂度
- 时间复杂度：`O(N)`
- 空间复杂度：
  - 数组版：`O(N)`
  - 多链表拼接版：`O(1)`

多链表拼接版之所以是 `O(1)`，不是因为“没做多少事”，而是因为：

- 没有开数组
- 没有新建一堆节点
- 只是复用了原节点，并维护了少量指针变量

## 易错点
- 把“分区”误写成“完全排序”。
- 忘记题目可能要求稳定性，导致分区内部顺序被打乱。
- 节点接入新区后没有及时 `head.next = null`，后面很容易成脏链或成环。
- 拼接时只考虑正常情况，漏掉某一段为空的分支。

## 代码 / 相关实现
仓库中的对应实现：

- [PartitionList.java](/D:/PROJECT_HU/github/freeedoooooo/algs/solutions-java/leetcode/p086_partition_list/PartitionList.java)

仓库代码实现的是两段版：

- 小于 `x`
- 大于等于 `x`

主流程可以概括成：

```java
1. 准备 left / right 两条子链表的头尾指针
2. 遍历原链表，把当前节点摘下并接到对应分区尾部
3. 每处理完一个节点，先断开它和原链表的旧连接
4. 最后把 leftTail.next 指向 rightHead
```

代码里最值得注意的两处是：

```java
if (head.val < x) {
    ...
} else {
    ...
}
```

以及：

```java
ListNode next = head.next;
head.next = null;
head = next;
```

前者负责分类，后者负责“摘节点”。
真正把这题写稳的，往往不是分类判断本身，而是这个摘节点动作。

## 和记忆方式
- 这题不是链表完整排序，而是链表按规则重组。
- 数组版是在数组上做 `partition`，链表版是在链表上做“分段重连”。
- 想保留稳定性，就维护每个分区的尾指针，按出现顺序追加。
- 一看到这类链表重组题，就记得先想：怎么摘、怎么接、哪里要断开。
