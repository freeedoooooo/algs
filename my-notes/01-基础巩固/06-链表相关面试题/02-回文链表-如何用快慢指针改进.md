## 一句话结论
这题表面上是在判断“链表是不是回文”，本质上其实是在练一条很重要的链表思路：

```text
先用快慢指针找到结构分界点
再把右半边改造成可比较的形态
最后在做完判断后把链表恢复
```

所以它的价值，不只是会做一道回文题，而是学会“先找支点，再局部改结构，再恢复现场”这套链表套路。

## 为什么这题值得单独讲
如果是数组，判断回文很自然：

- 一个指针从左往右
- 一个指针从右往左
- 逐个比较就行

但链表的问题在于：

- 你只能从头往后走
- 不能直接从尾巴往前看
- 也不能像数组那样随时拿到对称位置

所以这题真正困难的地方不是“知道回文要左右比较”，而是：

```text
链表怎么获得“从右往左看”的能力
```

围绕这个问题，才会出现后面的 3 种解法。

## 如果不用快慢指针，会怎么做
最直接的做法是借助栈。

### 版本 1：整条链表入栈
流程很直白：

1. 从头遍历链表，把所有节点压栈
2. 再从头遍历一遍，同时不断弹栈比较

这样做很好理解，因为栈天然就提供了“从右往左看”的顺序。

缺点也明显：

- 额外空间是 `O(N)`

### 版本 2：只把后半段入栈
如果先用快慢指针找到中点，就没必要把整条链表都压栈了。
只把后半部分入栈，也能完成比较。

这样空间能降到：

- `O(N/2)`

### 版本 3：右半部分原地反转
这是这篇真正要讲的版本。

它的思路是：

- 先找到中点
- 再把右半部分原地逆序
- 这样右半部分就等于“从右往左读”的结果
- 然后左右两边同时往中间比较

这样额外空间就能压到：

- `O(1)`

所以这题其实不是单纯在讲一个题，而是在讲一条很典型的优化路径：

```text
借助额外结构
-> 只借一半结构
-> 尽量不借结构，改用原地操作
```

## 快慢指针在这题里到底干什么
在这题里，快慢指针不是主角里的“最终答案制造者”，而是一个非常关键的前置步骤。

它负责解决的问题是：

```text
这条链表应该从哪里一分为二
```

只有先把这个分界点找对，后面的事情才有地方下手：

- 从哪里开始反转右半边
- 左边和右边分别从哪里开始比较
- 奇数长度时中点要不要跳过

所以在回文链表里，快慢指针更像一个“找支点”的工具。

## 这题的整体思路
可以先把完整流程压缩成 4 步：

1. 用快慢指针找到中点或上中点
2. 从 `slow.next` 开始逆序右半部分
3. 左右两边同时往中间比较
4. 把右半部分再逆回去，恢复原链表

这 4 步里，真正关键的是第 1 步和第 4 步：

- 第 1 步决定你有没有把链表切对
- 第 4 步决定你的写法是不是完整、稳妥、工程化

## 为什么“反转右半部分”这步这么巧
先看一个回文链表：

```text
1 -> 2 -> 3 -> 2 -> 1
```

回文的意思是：

- 从左往右读：`1 2 3 2 1`
- 从右往左读：`1 2 3 2 1`

但链表没法直接从右往左走。
于是我们换个思路：

- 不是真的从右往左遍历
- 而是把右半边反转，让它变成“看起来像从右往左”

例如右半边原来是：

```text
2 -> 1
```

反转后就变成：

```text
1 -> 2
```

这样你就能用两个正向指针去比较：

- 左边从头开始往右走
- 右边从原尾部翻过来之后，也往右走

这就是这题最巧的地方：

```text
不去强行实现“从右往左遍历”
而是把右边改造成“也能从左往右比较”
```

## 结合例子看一遍
以这条链表为例：

```text
1 -> 2 -> 3 -> 2 -> 1
```

### 第 1 步：找到中点
用快慢指针后，`slow` 会落在 `3`。

这时链表可以理解成：

```text
左边：1 -> 2 -> 3
右边：2 -> 1
```

### 第 2 步：断开并反转右半边
从 `slow.next` 开始反转，也就是把：

```text
2 -> 1
```

变成：

```text
1 -> 2
```

这里代码里有个很关键的动作：

```java
slow.next = null;
```

这一步不是可有可无的细节，而是在告诉链表：

- 左边先到这里为止
- 右边接下来我要单独翻转

如果不断开，反转时就可能把链表拧成环。

### 第 3 步：左右比较
这时：

- `left` 从 `head` 出发
- `right` 从反转后的右半部分头节点出发

比较过程就变成：

```text
left : 1 -> 2 -> 3
right: 1 -> 2
```

只要一路都相等，它就是回文。

这里为什么只要让右边走完就够了？

因为：

- 奇数长度时，中点本来就不需要比较自己和自己
- 偶数长度时，两边长度本来就相等

所以右侧那一半走完，比较任务就完成了。

### 第 4 步：恢复链表
比较结束后，再把右半部分反转回去，恢复成原来的结构。

这一步的意义不只是“题解更完整”，而是一个很好的链表习惯：

- 判断回文本质上是查询
- 查询最好不要永久破坏原结构

这也是为什么仓库代码里最后专门写了恢复逻辑。

## 偶数长度时要注意什么
以：

```text
1 -> 2 -> 2 -> 1
```

为例，快慢指针结束后，`slow` 会停在前一个 `2`，也就是上中点。

这时：

- 左边是 `1 -> 2`
- 右边从 `slow.next` 开始，是 `2 -> 1`

这正好适合后面反转和比较。

所以这里要记住的不是“某段代码”，而是：

```text
偶数长度时
slow 通常停在左半部分最后一个节点
```

这样后续从 `slow.next` 开始处理，最自然。

## 这题到底在练什么能力
这题看起来像一道判断题，其实同时在练 4 种链表能力：

- 用快慢指针找结构分界点
- 原地反转一段链表
- 让两段链表同步比较
- 做完后恢复原结构

所以它在整章里的位置很重要。
它基本把后面链表题常见的几个动作串到一起了。

## 复杂度
- 时间复杂度：`O(N)`
- 空间复杂度：`O(1)`

时间主要花在：

- 找中点一次
- 反转一次
- 比较一次
- 恢复一次

虽然看起来步骤不少，但每个节点被处理的次数都还是常数级。

## 易错点
- 只记得“回文要比较左右”，但没先想清楚链表怎么拿到右往左的比较能力。
- 偶数长度时把 `slow` 的落点想错，导致右半部分起点错了。
- 反转前忘记 `slow.next = null`，容易把链表拧成环。
- 比较结束后漏掉恢复链表，代码虽然能过题，但习惯不够完整。

```java
3. 逆序右半部分
4. left 和 right 同步比较
5. 再把右半部分逆回去
```

其中最值得反复注意的两处是：

```java
while (fast.next != null && fast.next.next != null) {
    slow = slow.next;
    fast = fast.next.next;
}
```

以及：

```java
slow.next = null;
```

前者决定分界点，后者保证反转时不会把结构搞乱。

## 和记忆方式
- 这题不是在考“知不知道回文”，而是在考链表怎么制造对称比较条件。
- 快慢指针在这里的价值，是先找到一个稳定的中间支点。
- 右半部分反转的意义，是把“右往左比较”改写成“两个指针都往右比较”。
- 做完判断后恢复原链表，是这题最容易被忽略、但也最体现完整度的一步。
