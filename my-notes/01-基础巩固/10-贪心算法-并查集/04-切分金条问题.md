## 一句话结论
切分金条问题是一道经典贪心题：每次都先合并当前最小的两段，才能让总切分代价最小。
它表面上在讲“怎么切金条”，本质上其实是在练“最优合并顺序”。

## 核心知识点
- 每次切一段长度为 `L` 的金条，都要支付 `L` 的代价。
- 最终目标是把整根金条切成若干指定长度。
- 目标不是切得快，而是总代价最小。
- 正确贪心策略是每次优先处理最小的两段。
- 这题是哈夫曼树思想的原型题。

## 题意说明
给定一个正整数数组 `arr`：

- `arr[i]` 表示最终需要分出去的一段长度
- 原始金条总长度等于 `arr` 所有元素之和
- 每次切一段长度为 `L` 的金条，都要支付 `L` 的代价
- 你可以自由决定切分顺序

要求返回：

```text
把整根金条切成目标长度后，最少需要多少总代价
```

### 题意图
以 `arr = [10, 20, 30]` 为例：

```text
总长度 = 60
目标是切出 10、20、30 三段
```

你可以这样切：

```text
60
-> 10 + 50   花费 60
-> 20 + 30   再花费 50
总花费 110
```

也可以这样切：

```text
60
-> 30 + 30   花费 60
-> 10 + 20   再花费 30
总花费 90
```

所以同样的目标长度，切分顺序不同，总代价会不同。

## 图解
这题最容易理解的方法，不是一直盯着“切”，而是反过来看“合并”：

```text
10 20 30
先合并 10 和 20 -> 30   花 30
再合并 30 和 30 -> 60   花 60
总花费 90
```

为什么可以反着看？

因为：

```text
切分一次的代价 = 当前整段长度
反过来看，就等价于合并两段的代价 = 两段长度之和
```

于是整题就从“怎么切金条”变成了：

```text
若干段数字
按什么顺序两两合并，总代价最小
```

## 解题思路
### 1. 为什么每次先合并最小两段
谁越早参与合并，谁就越可能在后续被重复累加多次。

所以：

- 小数字应该尽量早点合并
- 大数字应该尽量晚一点参与多次累加

这正是：

```text
每次先取当前最小的两段
```

### 2. 为什么不能随便挑顺序
如果一开始就让大段过早参与：

- 这段大的代价会被重复加进后面的步骤
- 总代价就会被快速放大

所以这题不是靠一次排序就完事，而是每一轮都要重新从当前所有段里取最小的两个。

### 3. 为什么用小根堆
因为整个流程会不断重复这三步：

1. 取最小
2. 再取最小
3. 合并后塞回去

这正是小根堆最擅长的操作模式。

## 过程示例
以数组 `[6,7,8,9]` 为例：

第一轮：

```text
取 6 和 7
合并成 13
总代价 += 13
当前集合变成 [8,9,13]
```

第二轮：

```text
取 8 和 9
合并成 17
总代价 += 17
当前集合变成 [13,17]
```

第三轮：

```text
取 13 和 17
合并成 30
总代价 += 30
当前集合变成 [30]
```

最后总代价：

```text
13 + 17 + 30 = 60
```

## 复杂度
- 时间复杂度：`O(N log N)`
- 空间复杂度：`O(N)`

## 易错点
- 这题不是暴力试所有切法，而是标准最优合并问题。
- 不是“先切最大段”或“先切最左边”。
- 合并后的新长度必须重新放回集合，因为后面还会继续参与代价计算。
- 一次排序不够，因为每轮合并后最小值集合都会变化。

## 代码 / 伪代码
课程标准伪代码：

```java
int lessMoney(int[] arr) {
    PriorityQueue<Integer> heap = new PriorityQueue<>();
    for (int num : arr) {
        heap.add(num);
    }
    int sum = 0;
    while (heap.size() > 1) {
        int cur = heap.poll() + heap.poll();
        sum += cur;
        heap.add(cur);
    }
    return sum;
}
```

## 记忆点
- 切分金条问题本质是最优合并。
- 每次合并当前最小两段。
- 动态反复取最小，优先想到小根堆。
- 这题是哈夫曼树思想的直接原型题。
