## 一句话结论
并查集最常见的用途，可以压成一句话：

```text
不断发现“谁和谁应该并到一起”
最后维护若干个连通块 / 区域块 / 集合块
```

所以只要题目关心的是：

- 最终分成几块
- 两个点是否同组
- 加一条边后是否连通

就可以优先想并查集。

## 应用 1：大块区域合并
这一类题的特点是：

- 一开始有很多小块
- 过程中会不断发现“小块之间其实连着”
- 最后要统计还剩多少大块，或者问两个位置是否在同一块里

你可以把它想成：

```text
一开始大家是零散小岛
后来慢慢发现有桥连起来
并查集负责把这些小岛并成大岛
```

### 典型例题：岛屿数量
例如网格里：

- `1` 表示陆地
- `0` 表示水

如果两个陆地上下左右相邻，就属于同一个岛。

并查集常见做法是：

- 每个陆地格子看成一个点
- 相邻陆地就 `union`
- 最后统计集合个数，就是岛屿数量

### 典型例题：动态加陆地
如果题目不断把某些位置从水变成陆地，那么并查集会更合适，因为它很擅长：

```text
动态新增块
动态把相邻块合并
```

## 应用 2：图中的连通性维护
图里最常见的问题之一就是：

```text
两个点是否连通
加一条边之后，会不会把两个连通块接起来
```

并查集刚好擅长处理这种“连通性维护”。

### 典型例题：无向图连通性
例如不断给你边：

```text
(a, b)
(b, c)
(d, e)
```

并查集的处理方式很直接：

- 每个点先自己成组
- 读到一条边 `(u, v)`，就把 `u` 和 `v` 做一次 `union`
- 如果想判断两个点是否连通，就看它们是否同组

### 典型例题：Kruskal 最小生成树
在 `Kruskal` 里，并查集是标准工具：

- 按边权从小到大考虑每条边
- 如果这条边两端点原本不连通，就选它，并 `union`
- 如果已经连通，就跳过，否则会成环

所以在图论里，并查集最经典的作用就是：

```text
维护连通块
判断加边后是否成环
```

## 应用 3：业务记录合并
这类题表面上看像“去重”，但本质上是：

```text
不断发现记录之间有关系
再把这些关系整组成组地连起来
```

### 典型例题：账户合并 / 用户合并
例如：

- 两条记录邮箱相同
- 或手机号相同
- 或账号相同

就认为它们属于同一个人。

这时：

- 哈希表负责发现“字段撞车”
- 并查集负责把所有相关记录收拢成一组

## 用一句话记住这些应用
不管是区域题、图题，还是合并用户题，只要问题长这样：

- 有很多元素
- 元素之间会逐步建立连接关系
- 你关心的是最终分成几块 / 是否同组 / 是否连通

就可以优先想并查集。

也就是说，并查集最常出现的题型其实都能压成：

```text
动态连边
动态合并块
维护连通性
```

## 记忆点
- 区域合并题，经常能用并查集。
- 图里的连通性维护，经常能用并查集。
- 业务记录合并题，也经常能用并查集。
- 只要问题是“谁和谁属于同一组”，优先想到并查集。
