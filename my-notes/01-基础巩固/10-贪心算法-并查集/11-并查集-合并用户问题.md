## 一句话结论
这题是并查集最典型的业务落地题之一：

```text
如果两条用户记录在任意一个关键字段上相同
就认为它们属于同一个真实用户
```

真正难点不在“比较字段”，而在于：

```text
同一人关系有传递性
所以必须把所有相关记录整组成组地合并
```

## 先记结论
第一次看这题，先记住下面 4 句话：

```text
1. 任意一个关键字段相同，就要合并
2. 合并关系有传递性
3. 哈希表负责找“历史上见过的同字段用户”
4. 并查集负责把所有关联用户收拢成一组
```

如果只记一句话，可以记成：

```text
字段负责“发现关系”
并查集负责“传递合并”
```

## 题意说明
给定一批用户记录。  
每条记录里有若干个能够标识身份的字段，比如：

- 账号
- 邮箱
- 手机号

规则是：

- 如果两条记录任意一个关键字段相同，就认为它们是同一个人
- 这种“同一个人”的关系还具有传递性

要求返回：

```text
把所有应该合并的记录合并之后
最终还剩多少个真实用户
```

## 先把这题真正难点看清楚
这题最容易被低估的地方是：

```text
不是“找相同字段”这么简单
而是“找完以后还要把整串关系都连起来”
```

比如：

```text
u1 和 u2 因为邮箱相同要合并
u2 和 u3 因为手机号相同要合并
```

虽然 `u1` 和 `u3` 没有直接相同字段，但它们仍然是同一个人。  
因为：

```text
u1 ~ u2
u2 ~ u3
=> u1 ~ u3
```

这就是并查集特别适合这题的原因。

## 为什么这题很适合并查集
这题表面上像“字段去重”，但本质上不是简单去重，而是：

```text
不断发现两条记录之间有关系
再把这些关系整组成组地连起来
```

而这正是并查集最擅长的事情：

- 先把每个用户看成一个独立集合
- 发现两个用户应当视为同一人时，就执行一次 `union`
- 最后看还剩多少个集合

你可以把它理解成：

```text
哈希表负责发现“边”
并查集负责把这些边连成“块”
```

## 如果不用并查集，会麻烦在哪里
很多人第一反应会觉得：

```text
字段相同就合并一下，不就行了吗？
```

但难点在于，关系不是只合并一层，而是会不断传递。

比如：

```text
u1 和 u2 因为邮箱相同
u2 和 u3 因为手机号相同
u3 和 u4 因为账号相同
```

这时候最终结果其实是：

```text
u1、u2、u3、u4 全都得并成一组
```

如果不用并查集，常见的麻烦有这些：

### 1. 你得手动维护“组”
比如你可能会用：

- 一个 `List<Set<User>>`
- 或者一个 `Map<User, GroupId>`

但一旦发现两个不同组之间要合并，你就得：

- 找到这两个组
- 把一整组元素搬到另一组里
- 还要同步更新组标记

这在数据量一大时会很笨重。

### 2. 传递关系会让逻辑越来越绕
这题最烦的不是“发现直接相同”，而是“发现之后还要把整串关系补齐”。

如果手写合并逻辑，很容易出现：

- 只合并了直接撞字段的两个人
- 却漏掉了间接相连的其他人

也就是说，很容易把：

```text
u1 ~ u2
u2 ~ u3
```

错处理成两个局部关系，而不是一个整体集合。

### 3. 后续再来新关系时，维护成本很高
假设前面你已经把用户分成若干组了，后面又读到一条新记录，发现它同时和两个旧组有关系。  
这时你还得把两个旧组合并成一个更大的组。

如果没有并查集，这种“组合并组”的操作会越来越麻烦。

## 并查集的优势到底体现在哪
用了并查集之后，整个思路会被大幅简化成：

```text
发现关系 -> union
发现关系 -> union
发现关系 -> union
最后统计集合个数
```

你不需要自己维护完整分组内容的搬运过程，因为并查集天然就支持：

- 两个人是否已经同组
- 两个组合并
- 传递关系自动收拢

所以它的优势不是“代码更炫”，而是：

```text
把原本很绕的“动态分组 + 传递合并”
压成了稳定统一的集合操作
```

## 把“不用并查集”和“用并查集”对比着看
### 不用并查集
- 你要自己维护组的数据结构。
- 你要自己处理组合并组。
- 你要自己保证传递关系没有漏。
- 一旦关系链变长，代码会越来越容易乱。

### 用并查集
- 每个用户先单独成组。
- 每发现一条连接关系，就 `union` 一次。
- 传递关系会自动体现在最终集合里。
- 最后直接数集合个数即可。

所以这题里，并查集真正带来的提升是：

```text
不是只让代码更快
更是让问题建模本身变简单了
```

## 图解
假设有 4 个用户：

```text
u1 = (id1, mail1, phone1)
u2 = (id2, mail1, phone2)
u3 = (id3, mail3, phone3)
u4 = (id2, mail4, phone4)
```

先看直接关系：

- `u1` 和 `u2` 的 `mail1` 相同
- `u2` 和 `u4` 的 `id2` 相同

于是：

```text
u1 <-> u2 <-> u4
```

虽然 `u1` 和 `u4` 没有直接相同字段，但它们会因为传递性被并到一起。

最终分组就是：

```text
[u1, u2, u4]
[u3]
```

所以答案是：

```text
2
```

## 解题思路
这题最稳的做法是：

```text
并查集 + 多张哈希表
```

### 1. 并查集负责“分组”
每个用户先看成一个独立集合。  
只要发现两个用户应该视为同一人，就执行一次 `union`。

### 2. 哈希表负责“发现关系”
假设每个用户有三个关键字段 `a、b、c`，那就准备三张表：

- `aMap`
- `bMap`
- `cMap`

每张表的含义都是：

```text
key   = 某个字段值
value = 第一次出现这个字段值的用户
```

### 3. 扫描用户时怎么做
处理当前用户 `user` 时，对它的每个字段都做同样的检查：

- 如果这个字段值以前没出现过，就记到 map 里
- 如果以前出现过，说明找到了“同字段的历史用户”
- 这时把当前用户和那个历史用户执行 `union`

### 4. 为什么这样就够了
因为哈希表负责找到“局部直接关系”，而并查集会自动把这些关系扩展成整组传递关系。

也就是说：

```text
你不需要手动去追整条关系链
并查集会帮你把相关用户全部收拢
```

## 过程示例
继续看上面的 4 个用户。

### 处理 `u1`

- `id1` 第一次出现，记入 `aMap`
- `mail1` 第一次出现，记入 `bMap`
- `phone1` 第一次出现，记入 `cMap`

这时没有合并发生。

### 处理 `u2`

- `id2` 第一次出现，记入 `aMap`
- `mail1` 已经在 `bMap` 中出现过，对应 `u1`
- 所以执行：

```text
union(u2, u1)
```

- `phone2` 第一次出现，记入 `cMap`

### 处理 `u3`

- 三个字段都没撞上历史记录
- 所以 `u3` 先自己单独成组

### 处理 `u4`

- `id2` 已经出现过，对应 `u2`
- 所以执行：

```text
union(u4, u2)
```

- 其他字段没有撞上

最终并查集里会形成：

```text
[u1, u2, u4]
[u3]
```

## 复杂度
- 时间复杂度：近似 `O(N)`
- 空间复杂度：`O(N)`

这里的近似 `O(N)`，是因为：

- 每个用户只会被扫描一次
- 每个字段只做哈希表查询和并查集操作
- 并查集在路径压缩和按规模合并后，单次操作几乎可以看成常数

## 易错点
- 不是字段全部相同才合并，而是任意一个关键字段相同就合并。
- 不能只做局部去重，因为关系有传递性。
- 最后统计答案时，应该统计并查集里集合数量，不是统计哈希表里还有多少键。
- 哈希表负责找连接边，并查集负责真正合并；两者分工不要混。

## 代码 / 伪代码
下面这版伪代码更适合直接对应前面的思路去读：

```java
int mergeUsers(User[] users) {
    UnionFind<User> uf = new UnionFind<>(users);

    Map<String, User> aMap = new HashMap<>();
    Map<String, User> bMap = new HashMap<>();
    Map<String, User> cMap = new HashMap<>();

    for (User user : users) {
        if (aMap.containsKey(user.a)) {
            uf.union(user, aMap.get(user.a));
        } else {
            aMap.put(user.a, user);
        }

        if (bMap.containsKey(user.b)) {
            uf.union(user, bMap.get(user.b));
        } else {
            bMap.put(user.b, user);
        }

        if (cMap.containsKey(user.c)) {
            uf.union(user, cMap.get(user.c));
        } else {
            cMap.put(user.c, user);
        }
    }

    return uf.sets();
}
```

如果把这段流程压成一句话，就是：

```text
先用哈希表找“字段撞车”
一旦撞车，就用并查集合并用户
```

## 记忆点
- 合并用户题，本质是并查集处理传递合并关系。
- 任意字段相同就 `union`。
- 哈希表负责找“历史同字段用户”。
- 最后答案看并查集里还剩几个集合。
