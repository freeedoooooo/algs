## 一句话结论
哈夫曼树想解决的问题只有一个：

```text
怎样把一组权值组织成一棵树
让“权值 × 深度”的总和尽量小
```

它最核心的结论也只有一句话：

```text
每次先合并当前最小的两个权值
```

## 先记结论
第一次看这题，先记住下面 4 句话：

```text
1. 哈夫曼树优化的是带权路径长度
2. 小权值适合放深一点，大权值适合放浅一点
3. 每次先合并最小两个，是正确贪心
4. 实现上就是小根堆反复弹两次、塞回一次
```

如果只记一句话，那就是：

```text
小数多加几次，亏得少
大数多加几次，亏得多
```

## 这篇到底在讲什么
很多人第一次看到“哈夫曼树”这个名字，会以为这是某种特殊树结构。  
其实它更像一种构造原则：

```text
给你一组叶子节点权值
你来决定它们怎么两两合并成一棵二叉树
要求总代价最小
```

这里的“总代价”，就是：

```text
带权路径长度 WPL = Σ(叶子权值 × 该叶子到根的距离)
```

所以这题真正关心的不是树长什么样，而是：

```text
哪些数应该放深一点
哪些数应该放浅一点
```

## 为什么会有“深浅”问题
因为一个节点放得越深，它的权值就会被重复计算越多次。

比如有 3 个叶子权值：

```text
10 20 30
```

如果某棵树里它们的深度分别是：

```text
10 的深度 = 2
20 的深度 = 2
30 的深度 = 1
```

那么总带权路径长度就是：

```text
10×2 + 20×2 + 30×1 = 90
```

这时候你就能直观看到：

- 深度越深，乘的次数越多
- 被多乘的最好是小数字
- 大数字尽量别放太深

这就是哈夫曼树背后的直觉来源。

## 先用一个最小例子看懂
假设权值是：

```text
10 20 30
```

最优做法是先合并 `10` 和 `20`：

```text
10 + 20 -> 30
30 + 30 -> 60
```

对应的树可以粗略画成：

```text
        60
       /  \
     30    30
    /  \
  10   20
```

这棵树里：

- `10` 深度是 `2`
- `20` 深度是 `2`
- 另一个 `30` 深度是 `1`

所以：

```text
WPL = 10×2 + 20×2 + 30×1 = 90
```

如果你反过来，先合并 `20` 和 `30`：

```text
20 + 30 -> 50
10 + 50 -> 60
```

对应的树大概是：

```text
        60
       /  \
     10    50
          /  \
        20   30
```

这时候：

- `10` 深度是 `1`
- `20` 深度是 `2`
- `30` 深度是 `2`

所以：

```text
WPL = 10×1 + 20×2 + 30×2 = 110
```

两边一对比，直觉就出来了：

```text
应该让小权值待在更深的位置
让大权值尽量靠近根
```

## 哈夫曼树到底怎么构造
真正做题时，不需要你手动画整棵树。  
你只要不断重复一个动作：

1. 取当前最小的两个权值
2. 把它们合并成一个新权值
3. 再把这个新权值放回集合

为什么这个过程能代表“造树”？

因为每做一次合并，就等于：

```text
给这两个被合并的部分，共同加了一层深度
```

所以，谁越早被合并，谁以后就会被加更多层。  
这也就解释了为什么：

```text
越小的数，越应该早点被合并
```

## 完整过程示例
以权值 `[6, 7, 8, 9]` 为例。

### 第 1 轮

```text
取最小两个：6 和 7
合并得到：13
当前集合变成：[8, 9, 13]
```

这一步的意思是：

```text
6 和 7 这两个权值
从现在开始都会比根多一层深度
```

### 第 2 轮

```text
取最小两个：8 和 9
合并得到：17
当前集合变成：[13, 17]
```

### 第 3 轮

```text
取最小两个：13 和 17
合并得到：30
当前集合变成：[30]
```

构造结束。

整个过程中，合并代价之和是：

```text
13 + 17 + 30 = 60
```

这个和，恰好就等于最终哈夫曼树的带权路径长度。

## 为什么“合并代价之和”就是答案
这是这题最容易一下没转过来的地方。

每次合并两个部分，比如：

```text
6 和 7 合并成 13
```

其实意思是：

```text
6 和 7 下面所有叶子
都被整体往下压了一层
```

所以每次合并产生的和，会在最终答案里被记一次。  
整棵树造完以后，所有这些“加深一层”的代价累加起来，就正好是总带权路径长度。

你可以把它理解成：

```text
每次合并，都是给某一批叶子统一 +1 层深度
而新增代价，就是这批叶子的权值总和
```

## 为什么贪心是对的
哈夫曼树最核心的正确性结论是：

```text
当前最小的两个权值
在某棵最优树里，一定可以作为同一对兄弟节点出现
```

这句话一旦成立，后面就很顺了：

1. 先把最小两个合并，绝不会吃亏
2. 合并后把它们看成一个新节点
3. 问题规模变小，继续做同样的事

于是整个问题就被拆成了很多轮相同的小选择。

## 哈夫曼树和哈夫曼编码是什么关系
前面讲的哈夫曼树，解决的是：

```text
如何让带权路径长度最小
```

而哈夫曼编码，解决的是：

```text
如何给一批字符分配二进制编码
让整体编码长度尽量短
```

这两个问题本质上其实是同一个问题。

因为如果我们把：

- 字符出现频率，当成权值
- 字符本身，放在叶子节点
- 从根到叶子的路径长度，当成编码长度

那么：

```text
总编码长度 = Σ(字符频率 × 编码长度)
```

这和哈夫曼树里的：

```text
WPL = Σ(叶子权值 × 深度)
```

完全是同一个式子。

所以：

```text
最优哈夫曼树
天然就对应一套平均长度最短的前缀编码
```

## 哈夫曼编码怎么生成
规则非常直接：

```text
左边的边记为 0
右边的边记为 1
从根走到某个字符叶子节点
沿途经过的 0/1 串起来
就是这个字符的编码
```

比如有 4 个字符，出现次数分别是：

```text
A : 1
B : 2
C : 4
D : 8
```

构造过程如下：

```text
先合并 1 和 2 -> 3
再合并 3 和 4 -> 7
最后合并 7 和 8 -> 15
```

可以画成一棵树：

```text
            15
          /    \
         7      D
       /   \
      3     C
     / \
    A   B
```

如果约定：

```text
左 = 0
右 = 1
```

那么得到的编码可以是：

```text
A = 000
B = 001
C = 01
D = 1
```

注意这里左右谁记 `0`、谁记 `1` 并不影响总长度，  
影响的只是“具体编码长什么样”，不影响“这套编码是不是最优”。

## 为什么哈夫曼编码没有前缀冲突
哈夫曼编码有一个特别重要的性质：

```text
任何一个字符的编码
都不会是另一个字符编码的前缀
```

原因很简单：

```text
所有字符都只放在叶子节点
不会放在内部节点
```

这意味着如果你已经走到某个字符，对应路径一定结束在叶子上。  
既然它是叶子，就不可能再继续往下延伸成别的字符编码。

所以哈夫曼编码天然就是：

```text
前缀码 / 无歧义编码
```

这也是它能被直接解码的关键。

## 为什么出现频率高的字符编码更短
因为高频字符的权值更大。  
而哈夫曼树优化的目标是最小化：

```text
频率 × 编码长度
```

所以最优策略一定倾向于：

- 高频字符放浅一点
- 低频字符放深一点

这和前面讲的“大的数别放太深，小的数可以放深一点”完全一致。

从编码角度重新说一遍就是：

```text
常出现的字符，应该分配更短的编码
不常出现的字符，允许分配更长的编码
```

## 哈夫曼编码怎么解码
解码时不需要回溯，也不需要试很多种切分。  
只要从根开始按位往下走：

```text
读到 0 就往左
读到 1 就往右
走到叶子就得到一个字符
然后回到根继续读下一段
```

正是因为它没有前缀冲突，所以这种“边走边确定字符”的解码方式才成立。

## 和定长编码对比一下
假设还是 4 个字符：

```text
A : 1
B : 2
C : 4
D : 8
```

如果使用定长编码，因为有 4 个字符，所以每个字符都要 `2` 位。  
总长度是：

```text
(1 + 2 + 4 + 8) × 2 = 30
```

而上面的哈夫曼编码里，总长度是：

```text
1×3 + 2×3 + 4×2 + 8×1 = 25
```

所以它确实比定长编码更省。

这也顺便说明一件事：

```text
哈夫曼编码不是一定让每个字符都更短
而是让整体平均编码长度最优
```

如果字符频率差异更大，优势就会更明显。  
比如字符种类很多、而且某些字符远比其他字符常见时，哈夫曼编码通常会明显优于定长编码。

## 实现方式
因为每一轮都要做同样三件事：

1. 取最小
2. 再取最小
3. 合并后塞回去

所以最自然的数据结构就是小根堆。

## 复杂度
- 时间复杂度：`O(N log N)`
- 空间复杂度：`O(N)`

## 易错点
- 哈夫曼树不是二叉搜索树，也不是堆本身。
- 它优化的是带权路径长度，不是树高最小。
- 不是排一次序后从左到右扫完，而是每一轮都要重新取最小两个。
- 合并后的新值必须重新放回集合，因为它后面还会继续参与合并。

## 代码 / 伪代码
常见写法如下：

```java
int huffman(int[] arr) {
    PriorityQueue<Integer> heap = new PriorityQueue<>();
    for (int num : arr) {
        heap.add(num);
    }
    int sum = 0;
    while (heap.size() > 1) {
        int cur = heap.poll() + heap.poll();
        sum += cur;
        heap.add(cur);
    }
    return sum;
}
```

## 记忆点
- 哈夫曼树本质是在最小化 `权值 × 深度` 的总和。
- 小权值适合更深，大权值适合更浅。
- 每次合并当前最小两个，是它的核心贪心策略。
- 反复取最小、合并、放回，优先想到小根堆。
