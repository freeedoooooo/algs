## 一句话结论
这题讲的是经典的最优合并问题，也就是哈夫曼树思想：每次都先合并当前最小的两段，才能得到全局最小总代价。
实现上最自然的结构就是小根堆，因为你会反复取出当前最小的两个数，再把合并结果塞回去。

## 核心知识点
- 金条切分最小代价，是哈夫曼树原型题。
- 每次切分的代价等于“当前被切那一段的长度”。
- 把问题转成“若干权重的最优合并顺序”。
- 正确策略是每次合并当前最小的两部分。
- 数据结构上通常配小根堆。

## 图片转写 / 题意还原
原图 1 的题意是：

- 一块金条切两半，需要花费和长度数值一样的铜板
- 比如长度 `20` 的金条，不管怎么切，都要花 `20`
- 给定数组 `[10,20,30]`，表示三个人想分走这些长度
- 整块金条长度是 `60`
- 如果先切成 `10` 和 `50`，再切 `50` 为 `20` 和 `30`，总代价是 `110`
- 如果先切成 `30` 和 `30`，再把一个 `30` 切成 `10` 和 `20`，总代价是 `90`
- 返回最小代价

把这题补成完整说明，就是：

- 给定一个正整数数组 `arr`
- `arr[i]` 表示最终需要分出去的一段长度
- 原始金条总长度等于 `arr` 全部元素之和
- 每次切一段长度为 `L` 的金条，都要支付 `L` 的代价
- 你可以自行决定切分顺序
- 目标是让最终分出所有目标长度时，总切分代价最小

这题的本质不是“模拟切金条”，而是：

```text
给定若干段权重
如何以最小总代价完成合并 / 拆分过程
```

原图 2 还给了另一个经典贪心题：

- 给定数组 `costs`、`profits`、整数 `K`、初始资金 `M`
- 每个项目有花费和利润
- 最多只能做 `K` 个项目
- 做完一个项目立刻获得收益，用于解锁后续项目
- 目标是最后获得最多钱数

这也是典型的“堆 + 贪心”问题。

## 图解
以 `[10,20,30]` 为例：

### 错误切法

```text
60
-> 10 + 50   花 60
-> 20 + 30   再花 50
总花费 110
```

### 正确切法

```text
10 20 30
先合并 10 和 20 -> 30   花 30
再合并 30 和 30 -> 60   花 60
总花费 90
```

这正是哈夫曼树的合并过程。

## 解题思路
### 1. 为什么每次先合并最小两段
谁越早参与合并，谁就越可能被后续重复累加多次。

因此：

- 权重小的数，应该尽量放在更深处
- 权重大的数，应该尽量少被重复累加

这正对应哈夫曼树“最小权值优先合并”的结论。

### 2. 为什么用小根堆
因为每一轮都做同样三件事：

1. 取最小
2. 再取最小
3. 合并后塞回去

这是小根堆最擅长的操作模式。

## 复杂度
- 时间复杂度：`O(N log N)`
- 空间复杂度：`O(N)`

## 典型例子
数组 `[6,7,8,9]`：

- 先合并 `6 + 7 = 13`，花 `13`
- 堆里变成 `[8,9,13]`
- 再合并 `8 + 9 = 17`，花 `17`
- 堆里变成 `[13,17]`
- 再合并 `13 + 17 = 30`，花 `30`

总代价：

```text
13 + 17 + 30 = 60
```

## 易错点
- 不是原地切分顺序随便试，而是标准最优合并问题。
- “总代价最小”不等于“先把最大那段切掉”。
- 合并后的新长度要重新放回堆，因为它后面还会继续参与代价计算。
- 这个题用排序一次不够，关键是动态反复取最小。

## 代码 / 伪代码
课程标准伪代码：

```java
int lessMoney(int[] arr) {
    PriorityQueue<Integer> heap = new PriorityQueue<>();
    for (int num : arr) {
        heap.add(num);
    }
    int sum = 0;
    while (heap.size() > 1) {
        int cur = heap.poll() + heap.poll();
        sum += cur;
        heap.add(cur);
    }
    return sum;
}
```

- 金条切分最小代价 = 哈夫曼树原型。
- 每次合并当前最小两段。
- 反复取最小，优先想到小根堆。
- “最优合并”是这一题的关键词。
