## 一句话结论
并查集用来维护若干互不相交的集合，最常见操作是：两个元素是否在同一集合里，以及把两个集合合并。
它的高效来自两个优化：路径压缩把查找链拉平，按规模合并让树尽量矮。

## 核心知识点
- 每个节点都往上指向父节点。
- 一个集合由它的代表点标识。
- `find` 用来找代表点。
- `union` 用来合并两个集合。
- 路径压缩和按规模合并能把均摊复杂度压到接近 `O(1)`。

## 图片转写 / 题意还原
原图内容可以整理为：

### 图 1：并查集是什么
1. 每个节点都有一条往上指的指针
2. 节点往上找到的头节点，叫做所在集合的代表节点
3. 查询 `x` 和 `y` 是否属于同一个集合，就是看找到的代表节点是不是同一个
4. 把 `x` 和 `y` 所在集合的所有点合并成一个集合，只需要让小集合代表点挂到大集合代表点下面

### 图 2：并查集的优化
1. 节点往上找代表点的过程中，把沿途链变成扁平的
2. 小集合挂在大集合的下面
3. 如果方法调用很频繁，那么单次调用代价可视为 `O(1)`

## 图解
假设开始时有：

```text
1   2   3   4
```

各自独立成集合。

执行：

```text
union(1, 2)
union(3, 4)
union(2, 4)
```

最后结构会变成一个大集合。查询：

```text
isSameSet(1, 3) == true
```

路径压缩后，很多节点会直接指向代表点：

```text
1 -> root
2 -> root
3 -> root
4 -> root
```

## 解题思路
### 1. `find`
不断沿父指针向上走，直到找到代表点。

### 2. 路径压缩
在向上找的过程中，把沿途所有节点直接挂到代表点下面，后续再查就会非常快。

### 3. 按规模合并
合并两个集合时：

- 谁的集合小
- 就把谁挂到大的下面

这样树高增长得最慢。

## 复杂度
- 单次均摊时间复杂度：近似 `O(1)`
- 空间复杂度：`O(N)`

## 典型例子
社交账号合并：

- 如果账号 A 和账号 B 共用同一个邮箱
- 就认为它们属于同一个人

这类“满足某种关系就并到一起”的问题，很适合用并查集。

## 易错点
- 判断同集合，看的不是节点值相邻，而是代表点是否相同。
- 路径压缩不是必须写递归，也可以迭代实现。
- 如果不做按规模合并，最坏情况下树会退化得很深。
- 并查集只擅长连通性维护，不擅长枚举路径细节。

## 代码 / 伪代码
课程标准伪代码：

```java
Node find(Node x) {
    if (parent[x] != x) {
        parent[x] = find(parent[x]);
    }
    return parent[x];
}

boolean isSameSet(Node a, Node b) {
    return find(a) == find(b);
}

void union(Node a, Node b) {
    Node fa = find(a);
    Node fb = find(b);
    if (fa == fb) {
        return;
    }
    if (size[fa] >= size[fb]) {
        parent[fb] = fa;
        size[fa] += size[fb];
    } else {
        parent[fa] = fb;
        size[fb] += size[fa];
    }
}
```

## 记忆点
- 并查集维护“若干不相交集合”。
- `find` 找代表点，`union` 合并集合。
- 路径压缩让链变平。
- 小挂大，让树别长高。
