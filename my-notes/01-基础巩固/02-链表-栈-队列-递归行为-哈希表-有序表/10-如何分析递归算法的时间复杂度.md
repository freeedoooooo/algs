## 一句话结论
分析递归算法的时间复杂度，最常见的方法是先写出递推式，再判断是否可以直接套用 Master 公式。
如果递归过程能稳定地拆成若干个等规模子问题，那么通常可以很快得到复杂度结论。

## 核心知识点
- 先把递归写成递推式，而不是直接凭感觉猜复杂度。
- Master 公式只适用于“等规模分治”的递归。
- 递归复杂度通常由“子问题规模”和“本层额外工作量”共同决定。

### 1. Master 公式的适用场景
Master 公式用于分析递归算法时间复杂度的一类典型情况，尤其适合分治算法。

它要求递归满足下面这种形式：

```text
T(n) = a * T(n / b) + O(n^d)
```

其中：

- `T(n)`：问题规模为 `n` 时的时间复杂度。
- `a`：每次递归会调用多少个子问题。
- `b`：每个子问题的规模缩小到原来的多少分之一。
- `O(n^d)`：除递归调用之外，这一层自己额外做的工作。

### 2. 三种结论
比较 `log_b(a)` 和 `d` 的大小关系：

1. 如果 `d < log_b(a)`，那么：

```text
T(n) = O(n^(log_b(a)))
```

说明递归树下层的总开销更大，递归调用占主导。

2. 如果 `d = log_b(a)`，那么：

```text
T(n) = O(n^d * log n)
```

说明每一层的总开销大致相同，所以会多出一个 `log n` 的层数因子。

3. 如果 `d > log_b(a)`，那么：

```text
T(n) = O(n^d)
```

说明每一层本身的工作量已经占主导，递归展开不会改变主阶。

## 解题思路
遇到递归题时，可以按下面 3 步分析：

1. 写递推式。
2. 判断是不是等规模分治，能不能套 Master 公式。
3. 如果能套，就比较 `log_b(a)` 和 `d`；如果不能套，再换递归树、迭代展开或数学归纳等方法。

## 典型例子
### 例1：归并排序

```text
T(n) = 2T(n / 2) + O(n)
```

- `a = 2`
- `b = 2`
- `d = 1`
- `log_2(2) = 1`

所以属于第二种情况：

```text
T(n) = O(n log n)
```

### 例2：二分查找

```text
T(n) = T(n / 2) + O(1)
```

- `a = 1`
- `b = 2`
- `d = 0`
- `log_2(1) = 0`

同样属于第二种情况：

```text
T(n) = O(log n)
```

### 例3：每层工作量很重的递归

```text
T(n) = 2T(n / 2) + O(n^2)
```

- `a = 2`
- `b = 2`
- `d = 2`
- `log_2(2) = 1`

因为 `d > log_b(a)`，所以：

```text
T(n) = O(n^2)
```

## 易错点
- 不是所有递归都能套 Master 公式，比如子问题规模不一致、递归分支数量不固定时就不行。
- `O(n^d)` 表示当前这一层额外做的事，不包括递归调用内部的成本。
- 套公式前一定要先把递推式写准确，否则后面的结论都会错。

## 速记方法
- 子问题更多，看 `log_b(a)`。
- 当前层更重，看 `d`。
- 两边一样重，多一个 `log n`。
