## 章节导读
这一章的两题看起来完全不像一类题，但底层训练目标其实非常统一：都在教你先站到更高一层看问题。

- 打表技巧：先不要急着推公式，先把小规模答案全部打出来，再从结果里看规律
- 矩阵处理：先不要急着写坐标变化，先把整轮运动轨迹拆出来

如果把这一章压成一句话，就是：

```text
别一上来扎进细节
先找到问题的宏观结构
```

## 本章结论
- 有些题不是靠直接推导，而是先暴力打小表，再归纳出规律。
- 打表时最值得观察的是：奇偶性、取模周期、临界点、幂次点。
- 矩阵题最值得先找的是：对角线、外框环、分层交换这类宏观轨迹。
- 一旦宏观结构先被找出来，后面的代码实现通常只剩边界控制。
- 这一章本质是在训练“先抽象，再落代码”的习惯。

## 图片转写 / 章节笔记还原
这一章原始笔记页主要传达了两组信息。

### 1. 打表找规律的适用场景
- 输入参数简单，并且通常只有一个实际参数
- 返回值也简单，并且通常只有一个
- 可以先写暴力方法，把小规模答案全部打印出来
- 通过结果表反推规律，再把暴力版优化掉

### 2. 矩阵处理的核心方法
矩阵题里最重要的不是具体下标，而是“coding 上的宏观调度”。

典型例子包括：

- ZigZag 打印矩阵
- 转圈打印矩阵
- 原地旋转正方形矩阵

它们看起来写法不同，但本质都在做同一件事：

```text
先定义这一轮处理的整体轨迹
再处理当前这一轮
最后推进到下一轮
```
