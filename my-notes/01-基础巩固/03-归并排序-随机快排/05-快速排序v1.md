## 一句话结论
快速排序 1.0 的本质，是先选一个 `pivot`，再做一次“两路划分”：

- 左边都 `<= pivot`
- 右边都 `> pivot`

单次划分只会把一个 `pivot` 放到最终正确位置，然后递归处理左右两边。

## 核心知识点
- 快排 1.0 的重点不在“递归”二字，而在单次 `partition` 到底怎么做。
- 一次 `partition` 结束后，不是整个区间有序，而是只有一个 `pivot` 已经归位。
- 它维护的是一个不断扩张的 `<= pivot` 区域。
- 扫描过程中，凡是 `<= pivot` 的数都被交换到左边。
- 由于等于 `pivot` 的数也会被划到左边，所以快排 1.0 对重复元素不够友好。

## 图片转写 / 题意还原
这题想讲的不是“快排会递归”，而是：

```text
一个无序数组，如何围绕某个 pivot 做一次划分，
并且保证 pivot 最后落在它排序后应该在的位置。
```

也就是说，给定当前处理区间 `[l...r]` 和一个划分值 `pivot`，要把区间整理成：

```text
[ <= pivot | pivot | > pivot ]
```

注意这里的含义是：

- `pivot` 左边所有数都不大于它
- `pivot` 右边所有数都严格大于它
- 这时 `pivot` 的位置已经正确
- 但 `pivot` 左右两边各自内部仍然可能是乱的

## 图解
### 1. 先只看单次划分的目标
快排 1.0 的单次目标可以先画成这样：

```text
[ 已处理的 <= pivot 区 | 还没看过的区域 ]
```

扫描完成以后，希望变成：

```text
[ <= pivot | pivot | > pivot ]
```

### 2. `index` 指针到底表示什么
仓库实现里的关键变量是：

```text
index
```

它始终表示：

```text
[left ... index-1] 已经被整理成 <= pivot 的区域
```

因此在扫描过程中：

```text
[ <= pivot 区 | 未处理区 ]
  left      index
```

每遇到一个 `<= pivot` 的数，就把它交换到 `index` 位置，然后让 `index` 右移。

### 3. 为什么最后还要单独把 `pivot` 换一次
因为扫描时，所有 `<= pivot` 的数都被塞到左边了，这里面包含：

- 小于 `pivot` 的数
- 等于 `pivot` 的数

所以扫描结束后，左边会变成：

```text
[ <= pivot, <= pivot, <= pivot, ... ]
```

但“某个真正作为划分依据的 `pivot` 元素”未必正好在最后一个 `<= pivot` 的位置上。  
所以仓库代码额外记录了一个 `xi`，表示某个 `pivot` 当前在哪，最后把它和 `index - 1` 交换，固定到中间。

## 解题思路
### 1. 选一个 `pivot`
在仓库实现里，会从当前区间 `[l...r]` 随机挑一个值作为 `x`：

```java
int x = arr[l + (int) (Math.random() * (r - l + 1))];
```

这里先把它理解成“我们决定用这个数作为划分标准”就够了。

### 2. 从左到右扫描，把 `<= pivot` 的都塞到左边
维护两个信息：

- `index`：当前 `<= pivot` 区域的右边界后一个位置
- `xi`：某个值等于 `pivot` 的元素当前位置

初始时：

```text
index = left
```

然后从 `left` 扫到 `right`：

- 如果 `arr[i] <= pivot`
  - 交换 `arr[i]` 和 `arr[index]`
  - 如果交换后 `arr[index] == pivot`，就记录 `xi = index`
  - `index++`
- 如果 `arr[i] > pivot`
  - 什么都不做，继续往右扫

### 3. 扫完后再把 `pivot` 放到最终位置
扫描结束时：

```text
[left ... index-1] 全部都是 <= pivot
[index ... right] 全部都是 > pivot
```

这时只差最后一步：

```text
把记录到的 pivot 位置 xi
换到 index - 1
```

于是就得到：

```text
[ <= pivot | pivot | > pivot ]
```

这里 `index - 1` 就是 `pivot` 的最终位置 `mid`。

### 4. 再递归左右两边
拿到 `mid` 后，继续递归：

```text
[l ... mid-1]
[mid+1 ... r]
```

因为 `mid` 已经是最终正确位置，不需要再参与后续排序。

## 完整过程示例
下面用一个完整例子，把第一次 `partition` 和后续递归都走一遍。

示例数组：

```text
[4, 2, 5, 3, 1]
```

假设这次选到：

```text
pivot = 3
```

### 第 1 步：初始化

```text
arr   = [4, 2, 5, 3, 1]
index = 0
xi    = 未记录
```

此时含义是：

```text
[ <=3 区 | 未处理区 ]
[      | 4, 2, 5, 3, 1 ]
 ^
 index
```

### 第 2 步：扫描到 4

```text
4 > 3
```

所以不动：

```text
arr   = [4, 2, 5, 3, 1]
index = 0
```

### 第 3 步：扫描到 2

```text
2 <= 3
```

把 `2` 交换到 `index=0` 的位置：

```text
[2, 4, 5, 3, 1]
```

然后：

```text
index = 1
```

现在数组可以理解成：

```text
[ <=3 区 | 未处理区 ]
[   2   | 4, 5, 3, 1 ]
        ^
      index
```

### 第 4 步：扫描到 5

```text
5 > 3
```

不动：

```text
[2, 4, 5, 3, 1]
index = 1
```

### 第 5 步：扫描到 3

```text
3 <= 3
```

把它交换到 `index=1` 的位置：

```text
[2, 3, 5, 4, 1]
```

这时交换后 `arr[index] == 3`，所以记录：

```text
xi = 1
index = 2
```

此时结构是：

```text
[ <=3 区 | 未处理区 ]
[ 2, 3  | 5, 4, 1 ]
          ^
        index
```

### 第 6 步：扫描到 1

```text
1 <= 3
```

交换到 `index=2`：

```text
[2, 3, 1, 4, 5]
```

然后：

```text
index = 3
```

此时扫描结束。

### 第 7 步：把 `pivot` 换到最终位置
现在我们知道：

```text
[0...2] 都 <= 3
[3...4] 都 > 3
```

但当前 `pivot=3` 在 `xi=1`，还没到最终应该在的位置。  
最终位置应该是 `index - 1 = 2`，所以交换：

```text
swap(arr, xi, index - 1)
swap(arr, 1, 2)
```

数组变成：

```text
[2, 1, 3, 4, 5]
```

这时就得到了：

```text
[ <=3 | 3 | >3 ]
```

也就是：

```text
[2, 1 | 3 | 4, 5]
```

现在我们可以明确知道：

- `3` 已经在最终正确位置
- 左边都不大于 `3`
- 右边都大于 `3`

### 第 8 步：递归左右两边
接下来只需要继续处理：

```text
[2, 1]
[4, 5]
```

继续递归后会得到：

```text
[1, 2, 3, 4, 5]
```

## 递归树图示
把上面的过程再从整体上看一遍：

```text
[4, 2, 5, 3, 1]
        |
        | 选 pivot = 3，做一次 partition
        v
[2, 1 | 3 | 4, 5]
   /            \
  /              \
[2, 1]         [4, 5]
   |              |
   v              v
[1, 2]         [4, 5]
   \              /
    \            /
     v          v
    [1, 2, 3, 4, 5]
```

这张图最重要的结论是：

- 每一层只做一件事：找一个 `pivot` 并归位
- 真正完整有序，是很多次“单个 `pivot` 归位”叠加出来的结果

## 关键答疑
这道题最容易卡住的，往往不是“递归怎么写”，而是下面两个问题：

### 1. 扫描一遍后，怎么知道 `pivot` 在哪里
在这个实现里，并不是等扫描结束后再去数组里重新“找” `pivot`，而是在扫描过程中实时记录它的位置。

仓库里用的是：

```text
xi = pivot index
```

它的作用可以理解成一个“记事本”：

- 每当某个元素因为 `<= pivot` 被交换到 `index` 位置
- 就顺手检查一下 `arr[index]` 是不是正好等于 `pivot`
- 如果是，就立刻记下：

```java
if (arr[i] <= pivot) {
    swap(arr, i, index);
    if (arr[index] == pivot) {
        xi = index;
    }
    index++;
}
```

为什么要这样做？

因为扫描过程本质上是在不断把所有 `<= pivot` 的数往左边收拢。  
而 `pivot` 本身也满足 `<= pivot`，所以它也会被卷进左边这支队伍里。  
如果不随手记住它的位置，最后就不知道该把哪一个 `pivot` 送去“归位”。

### 2. `swap(arr, xi, index - 1)` 到底在干什么
这一步本质上是“归位动作”。

先看 `index - 1` 的含义。

循环结束时：

```text
index 指向的是 > pivot 区域的第一个位置
```

也等价于：

```text
index - 1 是 <= pivot 区域的最后一个位置
```

而扫描结束后的数组状态，其实更接近：

```text
[ 小于 pivot 的 + 等于 pivot 的混在一起 | 大于 pivot 的 ]
```

这还不够，因为快排需要的是：

```text
[ <= pivot | pivot | > pivot ]
```

也就是说，必须把某个真正的 `pivot` 挪到左侧区域的最右端，让它成为左右两边的分界线。  
所以最后这一步：

```java
swap(arr, xi, index - 1)
```

就是把“`pivot` 当前的临时座位”换到“它最终应该坐的终点站”。

### 3. 结合例子再看一遍
以前面的例子为例，扫描结束时是：

```text
数组  = [2, 3, 1, 4, 5]
index = 3
xi    = 1
```

这里：

- `index = 3` 表示下标 `0..2` 都已经属于 `<= 3` 区域
- 所以 `index - 1 = 2` 就是这个区域的最后一个位置
- `xi = 1` 表示当前这个 `pivot=3` 还坐在下标 `1`

于是执行：

```text
swap(arr, 1, 2)
```

数组变成：

```text
[2, 1, 3, 4, 5]
```

这时就清楚了：

- `3` 左边是 `[2, 1]`，都 `<= 3`
- `3` 右边是 `[4, 5]`，都 `> 3`
- `3` 真正坐到了它排序后应该在的位置

### 4. 可以把它想成“排队站位”
你可以把整个 `partition` 想成整理一排人：

1. 扫描阶段：把所有“身高 `<= 170cm`”的人都往左边赶
2. 记录阶段：眼睛一直盯着那个“正好 `170cm` 的标杆人物”，记住他现在站在哪
3. 归位阶段：等队伍大致站好后，再把这个标杆人物拎出来，放到左边这群人的最后一个位置

这样一来：

- 他左边全是比他矮或和他一样高的人
- 他右边全是比他高的人

### 5. 一句话记忆

```text
index - 1 是 <= 区的终点站
xi 是 pivot 当前的临时座位
最后一次 swap 是把 pivot 送回终点站
```

## 为什么 1.0 不太好理解
很多人第一次学快排 1.0 会卡在这几个点：

- 以为一次 `partition` 之后整个区间就有序了
- 不理解 `index` 到底表示什么
- 不理解为什么最后还要再交换一次 `pivot`

其实只要记住下面三句话，思路就顺了：

```text
index 左边已经都是 <= pivot
扫描时只负责把 <= pivot 的数不断塞到左边
最后再把某个 pivot 放到 <= 区域的最后一个位置
```

## 为什么 1.0 还不够好
问题在于：

如果数组里有很多值都等于 `pivot`，快排 1.0 只会固定其中一个位置，其他等于 `pivot` 的元素仍然可能在左右递归里继续出现。

例如：

```text
[3, 3, 3, 3, 3]
```

快排 1.0 不能一次把这批 `3` 整段跳过，只能一层层继续递归。  
这也是为什么后面的快排 2.0 会升级成：

```text
[ < pivot | = pivot | > pivot ]
```

## 复杂度
- 平均时间复杂度：`O(N log N)`
- 最坏时间复杂度：`O(N^2)`
- 额外空间复杂度：递归栈平均 `O(log N)`，最坏 `O(N)`

## 易错点
- `partition` 完成后，不是整个数组有序，只是一个 `pivot` 归位了。
- `index` 表示的是“`<= pivot` 区域的下一个空位”，不是当前扫描指针。
- 仓库实现里是先把所有 `<= pivot` 收到左边，再单独把 `pivot` 换到 `index - 1`。
- 快排 1.0 的问题不是不能排序，而是遇到大量重复值时不够优雅。

## 代码 / 伪代码
仓库里的对应实现：

- [QuickSort.java](../../../solutions-java/practice/algs01_sort/QuickSort.java)

核心框架：

```java
void quickSort1(int[] arr, int l, int r) {
    if (l >= r) {
        return;
    }
    int x = arr[l + (int)(Math.random() * (r - l + 1))];
    int mid = partition1(arr, l, r, x);
    quickSort1(arr, l, mid - 1);
    quickSort1(arr, mid + 1, r);
}
```

单次 `partition` 的思路可以概括成：

```java
int partition1(int[] arr, int left, int right, int pivot) {
    int index = left;
    int xi = -1;
    for (int i = left; i <= right; i++) {
        if (arr[i] <= pivot) {
            swap(arr, i, index);
            if (arr[index] == pivot) {
                xi = index;
            }
            index++;
        }
    }
    swap(arr, xi, index - 1);
    return index - 1;
}
```

## 记忆点
- 快排 1.0：一次 `partition` 只固定一个 `pivot`。
- `index` 左边始终是 `<= pivot` 区域。
- 扫描时把 `<= pivot` 的数都塞到左边。
- 最后再把某个 `pivot` 放到 `index - 1`，它就归位了。
