## 一句话结论
快速排序 1.0 的核心，是先选一个划分值 `pivot`，再把数组 partition 成：

- 左边都 `<= pivot`
- 右边都 `> pivot`

然后递归排序左右两边。

## 核心知识点
- 快排 1.0 使用的是经典两路 partition。
- partition 后只确定了一个 `pivot` 的最终位置。
- 等于 `pivot` 的元素不会被整体跳过，可能还会继续参与递归。
- 所以它能工作，但对重复元素不够友好。

## 图解
快排 1.0 可以先理解成最基础的两路划分：

```text
[ <= pivot | > pivot ]
```

例如：

```text
[5, 3, 7, 2, 6], pivot = 6
```

partition 后可能变成：

```text
[5, 3, 2, 6 | 7]
```

然后再递归处理左右两边。

## 解题思路
### 1. 选一个划分值
在仓库实现里，会先随机从当前区间 `[l...r]` 中挑一个数作为 `x`。

目标是把区间整理成：

```text
<= x | x | > x
```

### 2. 做两路 partition
维护一个边界 `index`，表示：

```text
[left ... index-1]  已经是 <= x 的区域
```

然后从左到右扫描：

- 如果当前元素 `<= x`
- 就把它交换到 `index` 位置
- 再让 `index++`

扫完整个区间后：

- `<= x` 的区域已经形成
- 但 `x` 自己可能不在最终该在的位置

所以仓库里还专门记录了 `x` 的位置 `xi`，
最后再把它交换到 `<= x` 区域的最后一个位置上。

这样就能把 `pivot` 固定到正确位置。

### 3. 递归左右两边
拿到 `pivot` 的最终位置 `mid` 后：

- 左边 `[l ... mid-1]` 继续排
- 右边 `[mid+1 ... r]` 继续排

## 典型例子
数组：

```text
[4, 2, 5, 3, 1]
```

假设选到的 `pivot = 3`。

### partition 后
会整理成类似：

```text
[2, 1, 3, 5, 4]
```

此时：

- `3` 左边都 `<= 3`
- `3` 右边都 `> 3`

然后递归：

- 排 `[2, 1]`
- 排 `[5, 4]`

最终得到有序数组。

## 为什么 1.0 还不够好
问题在于：

如果数组里有很多值都等于 `pivot`，
快排 1.0 只会固定其中一个位置，
其他等于 `pivot` 的元素还会继续递归。

这会造成：

- 递归层数增多
- 重复元素场景性能变差

这也是后面快排 2.0 要优化的地方。

## 复杂度
- 平均时间复杂度：`O(N log N)`
- 最坏时间复杂度：`O(N^2)`
- 额外空间复杂度：递归栈平均 `O(log N)`，最坏 `O(N)`

## 易错点
- partition 完后，并不是整个数组有序，只是 `pivot` 落到了正确位置。
- 两路划分时，相等元素可能被反复递归。
- 如果 pivot 总是选得很差，比如每次都选到最小值或最大值，就会退化。

快排 1.0 的核心框架：

```java
void quickSort1(int[] arr, int l, int r) {
    if (l >= r) {
        return;
    }
    int x = arr[l + (int)(Math.random() * (r - l + 1))];
    int mid = partition1(arr, l, r, x);
    quickSort1(arr, l, mid - 1);
    quickSort1(arr, mid + 1, r);
}
```

## 记忆点
- 1.0：partition 只返回一个点。
- 这个点左边 `<= pivot`，右边 `> pivot`。
- 能排，但对大量重复值不够友好。
