## 一句话结论
数组小和问题最适合用归并排序来做，因为它真正考的不是“排序”，而是“在 `merge` 时如何批量结算跨左右区间的信息”。
一旦理解了这题，就真正理解了题目里那句“merge sort 的实质”。

## 题目含义
对数组中每个位置 `arr[i]`，
统计它左边所有“小于等于它”的数之和，
再把所有位置的结果累加起来，
这个总和就叫“小和”。

例如：

```text
[1, 3, 5, 2, 4, 6]
```

逐个位置看：

- `1` 左边没有数，小和贡献 `0`
- `3` 左边小于等于 `3` 的有 `1`，贡献 `1`
- `5` 左边小于等于 `5` 的有 `1, 3`，贡献 `4`
- `2` 左边小于等于 `2` 的有 `1`，贡献 `1`
- `4` 左边小于等于 `4` 的有 `1, 3, 2`，贡献 `6`
- `6` 左边小于等于 `6` 的有 `1, 3, 5, 2, 4`，贡献 `15`

总小和：

```text
0 + 1 + 4 + 1 + 6 + 15 = 27
```

## 核心知识点
- 暴力做法会对每个位置向左扫描，复杂度是 `O(N^2)`。
- 用归并排序时，核心是统计“跨左右两半的贡献”。
- 左右两半各自内部的小和，由递归去解决。
- 真正高效的关键，是在 `merge` 时批量结算，不再一个个重新比。

## 为什么这题能用归并排序
假设数组已经被拆成：

```text
左半部分
右半部分
```

那么总小和可以拆成三部分：

1. 左半部分内部产生的小和
2. 右半部分内部产生的小和
3. 左边元素对右边元素产生的小和贡献

前两部分递归解决。

真正要想办法的是第三部分。

## merge 时到底在结算什么
假设当前归并时：

```text
左：1 3 5
右：2 4 6
```

注意现在左右两边都已经有序。

如果某一时刻比较到：

```text
左[i] = 1
右[j] = 2
```

发现：

```text
1 <= 2
```

那么因为右边从 `j` 开始往后都更大或相等，所以：

```text
1 不仅对 2 有贡献
1 还对 4、6 都有贡献
```

也就是说：

```text
1 的贡献可以一次性结算给右边剩余所有元素
```

这就是高效的根本原因。

## 解题思路
### 1. 递归拆分
和归并排序一样，把数组不断二分，直到只剩一个元素。

单个元素的小和为 `0`。

### 2. 递归求左右部分的小和
分别计算：

- 左半部分的小和
- 右半部分的小和

### 3. merge 时计算跨区间贡献
在合并两个有序部分时：

- 如果 `arr[left] <= arr[right]`
- 那么 `arr[left]` 会对右半部分当前以及后面所有剩余元素产生贡献

贡献值为：

```text
arr[left] * (右边剩余元素个数)
```

然后把 `arr[left]` 放进辅助数组，并让左指针前进。

如果：

```text
arr[left] > arr[right]
```

那当前左元素对这个右元素不产生贡献，
此时先拷贝右侧元素即可。

## 典型例子
数组：

```text
[1, 3, 5, 2, 4, 6]
```

看最终一次 merge：

```text
左：1 3 5
右：2 4 6
```

### 比较 1 和 2
因为：

```text
1 <= 2
```

所以 `1` 对右边当前及后续所有元素都有贡献：

```text
1 * 3 = 3
```

### 比较 3 和 2
因为：

```text
3 > 2
```

所以 `3` 不能对 `2` 产生贡献，先拷贝 `2`。

### 比较 3 和 4
因为：

```text
3 <= 4
```

所以 `3` 对右边剩余 `[4, 6]` 都有贡献：

```text
3 * 2 = 6
```

### 比较 5 和 4
`5 > 4`，无贡献，先拷贝 `4`。

### 比较 5 和 6
`5 <= 6`，所以：

```text
5 * 1 = 5
```

这一轮 merge 的跨区间贡献就是：

```text
3 + 6 + 5 = 14
```

再加上左右子过程各自递归算出的内部贡献，
总小和就是最终答案 `27`。

## 归并排序的“实质”到底是什么
这题最重要的不是记住“小和”本身，
而是看懂归并排序真正厉害的地方：

- 左右两边一旦有序
- 某次比较的结果就能扩展成一批信息
- 不是只服务于“当前这一个元素”

对小和来说是：

- 一次比较，结算一个左元素对右边一批元素的贡献

对逆序对、降序对来说则是：

- 一次比较，结算一批配对关系

所以说：

```text
merge sort 的实质，不只是排序，而是借助有序性批量结算跨区间信息
```

## 复杂度
- 时间复杂度：`O(N log N)`
- 空间复杂度：`O(N)`

因为它本质上还是一次归并排序，只是在 `merge` 时多做了常数级统计。

## 易错点
- 不是在 `arr[left] > arr[right]` 时统计贡献，而是在 `arr[left] <= arr[right]` 时统计。
- 贡献不是只加一次 `arr[left]`，而是要乘上“右边剩余元素个数”。
- 左右两边必须先有序，这个批量结算逻辑才成立。
- 如果题目要求“严格小于”还是“小于等于”，比较符号一定要看清。

更清晰的 `mergeSmallSum` 核心写法可以理解为：

```java
long mergeSmallSum(int[] arr, int l, int m, int r) {
    int p1 = l;
    int p2 = m + 1;
    int[] help = new int[r - l + 1];
    int i = 0;
    long ans = 0;

    while (p1 <= m && p2 <= r) {
        if (arr[p1] <= arr[p2]) {
            ans += (long) arr[p1] * (r - p2 + 1);
            help[i++] = arr[p1++];
        } else {
            help[i++] = arr[p2++];
        }
    }

    while (p1 <= m) {
        help[i++] = arr[p1++];
    }
    while (p2 <= r) {
        help[i++] = arr[p2++];
    }

    // 拷回原数组
    return ans;
}
```

## 和记忆点
- 小和不是一个个位置暴力往左看。
- 真正的高效点在 `merge`。
- 一旦左值小于等于右值，就批量结算贡献。
- 这题看懂了，后面降序对就只是在“统计内容”上换了个方向。
