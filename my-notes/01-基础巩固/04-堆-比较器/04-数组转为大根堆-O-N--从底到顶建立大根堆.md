## 一句话结论
把无序数组直接转成大根堆，最优做法不是一个个 `heapInsert`，而是从后往前对每个位置做 `heapify`。
这样建堆的总时间复杂度是 `O(N)`，而不是很多人第一反应里的 `O(N log N)`。

## 核心知识点
- “一个个插入建堆”是 `O(N log N)`。
- “从最后一个非叶子节点开始向上 heapify” 是 `O(N)`。
- 差别不在单次调整，而在“绝大多数节点离底层很近，调整很短”。
- 这也是堆排序高效的重要前提。

## 两种建堆方式
### 方式 1：从顶到底建堆
把数组元素一个个加入堆中，每加一个都执行 `heapInsert`：

```text
第 1 个元素插入
第 2 个元素插入
...
第 N 个元素插入
```

每次 `heapInsert` 最坏都是 `O(log N)`，
所以总复杂度是：

```text
O(N log N)
```

### 方式 2：从后往前建堆
把整个数组先看成完全二叉树，
然后从最后一个位置一路往前，对每个位置执行 `heapify`。

这就是：

```text
从底向上建堆
```

总复杂度是：

```text
O(N)
```

## 为什么从后往前 heapify 是 O(N)
直觉上很多人会误以为：

- 一共有 `N` 个位置
- 每个位置 `heapify` 最坏 `O(log N)`
- 所以总共是 `O(N log N)`

但这其实高估了。

原因在于：

- 叶子节点根本不需要调整
- 靠近叶子的节点，只会下沉 1~2 层
- 只有很少数靠近根的节点，才可能走到接近 `log N` 的长度

也就是说：

```text
大部分节点的调整代价都很小
```

把这些代价加总起来，最终是线性级别：

```text
O(N)
```

## 解题思路
### 1. 把数组看成完全二叉树
数组本身就已经满足完全二叉树的“形状”。

### 2. 从后往前执行 heapify
从 `n - 1` 开始，一直到 `0`：

```java
for (int i = n - 1; i >= 0; i--) {
    heapify(arr, i, n);
}
```

为什么这样是对的：

- 后面的节点更靠近底部
- 它们先被调整好
- 那么前面节点在调整时，下面子树已经是合法堆结构

这就保证整棵堆能逐步建立起来。

## 典型例子
假设数组：

```text
[3, 5, 1, 2, 4]
```

把它看成完全二叉树后，
从后往前做 `heapify`：

- 叶子节点不会动
- 处理到父节点时，子树已经局部有序
- 最终根节点会被调整成最大值

结果可能变成：

```text
[5, 4, 1, 2, 3]
```

这时整个数组就已经是大根堆。

## 和上一题的关系
上一题讲的是：

- 堆排序时，从后往前 `heapify` 建堆

这一题单独强调的是：

- 为什么这个建堆过程是 `O(N)`
- 为什么它比一个个 `heapInsert` 更优

## 复杂度
- 从顶到底建堆：`O(N log N)`
- 从后往前 `heapify` 建堆：`O(N)`
- 额外空间复杂度：`O(1)`

## 易错点
- “从底到顶建堆”说的是整体建堆顺序，不是单次 `heapify` 的方向。
- 单次 `heapify` 仍然是向下调整。
- `O(N)` 不是因为单次 `heapify` 变快了，而是因为整体均摊后更便宜。
