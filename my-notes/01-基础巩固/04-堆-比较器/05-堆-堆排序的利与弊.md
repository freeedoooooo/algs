## 一句话结论
堆排序最突出的优点，是：

- 时间复杂度稳定在 `O(N log N)`
- 额外空间复杂度做到 `O(1)`

但它也有很明显的缺点：

- 不稳定
- 常数因子通常不如快排友好
- 数据访问局部性差，实际运行速度常常不如随机快排

所以堆排序是一个“理论上很稳、空间上很省，但工程上不一定最快”的排序。

## 核心知识点
- 堆排序的优势不只是能排序，而是“原地排序 + 最坏时间也不退化到 `O(N^2)`”。
- 堆排序的额外空间复杂度是 `O(1)`，因为所有调整都在原数组上完成。
- 堆排序不稳定，相同元素的相对顺序可能被打乱。
- 快排平均速度往往更好，但最坏情况可能退化；堆排则更稳。
- 归并排序稳定、思路规整，但通常需要 `O(N)` 的额外辅助数组。

## 图片转写 / 题意还原
这篇题真正想讨论的，不是“堆排序怎么写”，而是：

```text
为什么堆排序值得学
以及为什么它又不是实际工程里最常用的通用排序
```

也就是说，要从三个角度评价堆排序：

1. 时间复杂度怎么样
2. 空间复杂度怎么样
3. 和快排、归并比起来，到底强在哪、弱在哪

## 图解
可以把三种常见 `O(N log N)` 排序先粗略对照成这样：

```text
快速排序：平均快，但最坏可能退化
归并排序：稳定，但要额外空间
堆排序  ：空间省，最坏也稳，但常数偏大
```

再压缩成一张对比图：

```text
排序算法   时间复杂度           额外空间     稳定性
快排       平均 O(N log N)      O(log N)     不稳定
归并       O(N log N)           O(N)         稳定
堆排       O(N log N)           O(1)         不稳定
```

这张表里，堆排序最亮眼的位置就是：

```text
时间不差
空间最省
```

## 堆排序的优点
### 1. 额外空间复杂度是 `O(1)`
这是堆排序最常被强调的优点。

为什么能做到？

因为堆排序的整个过程：

- 建堆在原数组上做
- 堆顶和末尾交换在原数组上做
- `heapify` 也在原数组上做

例如数组：

```text
[4, 1, 5, 2, 3]
```

无论是建堆、交换还是下沉调整，始终都只是在这一个数组内部改位置，不需要像归并排序那样再额外准备一个 `help[]`。

所以它的额外空间复杂度是：

```text
O(1)
```

### 2. 时间复杂度很稳定
堆排序无论最好、平均还是最坏，整体量级都不会掉出：

```text
O(N log N)
```

这是它相比普通快排一个很重要的优势。  
快排在选 `pivot` 很差时可能退化到 `O(N^2)`，而堆排序不会出现这种级别的退化。

### 3. 适合“空间很紧”的场景
如果题目或环境特别强调：

- 不能开大辅助数组
- 希望原地排序
- 又不想承担快排最坏退化风险

那堆排序会是一个很有竞争力的选择。

## 堆排序的缺点
### 1. 不稳定
堆排序在交换堆顶和末尾、以及 `heapify` 的过程中，会频繁跨位置交换元素。

这意味着：

```text
相同元素的先后顺序可能被打乱
```

比如原来两个相等元素：

```text
a(5), b(5)
```

排序后它们的顺序不一定还是 `a` 在前、`b` 在后。  
所以如果题目要求“稳定排序”，堆排序就不合适。

### 2. 实际运行常常没有快排快
从大 O 看，堆排序和快排都能做到 `O(N log N)`。  
但真正跑起来，堆排序通常没有随机快排快，原因主要有：

- 堆排序每轮都要做“父子跳跃式”比较和交换
- 访问位置不连续，缓存局部性较差
- 常数因子通常更大

所以工程实践里，通用数组排序往往更偏向快排、归并或它们的混合版本，而不是纯堆排序。

### 3. 思维和代码都不如归并自然
归并排序的结构是：

```text
分成两半
各自排好
再合并
```

直观上更顺。

而堆排序要求你一直同时记住：

- 当前堆范围
- 堆顶和末尾交换
- `heapify` 的下沉逻辑

所以不少人第一次学时，会觉得它“能懂原理，但过程不够顺手”。

## 和其他排序的对比
### 和快速排序比
堆排序更强的地方：

- 最坏时间复杂度更稳，不会掉到 `O(N^2)`
- 额外空间更省，可以做到 `O(1)`

快速排序更强的地方：

- 平均性能通常更好
- 工程实现往往更快
- 在现代 CPU/cache 环境下常数更友好

一句话：

```text
堆排更稳，快排通常更快
```

### 和归并排序比
堆排序更强的地方：

- 不需要 `O(N)` 辅助数组
- 可以原地完成排序

归并排序更强的地方：

- 稳定
- 分治结构清晰
- 在链表排序、外排序等场景非常自然

一句话：

```text
堆排更省空间，归并更稳定
```

## 典型例子
假设数组：

```text
[4, 1, 5, 2, 3]
```

堆排序会：

1. 在原数组上建成大根堆
2. 把堆顶最大值换到末尾
3. 对前面的有效堆范围继续 `heapify`

整个过程中：

- 没有新建一个长度为 `N` 的辅助数组
- 只是不断在原数组内部交换元素

所以它的“利”很明显：

```text
原地排序，额外空间 O(1)
```

但与此同时，如果数组里有相等元素，比如：

```text
[5a, 2, 5b, 1]
```

经过堆排序后，`5a` 和 `5b` 的前后顺序不一定还能保持。  
这就是它的“弊”之一：不稳定。

## 什么时候更适合用堆排序
更适合的场景：

- 题目明确要求额外空间尽量小
- 想要一个最坏时间复杂度稳定的 `O(N log N)` 排序
- 需要原地排序，不方便开辅助数组

不太适合的场景：

- 题目要求稳定排序
- 更看重实际跑速而不是理论上界
- 已经有成熟快排/归并库可以直接用

## 常见误区：为什么堆排序不是 `O(N)`
很多人会把下面两件事混在一起：

- 高效建堆是 `O(N)`
- 堆排序整体是 `O(N log N)`

之所以会混，是因为堆排序的第一步“从后往前整体 `heapify` 建堆”确实是：

```text
O(N)
```

但建完堆以后，排序还没结束。  
后面还要反复执行大约 `N` 轮：

1. 把堆顶最大值交换到末尾
2. 堆大小减 1
3. 对新的堆顶做一次 `heapify`

其中第 3 步单次是：

```text
O(log N)
```

所以排序阶段总共是：

```text
O(N log N)
```

把两段加起来：

```text
建堆      O(N)
排序      O(N log N)
整体      O(N log N)
```

所以真正应该记住的是：

```text
高效建堆是 O(N)
堆排序整体不是 O(N)，而是 O(N log N)
```

## 易错点
- `O(1)` 说的是额外空间，不是总空间。
- 堆排序“空间省”不等于“工程上一定最快”。
- 不要把“最坏复杂度稳定”误解成“所有情况下都优于快排”。
- 堆排序不是稳定排序，这一点在面试里很容易被追问。
- 不要把“建堆是 `O(N)`”误记成“堆排序整体是 `O(N)`”。

## 代码 / 伪代码
仓库里对应实现：

- [HeapSort.java](../../../solutions-java/practice/algs01_sort/HeapSort.java)

堆排序能做到 `O(1)` 额外空间，关键就是这些操作都原地完成：

```java
for (int i = n - 1; i >= 0; i--) {
    heapify(arr, i, n);
}
int size = n;
while (size > 1) {
    size--;
    swap(arr, 0, size);
    heapify(arr, 0, size);
}
```

代码里没有新建与 `N` 同级的辅助数组，这正是它空间优势的来源。

## 记忆点
- 堆排序的最大优点：`O(N log N)` 稳定上界 + 额外空间 `O(1)`。
- 堆排序的主要缺点：不稳定，而且实际常数通常不如快排。
- 和快排比：堆排更稳，快排通常更快。
- 和归并比：堆排更省空间，归并更稳定。
