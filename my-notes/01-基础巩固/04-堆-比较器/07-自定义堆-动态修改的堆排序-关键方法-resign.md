## 一句话结论
如果堆里的对象会在入堆后动态改值，那么语言自带的 `PriorityQueue` 往往不够用，这时就需要手写“可定位、可重排”的自定义堆。
其中最关键的方法就是 `resign`：当一个元素的优先级发生变化后，重新把它调整到正确位置。

- 手写堆通常会增加“对象到下标的位置表”，从而支持动态改值后的重新调整。

### 图 2：注意事项
- 如果放入的是自定义对象，要正确实现 `equals()` 和 `hashCode()`。
- `PriorityQueue.remove(Object)` 的时间复杂度通常是 `O(N)`，因为要线性扫描。
- `PriorityQueue` 不适合直接修改堆中对象后还指望它自动重排。
- 如果有频繁修改优先级的需求，更适合使用自定义索引堆 / 支持 `resign` 的堆。

## 核心知识点
- 普通堆擅长：插入、弹出堆顶。
- 普通堆不擅长：找到堆中任意一个对象并修改后快速重排。
- 自定义堆需要额外维护“对象 -> 下标”的映射表。
- `resign` 的本质是：某个对象权重变了后，重新执行上浮和下沉。

## 为什么语言自带堆不够用
以 Java 的 `PriorityQueue` 为例：

- 它确实是堆
- 也确实能快速拿堆顶

但如果你已经把一个对象放进堆里，
然后修改了这个对象里影响排序的字段，
`PriorityQueue` 并不会自动帮你重新调整顺序。

如果你想手动：

1. 先删掉这个对象
2. 改值
3. 再重新放进去

那么问题是：

- `remove(Object)` 往往需要先线性查找
- 复杂度通常是 `O(N)`

这就不适合高频动态更新的场景。

## 自定义堆怎么解决
核心思路是额外维护一张位置表：

```text
对象 -> 当前在堆数组中的下标
```

这样好处是：

- 你能快速找到某个对象在堆里的位置
- 修改它的优先级后，可以直接从那个位置重新调整

这就是 `resign` 存在的意义。

## 什么是 resign
假设堆中某个对象的权重变了：

- 可能变大了
- 也可能变小了

此时它原来的位置可能已经不合法。

但我们事先并不知道它应该：

- 往上走
- 还是往下走

所以最稳妥的写法就是：

1. 对它先尝试上浮（heapInsert）
2. 再尝试下沉（heapify）

这一步综合起来，就是：

```text
resign(obj)
```

## 解题思路
### 1. 堆数组
用数组保存堆结构。

### 2. 位置表
用 `HashMap<T, Integer>` 记录每个对象当前在数组中的位置。

### 3. 比较器
通过比较器定义对象优先级，而不是写死成数字大小。

### 4. resign
当对象内部关键字段变化后：

- 根据位置表找到它所在下标
- 同时做上浮和下沉
- 更新位置表

## 典型场景
### 场景 1：TopK 动态更新
堆里存的是用户对象，
用户分数会变化。

如果用普通 `PriorityQueue`：

- 很难高效找到那个对象
- 改完分数后堆也不会自动调整

如果用自定义堆：

- 先在位置表里定位
- 再执行 `resign`

就能快速恢复正确顺序。

### 场景 2：图算法中的改权
像：

- Dijkstra 改进版

就很依赖“某个节点距离更新后，堆中位置也能快速调整”。

## 复杂度
在位置表支持下：

- 插入：`O(log N)`
- 删除堆顶：`O(log N)`
- `resign`：`O(log N)`
- 查询某对象位置：`O(1)` 平均

如果没有位置表，
很多“先找到对象再调整”的操作就会退化成 `O(N)`。

## 易错点
- 修改了对象权重后，不能指望系统堆自动重排。
- `equals()` 和 `hashCode()` 不正确，位置表就可能失效。
- 交换堆中元素时，别忘了同步更新位置表。
- `resign` 不是简单重建整个堆，而是针对单个元素做局部修复。

## 代码 / 伪代码
仓库里这一题还没有完整的自定义索引堆实现，
但上一题和图片已经把需求背景给出来了。

标准思路可以写成：

```java
class HeapGreater<T> {
    ArrayList<T> heap;
    HashMap<T, Integer> indexMap;
    Comparator<? super T> comp;

    void resign(T obj) {
        int index = indexMap.get(obj);
        heapInsert(index);
        heapify(index);
    }
}
```

其中交换两个位置时要同步更新：

```java
indexMap.put(a, j);
indexMap.put(b, i);
```

## 和记忆点
- 系统堆：拿堆顶很方便，改中间元素不方便
- 自定义堆：多一张位置表
- `resign`：改值后，重新上浮 + 下沉
- 关键目标：让动态修改也能保持 `O(log N)`
