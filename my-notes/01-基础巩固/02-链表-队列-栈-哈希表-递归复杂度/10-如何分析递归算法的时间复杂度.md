## 一句话结论
分析递归算法的时间复杂度，最常见的方法是先写出递推式，再判断是否可以直接套用 Master 公式。
如果递归过程能稳定地拆成若干个等规模子问题，那么通常可以很快得到复杂度结论。

## 核心知识点
- 先把递归写成递推式，而不是直接凭感觉猜复杂度。
- Master 公式只适用于“等规模分治”的递归。
- 递归复杂度通常由“子问题规模”和“本层额外工作量”共同决定。

## 常用分析方法
分析递归复杂度时，常见有 3 种思路：

### 1. 写递推式后套 Master 公式
这是最常见、也最快的方法，适合等规模分治。

标准形式：

```text
T(n) = a * T(n / b) + O(n^d)
```

其中：

- `a`：子问题个数
- `b`：每个子问题的缩小倍数
- `O(n^d)`：当前层额外工作

比较 `log_b(a)` 和 `d`：

1. `d < log_b(a)`：

```text
T(n) = O(n^(log_b(a)))
```

2. `d = log_b(a)`：

```text
T(n) = O(n^d * log n)
```

3. `d > log_b(a)`：

```text
T(n) = O(n^d)
```

### 2. 画递归树
如果公式不太好直接看出来，可以把递归展开成一棵树：

- 看总共有多少层
- 看每一层做多少工作
- 再把每层工作量加起来

这个方法特别适合帮助理解：

- 为什么有时候会多一个 `log n`
- 为什么有时候是底层主导
- 为什么有时候是本层主导

### 3. 直接估层数和每层代价
有些递归题虽然没有标准 Master 形式，但可以直接观察：

- 规模每次缩小多少
- 会递归多少层
- 每层做多少事

例如：

- 每次规模减半，层数通常是 `log n`
- 每层都要扫一遍数组，单层就是 `O(n)`
- 所以总复杂度可能是 `O(n log n)`

## 解题思路
遇到递归题时，可以按下面 4 步分析：

1. 写递推式。
2. 判断是不是等规模分治，能不能套 Master 公式。
3. 如果能套，就比较 `log_b(a)` 和 `d`。
4. 如果不能套，再换递归树、迭代展开、估层数或数学归纳等方法。

## 什么时候不能硬套 Master 公式
下面这些情况通常不能直接硬套：

- 子问题规模不一致
- 递归分支数量不固定
- 递归式不是 `n / b` 这种规整缩放
- 当前层代价不好直接写成 `O(n^d)`

比如：

```text
T(n) = T(n - 1) + O(1)
```

它就不是标准分治递归，不能直接套 Master 公式。

这时更适合从“递归深度”去想：

- 一共递归 `n` 层
- 每层做 `O(1)`
- 所以总复杂度是 `O(n)`

## 典型例子
### 例1：归并排序

```text
T(n) = 2T(n / 2) + O(n)
```

- `a = 2`
- `b = 2`
- `d = 1`
- `log_2(2) = 1`

所以属于第二种情况：

```text
T(n) = O(n log n)
```

### 例2：二分查找

```text
T(n) = T(n / 2) + O(1)
```

- `a = 1`
- `b = 2`
- `d = 0`
- `log_2(1) = 0`

同样属于第二种情况：

```text
T(n) = O(log n)
```

### 例3：每层工作量很重的递归

```text
T(n) = 2T(n / 2) + O(n^2)
```

- `a = 2`
- `b = 2`
- `d = 2`
- `log_2(2) = 1`

因为 `d > log_b(a)`，所以：

```text
T(n) = O(n^2)
```

## 易错点
- 不是所有递归都能套 Master 公式，比如子问题规模不一致、递归分支数量不固定时就不行。
- `O(n^d)` 表示当前这一层额外做的事，不包括递归调用内部的成本。
- 套公式前一定要先把递推式写准确，否则后面的结论都会错。
- 不要把“这一层的代价”误写成“整个递归的总代价”。

## 和记公式题的区别
- `递归复杂度公式` 更偏“结论记忆卡片”
- 这一题更偏“实际分析流程”

两题配合起来记：

- 先知道公式是什么
- 再知道什么时候该用、什么时候不能乱用

## 速记方法
- 子问题更多，看 `log_b(a)`。
- 当前层更重，看 `d`。
- 两边一样重，多一个 `log n`。
