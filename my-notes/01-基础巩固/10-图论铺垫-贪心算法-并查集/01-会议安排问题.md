## 一句话结论
会议安排问题是最经典的贪心题之一：在只有一个会议室的前提下，要让宣讲场次最多，正确策略是优先选择结束时间最早的会议。
它训练的不是排序本身，而是“为什么结束最早的局部选择，不会妨碍后续装入更多会议”。

## 核心知识点
- 这是标准区间调度问题。
- 目标不是总时长最长，而是安排场次最多。
- 正确贪心策略：按结束时间从小到大排序。
- 每次选择开始时间不早于当前时间点的最早结束会议。
- 这个题是很多贪心证明题的起点。

## 图片转写 / 题意还原
原图 1 表述的是本题：

- 一些项目要占用一个会议室宣讲
- 会议室不能同时容纳两个项目
- 给你每个项目的开始时间和结束时间
- 你来安排宣讲日程，要求会议室进行的宣讲场次最多
- 返回最多的宣讲场次

把题目补成完整形式，就是标准单会议室区间调度：

- 有若干个会议或项目宣讲
- 每个会议都有开始时间 `start` 和结束时间 `end`
- 同一时刻会议室只能容纳一个会议
- 一个会议一旦被选，就必须完整占用它自己的时间区间
- 目标不是安排总时长最长，而是安排的会议数量最多
- 输出最多能安排多少场

这题本质上就是经典区间调度问题，重点不在“会议”本身，而在：

```text
如何找到一个局部选择标准
让后续还能装下尽可能多的区间
```

## 图解
如果有一组会议：

```text
A: 1 ~ 4
B: 2 ~ 3
C: 3 ~ 5
D: 0 ~ 6
```

按结束时间排序后：

```text
B(2,3) -> A(1,4) -> C(3,5) -> D(0,6)
```

贪心过程：

```text
先选 B，当前结束时间变成 3
再选 C，因为 C.start = 3
A 已经过期
D 覆盖太长，也装不下更多会议
```

最后得到 2 场。

## 解题思路
### 1. 为什么按结束时间排序
如果你当前必须在若干可选会议里挑一个，那么：

- 结束越早
- 给后面留下的空余时间就越多

因此“结束最早”是最自然的局部最优标准。

### 2. 如何实现
先把所有会议按结束时间升序排序，然后从前往后扫描：

- 如果当前会议开始时间 `>= timeline`
- 就选择它，并把 `timeline` 更新为该会议结束时间

### 3. 为什么不是按开始时间或持续时间
- 按开始时间早，不代表后续空间更大
- 按持续时间短，也不一定释放得更早
- 真正关键的是“结束得早”

## 复杂度
- 时间复杂度：`O(N log N)`，主要来自排序
- 空间复杂度：`O(1)` 或 `O(log N)`，取决于排序实现

## 典型例子
给定会议：

```text
(1,3) (2,4) (3,5) (0,7) (5,6)
```

按结束时间排序：

```text
(1,3) (2,4) (3,5) (5,6) (0,7)
```

选择过程：

- 先选 `(1,3)`
- `(2,4)` 不能选
- 选 `(3,5)`
- 选 `(5,6)`
- `(0,7)` 不能选

答案是 `3`。

## 易错点
- 目标是“场次最多”，不是“占用总时长最大”。
- 排序关键字是结束时间，不是开始时间。
- 判断能否接上时，通常允许下一个会议的开始时间等于当前结束时间。
- 想验证贪心策略时，可以先写暴力枚举的小样本对照器。

## 代码 / 伪代码
课程标准伪代码如下：

```java
int bestArrange(Program[] programs) {
    Arrays.sort(programs, (a, b) -> a.end - b.end);
    int timeline = 0;
    int ans = 0;
    for (Program p : programs) {
        if (p.start >= timeline) {
            ans++;
            timeline = p.end;
        }
    }
    return ans;
}
```

## 记忆点
- 会议安排看结束时间。
- 越早结束，越给后面留空间。
- 区间调度是贪心入门题。
- 不放心就先写暴力对拍。
