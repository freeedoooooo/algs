## 一句话结论
这题是并查集最典型的业务落地题之一：

```text
如果两条用户记录在任意一个关键字段上相同
就认为它们属于同一个真实用户
```

真正难点不在“比较字段”，而在于：

```text
同一人关系有传递性
所以必须把所有相关记录整组成组地合并
```

## 先记结论
第一次看这题，先记住下面 4 句话：

```text
1. 任意一个关键字段相同，就要合并
2. 合并关系有传递性
3. 哈希表负责找“历史上见过的同字段用户”
4. 并查集负责把所有关联用户收拢成一组
```

如果只记一句话，可以记成：

```text
字段负责“发现关系”
并查集负责“传递合并”
```

## 题意说明
给定一批用户记录。  
每条记录里有若干个能够标识身份的字段，比如：

- 账号
- 邮箱
- 手机号

规则是：

- 如果两条记录任意一个关键字段相同，就认为它们是同一个人
- 这种“同一个人”的关系还具有传递性

要求返回：

```text
把所有应该合并的记录合并之后
最终还剩多少个真实用户
```

## 为什么这题适合并查集
这题表面上像“字段去重”，但本质上不是简单去重，而是：

```text
不断发现两条记录之间有关系
再把这些关系整组成组地连起来
```

并查集正好擅长：

- 每个用户先自己成组
- 一旦发现两个用户应当视为同一人，就 `union`
- 最后数还剩多少个集合

你可以把它理解成：

```text
哈希表负责发现“边”
并查集负责把这些边连成“块”
```

## 如果不用并查集，会麻烦在哪里
如果不用并查集，常见麻烦是：

- 你得手动维护“哪些人属于哪一组”
- 你得自己处理“组合并组”
- 你得自己保证传递关系没有漏

一旦关系链变长，逻辑就会越来越绕。

## 图解
假设有 4 个用户：

```text
u1 = (id1, mail1, phone1)
u2 = (id2, mail1, phone2)
u3 = (id3, mail3, phone3)
u4 = (id2, mail4, phone4)
```

直接关系是：

- `u1` 和 `u2` 因为 `mail1` 相同要合并
- `u2` 和 `u4` 因为 `id2` 相同要合并

于是：

```text
u1 <-> u2 <-> u4
```

虽然 `u1` 和 `u4` 没有直接相同字段，但它们仍然会因为传递性被并到一起。

最终分组就是：

```text
[u1, u2, u4]
[u3]
```

所以答案是 `2`。

## 解题思路
最稳的做法是：

```text
并查集 + 多张哈希表
```

### 1. 并查集负责“分组”
每个用户先看成一个独立集合。  
只要发现两个用户应该视为同一人，就执行一次 `union`。

### 2. 哈希表负责“发现关系”
假设每个用户有三个关键字段 `a、b、c`，那就准备三张表：

- `aMap`
- `bMap`
- `cMap`

每张表的含义都是：

```text
key   = 某个字段值
value = 第一次出现这个字段值的用户
```

### 3. 扫描用户时怎么做
处理当前用户 `user` 时，对它的每个字段都做同样的检查：

- 如果这个字段值以前没出现过，就记到 map 里
- 如果以前出现过，说明找到了“同字段的历史用户”
- 这时把当前用户和那个历史用户执行 `union`

### 4. 为什么这样就够了
因为哈希表负责找到“局部直接关系”，并查集会自动把这些关系扩展成整组传递关系。

## 代码 / 伪代码
下面这版伪代码更适合直接对应前面的思路去读：

```java
int mergeUsers(User[] users) {
    UnionFind<User> uf = new UnionFind<>(users);

    Map<String, User> aMap = new HashMap<>();
    Map<String, User> bMap = new HashMap<>();
    Map<String, User> cMap = new HashMap<>();

    for (User user : users) {
        if (aMap.containsKey(user.a)) {
            uf.union(user, aMap.get(user.a));
        } else {
            aMap.put(user.a, user);
        }

        if (bMap.containsKey(user.b)) {
            uf.union(user, bMap.get(user.b));
        } else {
            bMap.put(user.b, user);
        }

        if (cMap.containsKey(user.c)) {
            uf.union(user, cMap.get(user.c));
        } else {
            cMap.put(user.c, user);
        }
    }

    return uf.sets();
}
```

一句话概括就是：

```text
先用哈希表找“字段撞车”
一旦撞车，就用并查集合并用户
```

## 复杂度
- 时间复杂度：近似 `O(N)`
- 空间复杂度：`O(N)`

因为：

- 每个用户只会被扫描一次
- 每个字段只做哈希表查询和并查集操作
- 并查集在路径压缩和按规模合并后，单次操作几乎可以看成常数

## 易错点
- 不是字段全部相同才合并，而是任意一个关键字段相同就合并。
- 不能只做局部去重，因为关系有传递性。
- 最后统计答案时，应该统计并查集里集合数量，不是统计哈希表里还有多少键。
- 哈希表负责找连接边，并查集负责真正合并；两者分工不要混。

## 记忆点
- 合并用户题，本质是并查集处理传递合并关系。
- 任意字段相同就 `union`。
- 哈希表负责找“历史同字段用户”。
- 最后答案看并查集里还剩几个集合。
