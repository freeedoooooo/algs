## 一句话结论
图题真正难的地方，往往不在算法本身，而在于：

```text
先把题目输入统一转成一个你能稳定编码的图结构
```

只要图表示清楚了，后面的 BFS、DFS、拓扑、最小生成树、最短路，基本都只是往这套结构上套模板。

## 先记结论
第一次看图论，建议先记住这 4 句话：

```text
1. 图 = 点集 + 边集
2. 边可以有方向，也可以带权
3. 图有很多表示方法，但要选最适合当前题目的
4. 面试里常先把输入转成统一结构，再写算法
```

如果只记一句话，可以记成：

```text
图题先建模，再上算法
```

## 题意说明
这篇不是某一道题，而是图论的入门准备笔记。  
它主要解决两个问题：

1. 图到底是什么
2. 图在代码里通常怎么表示

所以这篇的重点不是某一个算法，而是：

```text
先把“图”这件事本身讲清楚
```

## 什么是图
图由两部分组成：

- 点的集合
- 边的集合

可以写成：

```text
Graph = Nodes + Edges
```

图里最常见的变化有两个：

### 1. 有向图 / 无向图
- 有向图：边有方向，例如 `A -> B`
- 无向图：边没方向，例如 `A -- B`

编码时，无向图通常也可以统一转成两条有向边：

```text
A -> B
B -> A
```

### 2. 带权图 / 不带权图
- 不带权：只关心有没有边
- 带权：边上还带数字，例如距离、代价、时间

例如：

```text
A -> B (5)
```

表示从 `A` 到 `B` 有一条权值为 `5` 的边。

## 图的常见术语
### 1. 邻接点
如果有边：

```text
A -> B
```

那就说 `B` 是 `A` 的一个邻接点。

### 2. 入度 / 出度
在有向图里：

- 入度：有多少条边指向这个点
- 出度：有多少条边从这个点发出

例如：

```text
A -> C
B -> C
C -> D
```

那：

- `C` 的入度是 `2`
- `C` 的出度是 `1`

### 3. 路径 / 连通
- 路径：从一个点走到另一个点的边序列
- 连通：两个点之间是否能走到

这些概念后面会在遍历、最短路、最小生成树里频繁出现。

## 图的表示方法
图的表示方法不止一种。  
常见有下面 4 类。

### 1. 邻接矩阵
适合点数不大、需要快速判断“两个点之间是否直接有边”的场景。

例如：

```text
    A B C
A   0 1 1
B   0 0 1
C   0 0 0
```

这里 `matrix[i][j] = 1` 表示有边 `i -> j`。

#### 优点
- 判断两点之间是否有边很快
- 形式规整，容易理解

#### 缺点
- 空间通常是 `O(N^2)`
- 图很稀疏时会浪费很多空间

### 2. 邻接表
适合稀疏图，也是算法题里最常见的思路之一。

例如：

```text
A: B, C
B: C
C: -
```

表示：

```text
A -> B
A -> C
B -> C
```

#### 优点
- 省空间
- 遍历某个点的所有邻居很方便

#### 缺点
- 判断两点之间是否直接有边，不如邻接矩阵直接

### 3. 边集表示
有时题目本身就直接给边数组：

```text
(A, B, 5)
(A, C, 2)
(B, D, 7)
```

这种写法很适合建图输入，也常用于最小生成树、最短路等题的原始输入。

### 4. 自定义结构 `Node / Edge / Graph`
这是课程里最推荐的统一写法。

因为它适合把很多图算法都挂到同一套结构上：

```text
Graph
├─ nodes: 所有点
└─ edges: 所有边

Node
├─ value: 点值
├─ in: 入度
├─ out: 出度
├─ nexts: 邻接点
└─ edges: 从该点发出的边

Edge
├─ weight: 权值
├─ from: 起点
└─ to: 终点
```

## 为什么很多面试题喜欢先转成统一结构
因为题目给你的输入格式往往五花八门：

- 二维数组
- 三元组边集
- 字符串关系
- 邻接表

如果每道题都直接在原始输入上硬写，代码会很乱。

更稳的套路是：

```text
先把输入转成熟悉的图结构
再在这套结构上写 BFS / DFS / 拓扑 / 最短路 / 最小生成树
```

这也是为什么很多图题，真正费时间的是“建图”而不是“算法思想”。

## 什么时候用哪种表示方法
可以简单这样记：

- 邻接矩阵：点不多，且频繁判断两点是否直接相连
- 邻接表：图稀疏，最常用
- 边集：题目原始输入就是边，或者算法本来就更关注边
- 自定义 `Node / Edge / Graph`：想统一整套图算法模板

## 小例子：同一张图的多种表示
假设边集是：

```text
1 -> 2 (5)
1 -> 3 (2)
2 -> 4 (7)
```

### 1. 边集表示

```text
(1, 2, 5)
(1, 3, 2)
(2, 4, 7)
```

### 2. 邻接表表示

```text
1: 2(5), 3(2)
2: 4(7)
3: -
4: -
```

### 3. 自定义结构里的结果
建图后可以得到：

- `1.out = 2`
- `2.in = 1`
- `3.in = 1`
- `4.in = 1`
- `1.nexts = [2, 3]`
- `1.edges = [(1,2,5), (1,3,2)]`

这也说明，自定义结构其实是在把图题后面常用的信息都提前准备好。

## 代码 / 伪代码
课程里最常用的统一结构可以写成：

```java
class Node {
    int value;
    int in;
    int out;
    List<Node> nexts;
    List<Edge> edges;
}

class Edge {
    int weight;
    Node from;
    Node to;
}

class Graph {
    Map<Integer, Node> nodes;
    Set<Edge> edges;
}
```

如果题目给的是无向边 `A -- B`，通常建图时拆成两条有向边：

```text
A -> B
B -> A
```

## 易错点
- `nexts` 是邻接点，不是边。
- `edges` 通常保存从当前点发出的边。
- 无向图转成统一结构时，往往会拆成两条有向边。
- 建图时别忘了同步维护入度、出度、邻接点和边集。
- 图题经常不是算法难，而是建图细节容易写乱。

## 记忆点
- 图 = 点集 + 边集。
- 图可以有方向，也可以带权。
- 图有多种表示方法：邻接矩阵、邻接表、边集、自定义结构。
- 图题先建模，再上算法。
