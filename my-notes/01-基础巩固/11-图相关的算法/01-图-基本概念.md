## 一句话结论
图题真正难的地方往往不在算法本身，而在于怎么把输入统一转成一个适合编码的图结构。
课程里推荐的统一写法是自己定义 `Node / Edge / Graph` 三个类，后面 BFS、DFS、拓扑、最小生成树、最短路都直接复用。

## 核心知识点
- 图由点集和边集构成。
- 边可能有方向，也可能带权。
- 常见存法有邻接表、邻接矩阵，但面试更常先转成自定义结构。
- 节点里一般保存：值、入度、出度、邻接点、出边集合。
- `Graph` 里一般保存：所有节点、所有边。

## 图解
统一结构可以理解成：

```text
Graph
├─ nodes: 所有点
└─ edges: 所有边

Node
├─ value: 点值
├─ in: 入度
├─ out: 出度
├─ nexts: 邻接点
└─ edges: 从该点发出的边

Edge
├─ weight: 权值
├─ from: 起点
└─ to: 终点
```

## 解题思路
### 1. 为什么不直接死用邻接矩阵
邻接矩阵当然能表示图，但很多图题：

- 点数很大
- 边数稀疏
- 还要方便拿到入度、出边、邻接点

这时自定义结构更灵活。

### 2. 为什么无向图也能用有向图表达
无向边 `A -- B`，直接拆成：

```text
A -> B
B -> A
```

这样很多算法的框架都可以统一。

## 复杂度
- 建图时间复杂度：`O(V + E)`
- 建图空间复杂度：`O(V + E)`

## 典型例子
如果输入边集是：

```text
1 -> 2 (5)
1 -> 3 (2)
2 -> 4 (7)
```

建图后：

- `1.out = 2`
- `2.in = 1`
- `3.in = 1`
- `4.in = 1`

而 `1.nexts = [2, 3]`。

## 易错点
- `nexts` 是邻接点，不是边。
- `edges` 通常保存从当前点发出的边。
- 无向图转换后会出现两条有向边。
- 建图时别忘了同步维护入度和出度。

## 代码 / 伪代码
课程标准伪代码：

```java
class Node {
    int value;
    int in;
    int out;
    List<Node> nexts;
    List<Edge> edges;
}

class Edge {
    int weight;
    Node from;
    Node to;
}

class Graph {
    Map<Integer, Node> nodes;
    Set<Edge> edges;
}
```

## 记忆点
- 图题先建模。
- 推荐统一成 `Node / Edge / Graph`。
- 点里放入度、出度、邻接点、出边。
- 无向图可拆成双向有向边。
