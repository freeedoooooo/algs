## 一句话结论
如果元素进堆以后，优先级还会继续变化，那么语言自带的 `PriorityQueue` 往往就不够用了。

这时更合适的做法是自己维护一个“可定位、可重排”的堆：

- 用数组保存堆结构
- 用位置表记录“对象当前在哪”
- 当对象优先级变化时，调用 `resign`

其中 `resign` 的本质就是：

```text
这个元素改值了，
把它从当前位置重新调整回正确位置。
```

## 图片转写 / 题意还原
这道笔记真正想讲的，不是“堆还能自己写一遍”这么简单，而是：

```text
普通堆只擅长拿堆顶，
但工程里经常需要“堆中某个对象改值后还能快速重排”。
```

所以这里要解决的核心问题是：

1. 元素已经在堆里了
2. 这个元素的优先级后来变了
3. 我们希望仍然能在 `O(log N)` 内把它调回正确位置

这就是自定义优先级队列的意义。

## 先明确普通 `PriorityQueue` 卡在哪
普通堆最擅长的是：

- `push`
- `pop` 堆顶

也就是说，它很适合“只关心当前最值”的场景。

但如果你想做下面这件事：

```text
堆里已经有对象 A
后来 A 的权重变了
我想快速让它重新归位
```

普通 `PriorityQueue` 就会变得很别扭。

因为它通常不擅长：

- 快速定位堆中的某个任意对象
- 对象改值后自动重新排位置

如果你想手动处理，往往只能：

1. 先找到这个对象
2. 把它删掉
3. 改值
4. 再重新插入

问题在于：

- 第 1 步“找到它”就不轻松
- `PriorityQueue.remove(Object)` 往往是 `O(N)`

所以一旦有“高频更新优先级”的需求，普通系统堆就不够顺手了。

## 自定义堆到底多了什么
自定义优先级队列，并不是推翻堆本身，而是在普通堆基础上多维护了一张表：

```text
对象 -> 当前所在下标
```

常见结构可以记成三件套：

- `heap[]`：真正保存堆元素的数组
- `indexMap`：对象到下标的位置表
- `comp`：比较器，决定谁优先级更高

画成结构图就是：

```text
heap 数组:
index:  0      1      2      3      4
value: [A10] [B20] [C30] [D40] [E50]

indexMap:
A -> 0
B -> 1
C -> 2
D -> 3
E -> 4
```

这样一来，当你说“我要修改 `E`”时，就不用在堆里线性扫描找它了。

因为：

```text
先查 indexMap
立刻知道 E 在下标 4
```

然后从这个位置开始重新调整即可。

## 为什么位置表这么关键
没有位置表时，你面对的是：

```text
我知道对象是谁
但不知道它在堆数组哪个位置
```

有了位置表以后，流程会直接顺很多：

1. 通过对象拿到下标
2. 在这个下标上改值
3. 从这个下标开始做局部调整

这意味着我们不再需要：

- 全堆重建
- 线性扫描找对象

真正要修的，只是：

```text
这个对象所在的一小段路径
```

这也是为什么它能把更新复杂度控制在 `O(log N)`。

## `resign` 到底在干什么
假设堆中某个对象的权重变了。

这时它原来的位置可能已经不合法，但问题是：

```text
它到底应该往上走，还是往下走？
```

事先不一定知道。

因为：

- 如果权重变得更优，它可能要上浮
- 如果权重变得更差，它可能要下沉

所以 `resign` 最稳妥的理解就是：

```text
从当前下标出发，
把这个改值后的元素重新放回它应该在的位置。
```

课程里常见写法是：

```java
void resign(T obj) {
    int index = indexMap.get(obj);
    heapInsert(index);
    heapify(index);
}
```

可以把它理解成：

1. 先试试能不能往上走
2. 再试试需不需要往下走

虽然看起来像“两边都做”，但实际真正发生位移的通常只有一个方向。

## 图解 1：优先级变好，向上调整
假设我们维护的是小根堆，数值越小优先级越高。

初始状态：

```text
heap:
index:  0      1      2      3      4
value: [A10] [B20] [C30] [D40] [E50]
```

此时：

```text
indexMap:
A -> 0
B -> 1
C -> 2
D -> 3
E -> 4
```

现在 `E` 的值从 `50` 改成了 `5`。

### 第一步：先定位

```text
indexMap.get(E) = 4
```

说明 `E` 当前在下标 `4`。

### 第二步：执行 `resign(E)`
此时 `E5` 明显比它父节点 `B20` 更小，所以要上浮。

先交换一次：

```text
index:  0      1      2      3      4
value: [A10] [E5 ] [C30] [D40] [B20]
```

位置表也要同步更新：

```text
A -> 0
E -> 1
C -> 2
D -> 3
B -> 4
```

继续比较，`E5` 仍然比父节点 `A10` 更小，于是再交换一次：

```text
index:  0      1      2      3      4
value: [E5 ] [A10] [C30] [D40] [B20]
```

最终 `E` 来到堆顶，堆重新合法。

这就是：

```text
优先级变好 -> 上浮
```

## 图解 2：优先级变差，向下调整
还是小根堆。

假设当前状态是：

```text
index:  0      1      2      3      4
value: [A10] [B20] [C30] [D40] [E50]
```

现在 `A` 的值从 `10` 改成了 `35`。

此时 `A35` 作为堆顶已经不合法，因为它比孩子 `B20`、`C30` 都大。

执行 `resign(A)` 后，它就应该下沉。

先和更小的孩子 `B20` 交换：

```text
index:  0      1      2      3      4
value: [B20] [A35] [C30] [D40] [E50]
```

再检查 `A35` 是否还需要继续往下。

它和 `D40`、`E50` 比已经没问题，于是停止。

这就是：

```text
优先级变差 -> 下沉
```

## 完整步骤
如果把自定义优先级队列的工作流压缩成流程，可以记成：

### 1. 入堆时
- 把对象放进 `heap[]`
- 记录到 `indexMap`
- 做一次上浮

### 2. 弹出堆顶时
- 交换堆顶和末尾
- 更新 `indexMap`
- 删除末尾
- 对新的堆顶做下沉

### 3. 对象改值时
- 先通过 `indexMap` 找到它的位置
- 改它的关键字段
- 调用 `resign`

### 4. `resign` 时
- 先尝试上浮
- 再尝试下沉
- 最终把该对象放回正确位置

## 为什么 `swap` 时一定要同步更新 `indexMap`
这是实现里最容易漏的一点。

因为堆数组一交换，位置表就立刻过期了。

比如原来：

```text
A -> 0
E -> 4
```

如果交换了 `A` 和 `E`，但你没更新位置表，那么接下来：

```text
indexMap.get(E)
```

拿到的还是旧位置。

后果就是：

- 后续 `resign(E)` 找错位置
- 删除错元素
- 整个堆结构越来越乱

所以每次交换都要把这两步绑在一起看：

```text
交换 heap[i] 和 heap[j]
同步更新 indexMap
```

## 为什么 `resign` 不是重建整个堆
有些人第一次听到“元素改值后重新调整”，会下意识想到：

```text
那我干脆整堆重建不就完了？
```

当然可以，但代价太大。

因为真正失效的，通常只有这个元素所在的局部路径。

也就是说，需要修复的只是：

- 它往上到根的一条链
- 或它往下到某个叶子的一条链

所以局部修复就够了。

这就是 `resign` 能做到 `O(log N)`，而不是 `O(N)` 或更高的原因。

## 典型场景
### 场景 1：TopK / 排行榜动态更新
堆里存的是用户对象，用户分数会不断变化。

这时你希望做到：

- 分数更新后立刻调整位置
- 不需要整个排行榜重排

自定义堆就很合适。

### 场景 2：图算法中的距离更新
像 Dijkstra 改进版里，某个节点的当前最短距离可能被更新得更小。

这时就很需要：

```text
找到这个节点
更新距离
重新在小根堆中归位
```

所以很多图算法里的“加强堆”，本质上就是这里讲的这套东西。

## 复杂度
在位置表支持下，常见操作复杂度是：

- 插入：`O(log N)`
- 删除堆顶：`O(log N)`
- `resign`：`O(log N)`
- 查询某对象位置：`O(1)` 平均

而如果没有位置表，很多“先找到某个对象再调整”的动作就会退化成 `O(N)`。

## 易错点
- 修改了对象内部关键字段后，不能指望系统堆自动重排。
- 自定义对象如果要做 `HashMap` 键，`equals()` 和 `hashCode()` 要正确。
- 每次交换堆中两个元素，都要同步更新 `indexMap`。
- `resign` 修的是单个元素的局部路径，不是整堆重建。
- 写 `resign` 时，重点不是死记“上浮 + 下沉”，而是理解“这个元素改值后方向未知，所以统一做局部修复”。

## 代码 / 伪代码
仓库里这一题还没有完整的自定义堆实现文件，但标准骨架通常会长这样：

```java
class HeapGreater<T> {
    ArrayList<T> heap;
    HashMap<T, Integer> indexMap;
    Comparator<? super T> comp;

    void resign(T obj) {
        int index = indexMap.get(obj);
        heapInsert(index);
        heapify(index);
    }

    void swap(int i, int j) {
        T a = heap.get(i);
        T b = heap.get(j);
        heap.set(i, b);
        heap.set(j, a);
        indexMap.put(a, j);
        indexMap.put(b, i);
    }
}
```

如果改成更容易理解的伪代码，可以记成：

```java
resign(obj):
    index = indexMap.get(obj)
    从 index 往上试
    从 index 往下试
```

## 记忆点
- 系统堆擅长拿堆顶，不擅长改中间元素后重排。
- 自定义优先级队列 = 堆数组 + 位置表 + 比较器。
- `resign` 就是“对象改值后重新归位”。
- 真正让它变强的关键，不是堆本身，而是那张 `对象 -> 下标` 的位置表。
