## 一句话结论
这题是并查集最典型的业务化落地：如果两个用户在若干身份字段里任意一个相同，就认为他们属于同一个真实用户，最后返回合并后的用户数。
核心不是字段比较本身，而是把“属于同一人”抽象成连通关系，然后用并查集维护传递闭包。

## 核心知识点
- 任意一个字段相同，就视为同一人。
- “同一人”关系具有传递性，天然适合并查集。
- 常见做法是为每个字段值建哈希表，记录第一次出现它的用户。
- 新用户遇到相同字段时，直接和旧用户 `union`。
- 最后统计并查集里集合个数即可。

## 图片转写 / 题意还原
原图给出的代码骨架是：

- `User` 类里有三个字段：`a、b、c`
- 如果两个用户 `a` 相同，或 `b` 相同，或 `c` 相同，就认为是一个人
- 请合并所有用户，返回合并之后的用户数量

配图旁边还强调了一点：

- 并查集在图和区域合并问题里非常常见

把题目整理成完整业务描述，就是：

- 给定一批用户记录
- 每条记录里有若干个能够标识身份的字段，例如账号、邮箱、手机号
- 如果两条记录任意一个关键字段相同，就认为它们属于同一个真实用户
- 这种“同一人”关系还具有传递性
- 最终要求把所有应当合并的记录合并成若干组
- 返回合并后还剩多少个真实用户

## 图解
假设有 4 个用户：

```text
u1 = (id1, mail1, phone1)
u2 = (id2, mail1, phone2)
u3 = (id3, mail3, phone3)
u4 = (id2, mail4, phone4)
```

关系是：

- `u1` 和 `u2` 因为 `mail1` 相同要合并
- `u2` 和 `u4` 因为 `id2` 相同要合并

于是：

```text
u1、u2、u4 是同一组
u3 单独一组
```

最终答案是 `2`。

## 解题思路
### 1. 三张字段表
分别维护：

- `aMap`
- `bMap`
- `cMap`

键是字段值，值是第一次出现该字段的用户。

### 2. 扫描每个用户
处理一个用户时：

- 如果字段值没出现过，就放进 map
- 如果出现过，就把当前用户和旧用户 `union`

### 3. 为什么这样能覆盖传递性
因为并查集会把所有相关联的用户自动收拢到一个集合里。

## 复杂度
- 时间复杂度：近似 `O(N)`
- 空间复杂度：`O(N)`

## 典型例子
如果：

```text
u1.a == u2.a
u2.c == u3.c
```

虽然 `u1` 和 `u3` 没有直接相同字段，但因为：

```text
u1 ~ u2
u2 ~ u3
```

所以它们最终也会被并到同一个集合中。

## 易错点
- 不是字段都相同才合并，而是任意一个相同就合并。
- 关系有传递性，不能只做局部去重。
- 统计答案时，应该统计代表点数量，不是剩余 map 数量。

## 代码 / 伪代码
课程标准伪代码：

```java
int mergeUsers(User[] users) {
    UnionFind<User> uf = new UnionFind<>(users);
    Map<String, User> aMap = new HashMap<>();
    Map<String, User> bMap = new HashMap<>();
    Map<String, User> cMap = new HashMap<>();

    for (User user : users) {
        if (aMap.containsKey(user.a)) uf.union(user, aMap.get(user.a));
        else aMap.put(user.a, user);

        if (bMap.containsKey(user.b)) uf.union(user, bMap.get(user.b));
        else bMap.put(user.b, user);

        if (cMap.containsKey(user.c)) uf.union(user, cMap.get(user.c));
        else cMap.put(user.c, user);
    }
    return uf.sets();
}
```

## 记忆点
- 账号合并、身份合并，优先想到并查集。
- 任意字段相同就 `union`。
- 哈希表负责找“同字段的历史用户”。
- 最后看集合个数。
