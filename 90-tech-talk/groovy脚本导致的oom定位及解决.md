# 数据准备 OOM 定位及解决

## 1. 问题背景

数据准备任务在执行指标计算时出现内存异常，核心表现为：

- 计算过程中 `FGC` 频率高。
- 即使任务执行完成，老年代一度无法及时回收。
- 动态执行的 Groovy 脚本、类加载器、Metaspace 占用存在明显异常。

本文基于原始排查记录，对问题现象、定位过程、根因判断和优化方案进行整理。

## 2. 初始现象

最初系统使用的是**全局静态类加载器**。

通过 `jstat -gc` 观察到：

- 计算完成后，老年代仍然持续无法被 `FGC` 回收。
- `FGC` 次数从 `710` 增加到 `765`。
- 老年代占用长期维持在较高水位，没有明显下降。

结论：

- 全局静态类加载器场景下，动态加载出来的类和关联对象没有被及时释放。
- 这会导致老年代对象持续滞留，进一步放大 Full GC 频率。

## 3. 第一轮优化尝试

优化思路改为：

- 每次计算都创建新的类加载器。
- 计算完成后主动回收资源。

优化后的观测结果：

- 计算完成后，老年代最终可以被回收。
- 但在计算过程中，老年代仍然无法及时回收。
- 因此 `FGC` 频率依然偏高。

当时记录到的关键指标：

- `OC = 1398M`
- `OU = 225M`
- `MC = 446M`
- `MU = 185M`

结论：

- “每次创建并回收类加载器”相比“全局静态类加载器”已经明显改善。
- 但这还没有彻底解决计算过程中的高频 GC 问题。

## 4. 堆内实例对比

### 4.1 静态类加载器场景

`jmap -histo` 统计显示，内存中存在大量 Groovy 相关对象，例如：

- `org.codehaus.groovy.runtime.metaclass.MetaMethodIndex$Entry`
- `java.util.LinkedHashMap$Entry`
- `org.codehaus.groovy.util.FastArray`

其中部分对象实例数和字节数都非常大，说明动态脚本执行后的元数据和关联结构大量堆积。

### 4.2 每次新建并回收类加载器场景

同样通过 `jmap -histo` 观察，TOP 对象规模显著下降，主要为：

- `char[]`
- `java.lang.reflect.Method`
- `int[]`
- `java.lang.String`
- `java.lang.ref.WeakReference`

结论：

- 问题的主要矛盾确实与 Groovy 脚本动态加载后的类、方法元数据及其关联缓存有关。
- 类加载器回收策略调整后，堆中异常对象规模明显收敛。

## 5. 进一步定位

排查过程中提出了两个关键问题：

### 5.1 类加载器是否一致

待验证项：

- Java 代码中的 `GroovyShell` 类加载器，是否与脚本内部 `new GroovyShell` 使用的是同一个类加载器。
- 可以通过 debug 某个脚本，查看相关对象内存地址或类加载器引用来验证。

这一步的意义在于确认：

- 是否存在额外的类加载链路。
- 是否存在脚本内部再次创建 GroovyShell，导致类无法统一释放的问题。

### 5.2 为什么任务结束后能回收，但执行过程中回收不下来

从 GC 现象看：

- 新生代增长非常快，很快被占满。
- 对象来不及在年轻代被回收，发生晋升进入老年代。
- 老年代 `OU` 持续上升，频繁触发 `Full GC`。
- 但任务完成后，`OU` 最终仍然可以下降。

这说明：

- 很多对象并不是“永久泄漏”，而是“在任务执行期间持续存活”。
- 真正的问题更像是**执行过程中的对象生命周期过长**，叠加动态类加载带来的 Metaspace 压力，最终表现为 OOM 风险和高频 Full GC。

## 6. GC 日志分析结论

重跑 `c1-data` 定时任务并打印 GC 日志后，可以得到几个重要结论：

- `EU` 增长速度非常快，说明 Eden 区对象创建速度很高。
- `YGC` 非常频繁。
- 几乎每次 `YGC` 后，`OU` 都在上涨，说明对象不断晋升到老年代。
- 老年代被快速填满后，系统频繁触发 `Full GC`。

文档中的进一步判断是：

- 可能在任务初期，新生代放不下大量对象，导致对象直接或快速晋升到老年代。
- 老年代中又持有新生代对象引用，导致 YGC 无法彻底清理。
- 形成“新生代压力大 -> 晋升加快 -> 老年代上涨 -> Full GC 频繁”的恶性循环。

同时，排查中还得出一个更明确的结论：

- `full gc` 基本都是由 **Metaspace 到达阈值** 触发的。

这意味着问题不仅仅是普通堆对象过多，还包括：

- 动态生成类过多。
- 类元数据没有及时卸载。
- 类加载器释放不彻底，导致 Metaspace 压力持续累积。

## 7. 根因判断

综合文档中的实验和日志，问题根因可以归纳为两部分。

### 7.1 直接根因

- 使用全局静态类加载器，导致动态 Groovy 脚本加载出的类无法及时卸载。
- 脚本相关类、方法元数据、缓存结构长期驻留，造成堆内对象与 Metaspace 持续增长。

### 7.2 触发放大因素

- 指标计算期间对象创建速度快，新生代回收跟不上。
- 大量对象晋升到老年代，`OU` 持续增长。
- Metaspace 持续上涨并达到阈值，触发 Full GC。
- Full GC 发生频繁，最终表现为任务执行慢、内存抖动严重，甚至 OOM。

## 8. 解决方案

文档最终给出的解决方向有两个。

### 8.1 调整 Metaspace 相关 JVM 参数

目的：

- 延缓因为 Metaspace 过快增长而触发的 Full GC。
- 给类卸载和对象回收争取更稳定的运行空间。

适用场景：

- 动态脚本较多、类元数据增长明显的任务。

注意：

- 这类参数调整只能缓解问题，不能替代类加载器释放和类卸载治理。

### 8.2 在脚本中主动释放 GroovyShell 的 ClassLoader

目的：

- 避免 `Script.class` 无法卸载。
- 让动态加载的 `script_*`、`index_*` 等类在 GC 后真正被卸载。

从文档后续验证结果来看：

- 优化后，`dump` 会触发一次 Full GC。
- Full GC 之后，`script_` 和 `index_` 开头的动态加载类都已经被卸载。
- 再次执行指标时，可以看到新的 `index_*` 类被重新创建。
- 垃圾回收之后，实例数可以降为 `0`。
- 只有刚刚执行过的指标会保留一个实例。

这说明：

- 优化已经能够让动态类按预期完成“创建 -> 使用 -> 回收 -> 卸载”的闭环。
- 之前无法释放的问题，确实与类加载器和脚本类卸载链路有关。

## 9. 当前结论

本次问题本质上不是单一的“堆泄漏”，而是：

- 动态脚本类加载策略不合理，导致类和元数据无法及时卸载。
- 指标执行过程中对象创建量过大，造成年轻代压力和对象晋升。
- Metaspace 阈值触发 Full GC，使问题进一步放大。

最终有效的治理方向是：

- 不再使用全局静态类加载器。
- 每次执行创建独立类加载器，并在执行后主动释放。
- 配合优化 GroovyShell 生命周期，确保 `Script.class` 可卸载。
- 结合 Metaspace JVM 参数调整，降低 Full GC 触发频率。

## 10. 待继续验证的问题

文档中仍保留了以下待确认项：

- Java 代码里的 `GroovyShell` 与脚本中 `new GroovyShell` 的类加载器是否一致。
- 脚本中的静态变量是否仍然会把部分动态类或实例间接保活。

其中一个重点怀疑点是：

- 脚本中的静态变量可能导致类似 `TB_*` 相关 `clazz` 的实例即使在垃圾回收后仍被保留。
- 这会造成大量以 `Script_*` 开头的类持续存在。

建议后续继续通过以下手段验证：

- 使用 `jcmd <pid> GC.class_histogram`
- 使用 `jmap -histo:live <pid>`
- 在关键脚本执行点打断点，观察类加载器引用链和静态变量引用链

## 11. 建议沉淀

如果后续要把这套经验沉淀为稳定方案，建议形成以下规范：

- 动态脚本执行必须使用短生命周期类加载器。
- 脚本执行完成后必须显式释放 `GroovyShell` / `ClassLoader` 引用。
- 避免在脚本静态变量中缓存大对象、类对象或执行上下文。
- 将 Metaspace、FGC 次数、类卸载情况纳入巡检指标。
- 对定时任务增加 GC 日志和类直方图的标准排查手册。

## 12. 一句话结论

这次数据准备 OOM 的核心原因，是 **Groovy 动态脚本类加载与卸载策略不当，叠加执行期对象快速晋升和 Metaspace 膨胀，最终导致 Full GC 频繁和内存风险升高**；优化的关键在于 **类加载器短生命周期化 + 主动释放 GroovyShell + 合理设置 Metaspace 参数**。

## 13. 结论复核

结合原始记录中的 `jstat`、`jmap -histo`、动态类卸载现象，这份文档里的结论可以分成两类。

### 13.1 证据较充分的结论

- 全局静态类加载器存在明显问题，这一点基本成立。
- 问题与 Groovy 动态类、类加载器、Metaspace 强相关，这一点基本成立。
- “短生命周期类加载器 + 主动释放 GroovyShell / ClassLoader” 是正确优化方向，这一点基本成立。

支撑这些结论的证据包括：

- 从全局静态类加载器切换到“每次新建并回收类加载器”后，堆内异常对象规模明显下降。
- 优化后任务结束时老年代能够回收。
- Full GC 触发点与 Metaspace 阈值相关。
- 优化后 `script_*`、`index_*` 等动态类在 Full GC 后可以被卸载。

### 13.2 目前仍属于推断的结论

- “对象晋升导致老年代上涨”更适合作为现象解释或放大机制，而不是最终根因。
- “老年代引用新生代对象，导致 YGC 无法清理干净”这一表述不够严谨，目前缺少直接证据。
- “不是堆泄漏，只是执行期对象生命周期长”这一判断下得偏早，仍需继续验证是否存在静态变量、缓存或类元数据保活。
- “调整 Metaspace 参数即可解决问题”不成立，参数调优只能缓解，不能替代根因治理。

因此，更严谨的表达应当是：

- 已确认问题与动态类加载和类卸载链路有关。
- 已确认 Metaspace 压力是 Full GC 的重要触发因素。
- 执行期对象创建快、晋升快，是 GC 压力的重要放大因素。
- 是否还存在脚本静态变量、缓存链路导致的对象保活，仍需继续验证。

## 14. 提炼要点

这次问题可以提炼为 4 个核心要点：

- 问题现象：任务执行时 `YGC / FGC` 频繁，老年代和 Metaspace 持续升高，存在 OOM 风险。
- 关键证据：全局静态类加载器场景下，Groovy 相关元数据对象大量堆积；改成独立类加载器并主动回收后，对象规模明显下降，动态类可被卸载。
- 核心判断：主要问题不在普通业务对象本身，而在 Groovy 动态类加载后的类、元数据和类加载器没有及时释放；同时执行期对象创建速度快，放大了 GC 压力。
- 有效方案：取消全局静态类加载器，改为短生命周期类加载器；主动释放 `GroovyShell/ClassLoader`；继续排查脚本静态变量和缓存；Metaspace 参数仅作为辅助缓解手段。

## 15. 建议对外结论

如果需要对外汇报或沉淀为正式结论，建议使用下面这版表述：

> 本次数据准备 OOM / 高频 Full GC 的主要原因，是 Groovy 动态脚本执行过程中使用了不合理的类加载策略，导致动态类及其元数据无法及时卸载，Metaspace 持续膨胀；同时任务执行期间对象创建速度快，进一步放大了年轻代回收和对象晋升压力。
>
> 已验证有效的优化手段是：使用短生命周期类加载器、执行后主动释放 `GroovyShell/ClassLoader`，确保动态类可卸载。Metaspace JVM 参数调整只能缓解症状，不能替代根因治理。
