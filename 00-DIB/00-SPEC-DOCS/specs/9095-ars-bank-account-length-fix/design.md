# 9095 银行账号长度限制修复 - 技术设计

## 设计概述

本需求旨在修改前端银行账号验证规则，将最小长度从 10 位调整为 9 位，支持 9-30 位的银行账号输入。

## 问题分析

### 当前问题
- 前端验证规则使用正则表达式 `/^[0-9]\d{9,29}$/`
- 该正则要求：第一位 0-9，后续 9-29 位，总共 **10-30 位**
- 导致 9 位银行账号无法通过验证

### 根本原因
正则表达式中的 `\d{9,29}` 表示 9-29 位数字，加上前面的 `[0-9]` 一位，总共是 10-30 位。

## 技术方案

### 1. 正则表达式修改

#### 修改前（错误）
```javascript
/^[0-9]\d{9,29}$/
```
- 含义：第一位 0-9，后续 9-29 位数字
- 总长度：10-30 位

#### 修改后（正确）
```javascript
/^[0-9]\d{8,29}$/
```
- 含义：第一位 0-9，后续 8-29 位数字
- 总长度：9-30 位
- **首位可以为 0**（符合实际业务需求）

### 2. 验证规则说明

#### 正则表达式详解
```javascript
/^[0-9]\d{8,29}$/
```
- `^` - 字符串开始锚点
- `[0-9]` - 第一位必须是数字 0-9（**包括 0**）
- `\d{8,29}` - 后续 8-29 位数字（等价于 `[0-9]{8,29}`）
- `$` - 字符串结束锚点

#### 验证范围
- **最小长度**：9 位（如 `123456789` 或 `012345678`）
- **最大长度**：30 位
- **允许字符**：仅数字 0-9
- **首位限制**：无限制，可以为 0

#### 测试用例
| 输入 | 长度 | 是否通过 | 说明 |
|------|------|----------|------|
| `123456789` | 9 | ✅ | 9位，首位非0 |
| `012345678` | 9 | ✅ | 9位，首位为0 |
| `1234567890` | 10 | ✅ | 10位 |
| `123456789012345678901234567890` | 30 | ✅ | 30位（最大） |
| `12345678` | 8 | ❌ | 8位（太短） |
| `1234567890123456789012345678901` | 31 | ❌ | 31位（太长） |
| `12345678a` | 9 | ❌ | 包含字母 |
| `123-456-789` | 11 | ❌ | 包含特殊字符 |

## 实现细节

### 1. Element Plus 表单验证机制

#### 验证规则配置
```typescript
// 方式1：在 rules 对象中定义
const rules = {
  bankAccount: [
    { required: true, message: '请输入银行账号', trigger: 'blur' },
    { pattern: /^[0-9]\d{8,29}$/, message: '请输入正确的银行账号', trigger: 'blur' }
  ]
}

// 方式2：在配置对象中定义
{
  field: 'bankAccount',
  widget: 'input',
  rules: [
    { pattern: /^[0-9]\d{8,29}$/, message: '请输入正确的银行账号' }
  ]
}
```

#### 验证触发时机
- `trigger: 'blur'` - 输入框失焦时触发验证
- `trigger: 'change'` - 输入值改变时触发验证

### 2. 配置文件说明

#### ContractCommonFormCfg.ts
- 作用：合同模块的表单配置中心
- 特点：配置驱动，统一管理表单字段属性和验证规则
- 影响范围：所有使用该配置的合同表单

#### formCfg.ts 系列
- 作用：各业务模块的表单配置文件
- 特点：动态生成表单配置，支持条件渲染
- 影响范围：对应业务模块的表单

### 3. 验证流程

```
用户输入 → 失焦事件 → 触发验证 → 正则匹配
                                    ↓
                              匹配成功 → 通过
                                    ↓
                              匹配失败 → 显示错误提示
```
